## วัตถุประสงค์การเรียนรู้

หลังจากศึกษาบทนี้แล้ว ผู้เรียนควรสามารถ:

- อธิบายการใช้งานโมดูล I/O ในฐานะส่วนหนึ่งของสถาปัตยกรรมคอมพิวเตอร์
- เข้าใจความแตกต่างระหว่าง **Programmed I/O** และ **Interrupt-Driven I/O** รวมถึงข้อดีข้อเสียของแต่ละแบบ
- นำเสนอภาพรวมการทำงานของ Direct Memory Access (DMA)
- นำเสนอภาพรวมการทำงานของ Direct Cache Access (DCA)
- อธิบายฟังก์ชันและการใช้งาน I/O Channels

## บทนำ

นอกจากหน่วยประมวลผล (processor) และชุดโมดูลหน่วยความจำแล้ว องค์ประกอบสำคัญลำดับที่สามของระบบคอมพิวเตอร์คือชุดโมดูล I/O แต่ละโมดูลจะเชื่อมต่อกับบัสระบบ (system bus) หรือสวิตช์กลาง (central switch) และทำหน้าที่ควบคุมอุปกรณ์ต่อพ่วง (peripheral device) หนึ่งชิ้นหรือมากกว่า

โมดูล I/O ไม่ใช่เพียงแค่ชุดตัวเชื่อมต่อทางกลสำหรับต่อสายไฟระหว่างอุปกรณ์กับระบบเท่านั้น แต่ในความเป็นจริง โมดูล I/O มีวงจรลอจิกสำหรับทำหน้าที่สื่อสารระหว่างอุปกรณ์ต่อพ่วงและบัสระบบ

### เหตุใดจึงไม่เชื่อมต่ออุปกรณ์ต่อพ่วงโดยตรงกับบัสระบบ?

ผู้อ่านอาจสงสัยว่าเหตุใดจึงไม่เชื่อมต่ออุปกรณ์ต่อพ่วงโดยตรงกับบัสระบบ เหตุผลมีดังนี้:

1. **ความหลากหลายของอุปกรณ์ต่อพ่วง** — มีอุปกรณ์ต่อพ่วงหลากหลายชนิดที่มีวิธีการทำงานต่างกัน การนำตรรกะที่จำเป็นทั้งหมดไว้ในหน่วยประมวลผลเพื่อควบคุมอุปกรณ์ต่างๆ นั้นเป็นสิ่งที่ไม่สามารถทำได้จริง

2. **ความไม่ตรงกันของอัตราการถ่ายโอนข้อมูล** — อัตราการถ่ายโอนข้อมูลของอุปกรณ์ต่อพ่วงมักช้ากว่าหน่วยความจำหรือหน่วยประมวลผลมาก ดังนั้นการใช้บัสความเร็วสูงสื่อสารโดยตรงกับอุปกรณ์ต่อพ่วงจึงไม่เหมาะสม

3. **อุปกรณ์บางชนิดเร็วกว่าหน่วยความจำ** — ในทางกลับกัน อัตราการถ่ายโอนข้อมูลของอุปกรณ์ต่อพ่วงบางชนิดเร็วกว่าหน่วยความจำหรือหน่วยประมวลผล ซึ่งความไม่ตรงกันนี้จะนำไปสู่ประสิทธิภาพที่ต่ำลงหากไม่ได้รับการจัดการอย่างเหมาะสม

4. **ความแตกต่างของรูปแบบข้อมูล** — อุปกรณ์ต่อพ่วงมักใช้รูปแบบข้อมูลและความยาวคำ (word length) ที่แตกต่างจากคอมพิวเตอร์ที่เชื่อมต่ออยู่

ดังนั้นจึงจำเป็นต้องมีโมดูล I/O โมดูลนี้มีฟังก์ชันหลัก 2 ประการ:

- เชื่อมต่อกับหน่วยประมวลผลและหน่วยความจำผ่านบัสระบบหรือสวิตช์กลาง
- เชื่อมต่อกับอุปกรณ์ต่อพ่วงหนึ่งชิ้นหรือมากกว่าผ่านสายเชื่อมต่อข้อมูลเฉพาะทาง

## 7.1 อุปกรณ์ภายนอก (External Devices)

การดำเนินงาน I/O สำเร็จได้ด้วยอุปกรณ์ภายนอกที่หลากหลาย ซึ่งทำหน้าที่เป็นสื่อกลางในการแลกเปลี่ยนข้อมูลระหว่างสภาพแวดล้อมภายนอกกับคอมพิวเตอร์ อุปกรณ์ภายนอกเชื่อมต่อกับคอมพิวเตอร์ผ่านสายเชื่อมต่อไปยังโมดูล I/O สายเชื่อมต่อนี้ใช้แลกเปลี่ยนสัญญาณควบคุม สถานะ และข้อมูลระหว่างโมดูล I/O กับอุปกรณ์ภายนอก อุปกรณ์ภายนอกที่เชื่อมต่อกับโมดูล I/O มักเรียกว่า **อุปกรณ์ต่อพ่วง (peripheral device)** หรือเรียกสั้นๆ ว่า **peripheral**

### การจำแนกประเภทอุปกรณ์ภายนอก

สามารถจำแนกอุปกรณ์ภายนอกได้เป็น 3 ประเภทหลัก:

| ประเภท | คำอธิบาย | ตัวอย่าง |
|--------|----------|---------|
| **Human-readable** | เหมาะสำหรับการสื่อสารกับผู้ใช้คอมพิวเตอร์ | จอแสดงผล (VDT), เครื่องพิมพ์ |
| **Machine-readable** | เหมาะสำหรับการสื่อสารกับอุปกรณ์อื่น | ดิสก์แม่เหล็ก, เทปแม่เหล็ก, เซ็นเซอร์, ตัวกระตุ้น (actuator) ในหุ่นยนต์ |
| **Communication** | เหมาะสำหรับการสื่อสารกับอุปกรณ์ระยะไกล | โมเด็ม, อุปกรณ์เครือข่าย |

> **หมายเหตุ:** ระบบดิสก์และเทปถูกมองเป็นทั้งอุปกรณ์ I/O (มุมมองโครงสร้าง) และอุปกรณ์หน่วยความจำ (มุมมองหน้าที่การใช้งานในลำดับชั้นหน่วยความจำ)

### ลักษณะทั่วไปของอุปกรณ์ภายนอก

อินเทอร์เฟซสู่โมดูล I/O อยู่ในรูปแบบสัญญาณควบคุม (control), ข้อมูล (data), และสถานะ (status):

- **สัญญาณควบคุม (Control signals)** — กำหนดฟังก์ชันที่อุปกรณ์จะดำเนินการ เช่น ส่งข้อมูลไปยังโมดูล I/O (INPUT/READ), รับข้อมูลจากโมดูล I/O (OUTPUT/WRITE), รายงานสถานะ หรือทำหน้าที่ควบคุมเฉพาะทาง
- **สัญญาณสถานะ (Status signals)** — แสดงสถานะของอุปกรณ์ เช่น READY/NOT-READY เพื่อบอกว่าอุปกรณ์พร้อมสำหรับการถ่ายโอนข้อมูลหรือไม่
- **วงจรลอจิกควบคุม (Control logic)** — ควบคุมการทำงานของอุปกรณ์ตามคำสั่งจากโมดูล I/O
- **ตัวแปลงสัญญาณ (Transducer)** — แปลงข้อมูลจากสัญญาณไฟฟ้าเป็นรูปแบบพลังงานอื่นในระหว่างการส่งออก และแปลงจากรูปแบบอื่นเป็นสัญญาณไฟฟ้าระหว่างการรับเข้า
- **บัฟเฟอร์ (Buffer)** — เก็บข้อมูลชั่วคราวในระหว่างการถ่ายโอนระหว่างโมดูล I/O กับสภาพแวดล้อมภายนอก บัฟเฟอร์ขนาด 8-16 บิตเป็นเรื่องปกติสำหรับอุปกรณ์แบบอนุกรม ในขณะที่อุปกรณ์แบบบล็อก เช่น ตัวควบคุมดิสก์ไดรฟ์ อาจมีบัฟเฟอร์ขนาดใหญ่กว่ามาก

### แป้นพิมพ์/จอภาพ (Keyboard/Monitor)

วิธีการที่พบบ่อยที่สุดในการโต้ตอบระหว่างคอมพิวเตอร์กับผู้ใช้คือการใช้แป้นพิมพ์ร่วมกับจอภาพ ผู้ใช้ป้อนข้อมูลผ่านแป้นพิมพ์ ข้อมูลนั้นถูกส่งไปยังคอมพิวเตอร์และอาจแสดงบนจอภาพด้วย รวมถึงจอภาพยังแสดงข้อมูลที่คอมพิวเตอร์ส่งมาด้วย

หน่วยพื้นฐานของการแลกเปลี่ยนคือตัวอักษร (character) แต่ละตัวอักษรมีรหัสที่เกี่ยวข้อง ซึ่งโดยทั่วไปมีความยาว 7 หรือ 8 บิต รหัสข้อความที่ใช้บ่อยที่สุดคือ **International Reference Alphabet (IRA)** ซึ่งแต่ละตัวอักษรแทนด้วยรหัสไบนารี 7 บิตเฉพาะ ทำให้สามารถแทนตัวอักษรได้ถึง 128 ตัว

ตัวอักษรมี 2 ประเภท:
- **ตัวอักษรที่พิมพ์ได้ (Printable characters)** — ตัวอักษร ตัวเลข และสัญลักษณ์พิเศษที่สามารถพิมพ์หรือแสดงบนหน้าจอได้
- **ตัวอักษรควบคุม (Control characters)** — ใช้ควบคุมการพิมพ์หรือแสดงผล เช่น carriage return หรือใช้ในกระบวนการสื่อสาร

สำหรับการรับข้อมูลจากแป้นพิมพ์ เมื่อผู้ใช้กดปุ่ม สัญญาณอิเล็กทรอนิกส์ถูกสร้างขึ้นและถูกแปลโดย transducer ในแป้นพิมพ์ให้เป็นรูปแบบบิตของรหัส IRA ที่สอดคล้องกัน รูปแบบบิตนี้จะถูกส่งไปยังโมดูล I/O ในคอมพิวเตอร์ ที่คอมพิวเตอร์สามารถเก็บข้อความในรหัส IRA เดียวกันได้

### ดิสก์ไดรฟ์ (Disk Drive)

ดิสก์ไดรฟ์มีวงจรอิเล็กทรอนิกส์สำหรับแลกเปลี่ยนข้อมูล สัญญาณควบคุม และสัญญาณสถานะกับโมดูล I/O รวมถึงวงจรควบคุมกลไกการอ่าน/เขียนดิสก์ ในดิสก์แบบหัวอ่านอยู่กับที่ (fixed-head disk) transducer สามารถแปลงระหว่างรูปแบบแม่เหล็กบนพื้นผิวดิสก์ที่เคลื่อนที่กับบิตในบัฟเฟอร์ของอุปกรณ์ สำหรับดิสก์แบบหัวอ่านเคลื่อนที่ (moving-head disk) จะต้องสามารถเลื่อนแขนดิสก์ไปตามรัศมีเข้าหรือออกจากพื้นผิวดิสก์ได้ด้วย

## 7.2 โมดูล I/O (I/O Modules)

### ฟังก์ชันของโมดูล

ฟังก์ชันหรือความต้องการหลักของโมดูล I/O แบ่งออกเป็นหมวดหมู่ดังต่อไปนี้:

1. **การควบคุมและการกำหนดเวลา (Control and timing)** — ประสานงานการไหลของข้อมูลระหว่างทรัพยากรภายในและอุปกรณ์ภายนอก
2. **การสื่อสารกับหน่วยประมวลผล (Processor communication)** — รับและตอบสนองต่อคำสั่งจาก CPU
3. **การสื่อสารกับอุปกรณ์ (Device communication)** — ส่งคำสั่งและรับสถานะจากอุปกรณ์ต่อพ่วง
4. **การบัฟเฟอร์ข้อมูล (Data buffering)** — ชดเชยความแตกต่างของอัตราการถ่ายโอนข้อมูล
5. **การตรวจสอบข้อผิดพลาด (Error detection)** — ตรวจจับและรายงานข้อผิดพลาดไปยังหน่วยประมวลผล

### ลำดับขั้นตอนการถ่ายโอนข้อมูล

ตัวอย่างการควบคุมการถ่ายโอนข้อมูลจากอุปกรณ์ภายนอกไปยังหน่วยประมวลผล:

1. หน่วยประมวลผลสอบถามโมดูล I/O เพื่อตรวจสอบสถานะของอุปกรณ์ที่เชื่อมต่อ
2. โมดูล I/O ส่งคืนสถานะของอุปกรณ์
3. หากอุปกรณ์ทำงานปกติและพร้อมส่งข้อมูล หน่วยประมวลผลส่งคำขอถ่ายโอนข้อมูลไปยังโมดูล I/O
4. โมดูล I/O รับหน่วยข้อมูล (เช่น 8 หรือ 16 บิต) จากอุปกรณ์ภายนอก
5. ข้อมูลถูกถ่ายโอนจากโมดูล I/O ไปยังหน่วยประมวลผล

### การสื่อสารกับหน่วยประมวลผล (Processor Communication)

การสื่อสารกับหน่วยประมวลผลประกอบด้วย:

- **การถอดรหัสคำสั่ง (Command decoding)** — โมดูล I/O รับคำสั่งจากหน่วยประมวลผล ซึ่งโดยทั่วไปส่งเป็นสัญญาณบนบัสควบคุม ตัวอย่างเช่น โมดูล I/O สำหรับดิสก์ไดรฟ์อาจรับคำสั่ง: READ SECTOR, WRITE SECTOR, SEEK track number, และ SCAN record ID

- **ข้อมูล (Data)** — ข้อมูลแลกเปลี่ยนระหว่างหน่วยประมวลผลและโมดูล I/O ผ่านบัสข้อมูล

- **การรายงานสถานะ (Status reporting)** — เนื่องจากอุปกรณ์ต่อพ่วงทำงานช้า จึงสำคัญที่จะต้องทราบสถานะของโมดูล I/O สัญญาณสถานะทั่วไปได้แก่ BUSY และ READY อาจมีสัญญาณสำหรับรายงานเงื่อนไขข้อผิดพลาดต่างๆ ด้วย

- **การรู้จำที่อยู่ (Address recognition)** — เช่นเดียวกับที่แต่ละคำในหน่วยความจำมีที่อยู่ แต่ละอุปกรณ์ I/O ก็มีที่อยู่ด้วย โมดูล I/O จึงต้องรู้จำที่อยู่เฉพาะสำหรับอุปกรณ์ต่อพ่วงแต่ละตัวที่ควบคุม

### การบัฟเฟอร์ข้อมูล (Data Buffering)

งานสำคัญของโมดูล I/O คือการบัฟเฟอร์ข้อมูล ในขณะที่อัตราการถ่ายโอนเข้าและออกจากหน่วยความจำหลักหรือหน่วยประมวลผลสูงมาก แต่สำหรับอุปกรณ์ต่อพ่วงจำนวนมาก อัตราดังกล่าวต่ำกว่าหลายระดับของขนาด ข้อมูลที่มาจากหน่วยความจำหลักถูกส่งไปยังโมดูล I/O อย่างรวดเร็ว ข้อมูลถูกบัฟเฟอร์ในโมดูล I/O จากนั้นส่งไปยังอุปกรณ์ต่อพ่วงตามอัตราข้อมูลของมัน ในทิศทางตรงข้าม ข้อมูลถูกบัฟเฟอร์เพื่อไม่ให้หน่วยความจำถูกผูกกับการดำเนินการถ่ายโอนที่ช้า

### การตรวจสอบข้อผิดพลาด (Error Detection)

โมดูล I/O มักรับผิดชอบในการตรวจสอบข้อผิดพลาดและรายงานข้อผิดพลาดไปยังหน่วยประมวลผล ข้อผิดพลาดมีสองประเภทหลัก:

1. **ข้อผิดพลาดทางกลและไฟฟ้า** — รายงานโดยอุปกรณ์ (เช่น กระดาษติด, แทร็กดิสก์เสีย)
2. **การเปลี่ยนแปลงรูปแบบบิต** — การเปลี่ยนแปลงโดยไม่ตั้งใจในรูปแบบบิตระหว่างการถ่ายโอนจากอุปกรณ์ไปยังโมดูล I/O ซึ่งมักใช้รหัสตรวจสอบข้อผิดพลาดบางรูปแบบ เช่น การใช้บิตพาริตี (parity bit) ในแต่ละตัวอักษร

### โครงสร้างโมดูล I/O (I/O Module Structure)

โมดูล I/O เชื่อมต่อกับส่วนอื่นของคอมพิวเตอร์ผ่านชุดสายสัญญาณ (เช่น สายบัสระบบ) ข้อมูลที่ถ่ายโอนเข้าและออกจากโมดูลถูกบัฟเฟอร์ในรีจิสเตอร์ข้อมูลหนึ่งตัวหรือมากกว่า นอกจากนี้อาจมีรีจิสเตอร์สถานะหนึ่งตัวหรือมากกว่าที่ให้ข้อมูลสถานะปัจจุบัน

โมดูล I/O แบ่งตามความสามารถ:
- **I/O channel / I/O processor** — โมดูลที่รับภาระการประมวลผลส่วนใหญ่ มีอินเทอร์เฟซระดับสูงกับหน่วยประมวลผล ใช้ในเมนเฟรม
- **I/O controller / Device controller** — โมดูลที่ค่อนข้างเรียบง่าย ต้องการการควบคุมโดยละเอียด พบบ่อยในไมโครคอมพิวเตอร์

## 7.3 Programmed I/O

มีสามเทคนิคที่เป็นไปได้สำหรับการดำเนินงาน I/O:

| เทคนิค | ใช้ Interrupt | การถ่ายโอน I/O-to-Memory |
|--------|--------------|------------------------|
| **Programmed I/O** | ไม่ใช้ | ผ่านหน่วยประมวลผล |
| **Interrupt-driven I/O** | ใช้ | ผ่านหน่วยประมวลผล |
| **Direct Memory Access (DMA)** | ใช้ | โดยตรง (ไม่ผ่านหน่วยประมวลผล) |

### ภาพรวมของ Programmed I/O

เมื่อหน่วยประมวลผลกำลังรันโปรแกรมและพบคำสั่งที่เกี่ยวกับ I/O จะดำเนินการคำสั่งนั้นโดยส่งคำสั่งไปยังโมดูล I/O ที่เหมาะสม ด้วย Programmed I/O โมดูล I/O จะดำเนินการที่ร้องขอและตั้งค่าบิตที่เหมาะสมในรีจิสเตอร์สถานะ I/O แต่โมดูล I/O **ไม่ดำเนินการใดๆ เพิ่มเติม** เพื่อแจ้งเตือนหน่วยประมวลผล โดยเฉพาะอย่างยิ่ง มันไม่ขัดจังหวะหน่วยประมวลผล

ดังนั้นจึงเป็นหน้าที่ของหน่วยประมวลผลที่จะตรวจสอบสถานะของโมดูล I/O เป็นระยะๆ จนกว่าจะพบว่าการดำเนินงานเสร็จสิ้น

**ข้อเสียหลัก:** เป็นกระบวนการที่ใช้เวลาซึ่งทำให้หน่วยประมวลผลไม่จำเป็นต้องว่างงาน

### คำสั่ง I/O (I/O Commands)

เมื่อดำเนินการคำสั่งที่เกี่ยวกับ I/O หน่วยประมวลผลส่งที่อยู่ระบุโมดูล I/O เฉพาะและอุปกรณ์ภายนอก รวมถึงคำสั่ง I/O ประเภทของคำสั่ง I/O มี 4 ประเภท:

1. **Control** — ใช้เปิดใช้งานอุปกรณ์ต่อพ่วงและบอกให้ทำงาน ตัวอย่างเช่น เทปแม่เหล็กอาจได้รับคำสั่งให้กรอกลับหรือเดินหน้าหนึ่งเรคคอร์ด
2. **Test** — ใช้ทดสอบเงื่อนไขสถานะต่างๆ ที่เกี่ยวข้องกับโมดูล I/O และอุปกรณ์ต่อพ่วง
3. **Read** — ทำให้โมดูล I/O รับรายการข้อมูลจากอุปกรณ์ต่อพ่วงและวางไว้ในบัฟเฟอร์ภายใน หน่วยประมวลผลจะได้รับรายการข้อมูลโดยร้องขอให้โมดูล I/O วางบนบัสข้อมูล
4. **Write** — ทำให้โมดูล I/O รับรายการข้อมูล (ไบต์หรือคำ) จากบัสข้อมูลและต่อมาส่งรายการข้อมูลนั้นไปยังอุปกรณ์ต่อพ่วง

### คำสั่ง I/O และการกำหนดที่อยู่

มีวิธีการกำหนดที่อยู่อุปกรณ์ภายนอก 2 รูปแบบ:

#### Memory-Mapped I/O
- มีพื้นที่ที่อยู่เดียวสำหรับตำแหน่งหน่วยความจำและอุปกรณ์ I/O
- หน่วยประมวลผลปฏิบัติต่อรีจิสเตอร์สถานะและข้อมูลของโมดูล I/O เหมือนตำแหน่งหน่วยความจำ
- ใช้คำสั่งเครื่องเดียวกันในการเข้าถึงทั้งหน่วยความจำและอุปกรณ์ I/O
- ข้อดี: สามารถใช้ชุดคำสั่งอ้างอิงหน่วยความจำที่มีขนาดใหญ่ได้ ทำให้เขียนโปรแกรมได้มีประสิทธิภาพมากขึ้น
- ข้อเสีย: ใช้พื้นที่ที่อยู่หน่วยความจำที่มีค่า

#### Isolated I/O
- พื้นที่ที่อยู่ I/O แยกจากพื้นที่หน่วยความจำ
- บัสมีสายคำสั่ง memory read/write และ input/output แยกกัน
- สายคำสั่งระบุว่าที่อยู่อ้างถึงตำแหน่งหน่วยความจำหรืออุปกรณ์ I/O
- ช่วงที่อยู่ทั้งหมดพร้อมใช้สำหรับทั้งหน่วยความจำและ I/O

```
ตัวอย่าง Memory-Mapped I/O (10-bit address):
- ที่อยู่ 516: รีจิสเตอร์ข้อมูลการรับเข้าจากแป้นพิมพ์
- ที่อยู่ 517: รีจิสเตอร์สถานะ/ควบคุมการรับเข้าจากแป้นพิมพ์

ตัวอย่างโปรแกรม Memory-Mapped I/O:
ADDRESS  INSTRUCTION  OPERAND    COMMENT
200      Load AC      "1"        โหลดตัวสะสม
         Store AC     517        เริ่มการอ่านจากแป้นพิมพ์
202      Load AC      517        รับไบต์สถานะ
         Branch if Sign = 0  202   วนซ้ำจนพร้อม
         Load AC      516        โหลดไบต์ข้อมูล

ตัวอย่างโปรแกรม Isolated I/O:
ADDRESS  INSTRUCTION  OPERAND    COMMENT
200      Load I/O     5          เริ่มการอ่านจากแป้นพิมพ์
201      Test I/O     5          ตรวจสอบความสมบูรณ์
         Branch Not Ready  201   วนซ้ำจนสมบูรณ์
         In           5          โหลดไบต์ข้อมูล
```

## 7.4 Interrupt-Driven I/O

### ปัญหาของ Programmed I/O

ปัญหาของ Programmed I/O คือหน่วยประมวลผลต้องรอนานสำหรับโมดูล I/O ที่เกี่ยวข้องให้พร้อมสำหรับการรับหรือส่งข้อมูล หน่วยประมวลผลขณะรอต้องสอบถามสถานะของโมดูล I/O ซ้ำๆ ส่งผลให้ประสิทธิภาพโดยรวมของระบบลดลงอย่างมาก

### หลักการทำงานของ Interrupt-Driven I/O

วิธีทางเลือกคือให้หน่วยประมวลผลส่งคำสั่ง I/O ไปยังโมดูล แล้วไปทำงานอื่นที่มีประโยชน์ต่อไป โมดูล I/O จะขัดจังหวะ (interrupt) หน่วยประมวลผลเพื่อขอบริการเมื่อพร้อมแลกเปลี่ยนข้อมูลกับหน่วยประมวลผล

**มุมมองของโมดูล I/O (สำหรับการรับเข้า):**
1. โมดูล I/O รับคำสั่ง READ จากหน่วยประมวลผล
2. โมดูล I/O ดำเนินการอ่านข้อมูลจากอุปกรณ์ต่อพ่วงที่เกี่ยวข้อง
3. เมื่อข้อมูลอยู่ในรีจิสเตอร์ข้อมูลของโมดูล โมดูลส่งสัญญาณขัดจังหวะไปยังหน่วยประมวลผลผ่านสายควบคุม
4. โมดูลรอจนกว่าข้อมูลจะถูกร้องขอโดยหน่วยประมวลผล
5. เมื่อมีการร้องขอ โมดูลวางข้อมูลบนบัสข้อมูลและพร้อมสำหรับการดำเนินงาน I/O อีกครั้ง

**มุมมองของหน่วยประมวลผล:**
1. หน่วยประมวลผลส่งคำสั่ง READ แล้วไปทำงานอื่น
2. ณ สิ้นสุดของแต่ละรอบคำสั่ง หน่วยประมวลผลตรวจสอบ interrupt
3. เมื่อ interrupt จากโมดูล I/O เกิดขึ้น หน่วยประมวลผลบันทึกบริบท (context) ของโปรแกรมปัจจุบัน
4. ประมวลผล interrupt: อ่านคำข้อมูลจากโมดูล I/O และเก็บในหน่วยความจำ
5. คืนค่าบริบทของโปรแกรมและดำเนินการต่อ

**ข้อดี:** มีประสิทธิภาพมากกว่า Programmed I/O เพราะขจัดการรอที่ไม่จำเป็น

**ข้อเสีย:** ยังคงใช้เวลาหน่วยประมวลผลมาก เพราะทุกคำข้อมูลที่ไปจากหน่วยความจำถึงโมดูลหรือจากโมดูล I/O ถึงหน่วยความจำต้องผ่านหน่วยประมวลผล

### การประมวลผล Interrupt

เมื่อ interrupt เกิดขึ้น ลำดับเหตุการณ์ฮาร์ดแวร์ดังต่อไปนี้เกิดขึ้น:

1. **อุปกรณ์ส่งสัญญาณ interrupt** ไปยังหน่วยประมวลผล
2. **หน่วยประมวลผลทำคำสั่งปัจจุบันให้เสร็จสิ้น** ก่อนตอบสนองต่อ interrupt
3. **หน่วยประมวลผลทดสอบ interrupt** ตรวจพบว่ามี interrupt และส่งสัญญาณยืนยันไปยังอุปกรณ์
4. **หน่วยประมวลผลบันทึกข้อมูลที่จำเป็น** ข้อมูลขั้นต่ำที่ต้องการคือ:
   - สถานะของหน่วยประมวลผลซึ่งอยู่ใน **Program Status Word (PSW)**
   - ตำแหน่งของคำสั่งถัดไป ซึ่งอยู่ใน **Program Counter (PC)**
   - ข้อมูลทั้งสองนี้ถูก push ไปยัง system control stack
5. **หน่วยประมวลผลโหลด Program Counter** ด้วยตำแหน่งเข้าของโปรแกรม interrupt-handling

จากนั้น ซอฟต์แวร์ดำเนินการดังนี้:

6. **บันทึกเนื้อหาของรีจิสเตอร์ทั้งหมด** ไปยัง stack
7. **ประมวลผล interrupt** — รวมถึงการตรวจสอบข้อมูลสถานะที่เกี่ยวกับการดำเนินงาน I/O และอาจส่งคำสั่งหรือการยืนยันเพิ่มเติมไปยังอุปกรณ์ I/O
8. **เมื่อการประมวลผล interrupt เสร็จสิ้น** ค่ารีจิสเตอร์ที่บันทึกไว้จะถูกดึงออกจาก stack และคืนค่าสู่รีจิสเตอร์
9. **กระทำครั้งสุดท้าย:** คืนค่า PSW และค่า Program Counter จาก stack ส่งผลให้คำสั่งถัดไปที่จะดำเนินการมาจากโปรแกรมที่ถูกขัดจังหวะก่อนหน้า

### ประเด็นการออกแบบ (Design Issues)

ประเด็นการออกแบบสองประเด็นเกิดขึ้นในการนำ interrupt I/O ไปใช้:

1. **การระบุอุปกรณ์ที่ส่ง interrupt** — เมื่อ interrupt เกิดขึ้น หน่วยประมวลผลจะทราบได้อย่างไรว่าอุปกรณ์ใดส่ง interrupt มา?
2. **การจัดลำดับความสำคัญ** — หาก interrupt หลายรายการเกิดขึ้น หน่วยประมวลผลจะตัดสินใจดำเนินการรายการใดก่อน?

**เทคนิค 4 ประเภทสำหรับการระบุอุปกรณ์:**

| เทคนิค | คำอธิบาย | ข้อเสีย |
|--------|---------|--------|
| **Multiple interrupt lines** | มีสายขัดจังหวะหลายสายระหว่างหน่วยประมวลผลและโมดูล I/O | ไม่สามารถใช้งานได้จริงเมื่อต้องการสายจำนวนมาก |
| **Software poll** | หน่วยประมวลผลสอบถามโมดูล I/O แต่ละตัวเพื่อตรวจสอบว่าใครส่ง interrupt | ใช้เวลามาก |
| **Daisy chain (hardware poll)** | สัญญาณยืนยัน interrupt ส่งผ่านโมดูลแบบต่อเนื่อง | — |
| **Bus arbitration (vectored)** | โมดูล I/O ต้องได้รับการควบคุมบัสก่อนจะสามารถยก interrupt request line | — |

**Daisy Chain (Vectored Interrupt):**
- โมดูล I/O ทั้งหมดใช้สาย interrupt request ร่วมกัน
- สาย interrupt acknowledge ต่อแบบ daisy chain ผ่านโมดูลต่างๆ
- เมื่อหน่วยประมวลผลรับรู้ interrupt จะส่งสัญญาณ interrupt acknowledge
- สัญญาณนี้แพร่กระจายผ่านโมดูล I/O ต่างๆ จนถึงโมดูลที่ร้องขอ
- โมดูลที่ร้องขอวาง **vector** บนสายข้อมูล ซึ่งเป็นที่อยู่ของโมดูล I/O หรือตัวระบุเฉพาะอื่น
- หน่วยประมวลผลใช้ vector เป็น pointer ไปยัง device-service routine ที่เหมาะสม

### Intel 82C59A Interrupt Controller

Intel 80386 มีสาย Interrupt Request (INTR) เดียวและสาย Interrupt Acknowledge (INTA) เดียว เพื่อให้ 80386 จัดการอุปกรณ์หลากหลายประเภทและโครงสร้างลำดับความสำคัญ จึงมักกำหนดค่าด้วยตัวควบคุม interrupt ภายนอก คือ 82C59A

- 82C59A เดี่ยวสามารถจัดการโมดูลได้สูงสุด 8 โมดูล
- การจัดเรียงแบบ cascade สามารถจัดการโมดูลได้สูงสุด 64 โมดูล
- 82C59A รับคำขอ interrupt จากโมดูลที่เชื่อมต่อ กำหนดว่า interrupt ใดมีลำดับความสำคัญสูงสุด และส่งสัญญาณไปยังหน่วยประมวลผลผ่านสาย INTR
- หน่วยประมวลผลตอบรับผ่านสาย INTA ทำให้ 82C59A วางข้อมูล vector ที่เหมาะสมบนบัสข้อมูล

**โหมด Interrupt ที่เป็นไปได้:**
- **Fully nested** — คำขอ interrupt เรียงลำดับตามลำดับความสำคัญจาก IR0 (สูงสุด) ถึง IR7
- **Rotating** — ในแอปพลิเคชันที่อุปกรณ์หลายตัวมีลำดับความสำคัญเท่ากัน อุปกรณ์หลังจากได้รับบริการแล้วจะได้รับลำดับความสำคัญต่ำสุดในกลุ่ม
- **Special mask** — ช่วยให้หน่วยประมวลผลยับยั้ง interrupt จากอุปกรณ์บางชนิด

### Intel 8255A Programmable Peripheral Interface

8255A เป็นชิปโมดูล I/O เอนกประสงค์ออกแบบมาสำหรับใช้กับ Intel 80386 โดยเฉพาะ ใช้กันอย่างแพร่หลายในฐานะชิปตัวควบคุมอุปกรณ์ต่อพ่วง

**สายสัญญาณหลักของ 8255A:**

| สาย | คำอธิบาย |
|-----|---------|
| D0–D7 | สาย I/O ข้อมูล ข้อมูลทั้งหมดที่อ่านและเขียนใน 8255A ผ่านสาย 8 สายนี้ |
| CS (Chip Select) | ถ้า logic 0 หน่วยประมวลผลสามารถอ่านและเขียนใน 8255A ได้ |
| RD (Read Input) | ถ้า logic 0 และ CS เป็น logic 0 เปิดใช้งานเอาต์พุตข้อมูล 8255A บนบัสข้อมูลระบบ |
| WR (Write Input) | ถ้า logic 0 และ CS เป็น logic 0 เขียนข้อมูลไปยัง 8255A จากบัสข้อมูลระบบ |
| RESET | 8255A ถูกวางไว้ในสถานะรีเซ็ต พอร์ตต่อพ่วงทั้งหมดถูกตั้งค่าเป็นโหมดรับเข้า |
| PA0–PA7, PB0–PB7, PC0–PC7 | สาย I/O 8 บิต สามารถเชื่อมต่อกับอุปกรณ์ต่อพ่วง |
| A0, A1 | ผสมรวมแบบลอจิกกำหนดว่าเขียนหรืออ่านรีจิสเตอร์ภายใน 8255A ใด |

**โหมดการทำงานของ 8255A:**

| โหมด | คำอธิบาย |
|------|---------|
| **Mode 0** | I/O พื้นฐาน กลุ่มสายภายนอก 3 กลุ่ม (8 สาย) ทำหน้าที่เป็น 3 พอร์ต I/O ขนาด 8 บิต แต่ละพอร์ตสามารถกำหนดเป็นรับเข้าหรือส่งออกได้ |
| **Mode 1** | พอร์ต A และ B สามารถกำหนดเป็นรับเข้าหรือส่งออก สาย port C ทำหน้าที่เป็นสายควบคุมสำหรับ A และ B (handshaking และ interrupt request) |
| **Mode 2** | โหมดสองทิศทาง พอร์ต A สามารถกำหนดเป็นสาย I/O หรือสายส่งออกสำหรับการรับส่งข้อมูลสองทิศทาง |

## 7.5 Direct Memory Access (DMA)

### ข้อเสียของ Programmed และ Interrupt-Driven I/O

แม้ว่า Interrupt-driven I/O จะมีประสิทธิภาพมากกว่า Programmed I/O แต่ยังคงต้องการการแทรกแซงอย่างจริงจังของหน่วยประมวลผลในการถ่ายโอนข้อมูลระหว่างหน่วยความจำและโมดูล I/O ข้อเสียที่สืบทอดมา:

1. **อัตราการถ่ายโอน I/O จำกัด** โดยความเร็วที่หน่วยประมวลผลสามารถให้บริการอุปกรณ์ได้
2. **หน่วยประมวลผลถูกผูกในการจัดการการถ่ายโอน I/O** — ต้องรันคำสั่งจำนวนมากสำหรับการถ่ายโอน I/O แต่ละครั้ง

เมื่อต้องย้ายข้อมูลปริมาณมาก จำเป็นต้องใช้เทคนิคที่มีประสิทธิภาพมากขึ้น: **Direct Memory Access (DMA)**

### ฟังก์ชัน DMA

DMA เกี่ยวข้องกับโมดูลเพิ่มเติมบนบัสระบบ โมดูล DMA สามารถเลียนแบบหน่วยประมวลผลและยึดการควบคุมระบบจากหน่วยประมวลผล เพื่อถ่ายโอนข้อมูลไปยังและจากหน่วยความจำผ่านบัสระบบ

เทคนิคที่พบบ่อยกว่าคือ **cycle stealing** — โมดูล DMA บังคับให้หน่วยประมวลผลหยุดการทำงานชั่วคราว

**ข้อมูลที่ส่งจากหน่วยประมวลผลไปยังโมดูล DMA:**
- ว่าต้องการ read หรือ write ผ่านสายควบคุม
- ที่อยู่ของอุปกรณ์ I/O ที่เกี่ยวข้อง ส่งผ่านสายข้อมูล
- ตำแหน่งเริ่มต้นในหน่วยความจำสำหรับอ่านหรือเขียน เก็บในรีจิสเตอร์ที่อยู่ของโมดูล DMA
- จำนวนคำที่จะอ่านหรือเขียน เก็บในรีจิสเตอร์นับข้อมูล (data count register)

จากนั้นหน่วยประมวลผลดำเนินงานอื่นต่อไป โมดูล DMA ถ่ายโอนบล็อกข้อมูลทั้งหมดทีละคำ โดยตรงไปยังหรือจากหน่วยความจำ โดยไม่ผ่านหน่วยประมวลผล เมื่อการถ่ายโอนเสร็จสิ้น โมดูล DMA ส่งสัญญาณ interrupt ไปยังหน่วยประมวลผล ดังนั้นหน่วยประมวลผลจึงเกี่ยวข้องเฉพาะที่จุดเริ่มต้นและจุดสิ้นสุดของการถ่ายโอนเท่านั้น

### การกำหนดค่า DMA

```
(a) Single-bus, detached DMA
Processor <---> DMA <---> Memory
                 |
           I/O    I/O

(b) Single-bus, integrated DMA-I/O
Processor <---> DMA <---> Memory
                 |   \
                I/O  I/O

(c) I/O bus
Processor <---> DMA <---> Memory
                 |
           [I/O Bus]
          /    |    \
         I/O  I/O  I/O
```

- **Single-bus, detached DMA** — ทุกโมดูลใช้บัสระบบเดียวกัน DMA ใช้ Programmed I/O เพื่อแลกเปลี่ยนข้อมูลระหว่างหน่วยความจำและโมดูล I/O ผ่านโมดูล DMA มีประสิทธิภาพต่ำ (แต่ละการถ่ายโอนคำใช้ 2 bus cycles)

- **Single-bus, integrated DMA-I/O** — ลดจำนวน bus cycles ที่ต้องการได้มากโดยรวมฟังก์ชัน DMA และ I/O เข้าด้วยกัน

- **I/O bus** — เชื่อมต่อโมดูล I/O กับโมดูล DMA โดยใช้ I/O bus ลดจำนวนอินเทอร์เฟซ I/O ในโมดูล DMA เป็นหนึ่ง และให้การกำหนดค่าที่ขยายได้ง่าย

### Intel 8237A DMA Controller

8237A เชื่อมต่อกับตระกูลโปรเซสเซอร์ 80*86 และหน่วยความจำ DRAM เพื่อให้ความสามารถ DMA

**ลำดับขั้นตอนการทำงาน (ถ่ายโอนข้อมูลจากหน่วยความจำไปยังดิสก์):**

1. อุปกรณ์ต่อพ่วง (เช่น disk controller) ยก DREQ (DMA request) สูง
2. DMA ยก HRQ (hold request) สูง ส่งสัญญาณไปยัง CPU ผ่าน HOLD pin
3. CPU ทำ bus cycle ปัจจุบันให้เสร็จสิ้นและตอบสนองโดยยก HLDA (hold acknowledge) สูง บอก 8237 DMA ว่าสามารถใช้บัสได้
4. DMA เปิดใช้งาน DACK (DMA acknowledge) บอกอุปกรณ์ต่อพ่วงว่าจะเริ่มถ่ายโอนข้อมูล
5. DMA เริ่มถ่ายโอนข้อมูลจากหน่วยความจำไปยังอุปกรณ์ต่อพ่วง โดยวางที่อยู่ของไบต์แรกบน address bus และเปิดใช้งาน MEMR อ่านไบต์จากหน่วยความจำลงใน data bus แล้วเปิดใช้งาน IOW เขียนไปยังอุปกรณ์ต่อพ่วง จากนั้น DMA ลดค่าตัวนับและเพิ่มค่า address pointer ซ้ำกระบวนการนี้จนกว่าจำนวนนับถึงศูนย์
6. หลังจาก DMA ทำงานเสร็จ จะปิดใช้งาน HRQ ส่งสัญญาณให้ CPU ว่าสามารถยึดการควบคุมบัสกลับคืนได้

**ลักษณะของ 8237 DMA:**
- เป็น **fly-by DMA controller** — ข้อมูลที่ย้ายไม่ผ่านชิป DMA และไม่ถูกเก็บในชิป DMA
- สามารถถ่ายโอนข้อมูลระหว่างพอร์ต I/O และที่อยู่หน่วยความจำเท่านั้น (ไม่ใช่ระหว่างสองพอร์ต I/O หรือสองตำแหน่งหน่วยความจำโดยตรง)
- มี 4 DMA channels ที่กำหนดค่าได้อิสระ (channels 0, 1, 2, 3)

**รีจิสเตอร์ควบคุม/คำสั่งของ 8237A:**

| รีจิสเตอร์ | คำอธิบาย |
|-----------|---------|
| **Command** | หน่วยประมวลผลโหลดรีจิสเตอร์นี้เพื่อควบคุมการทำงานของ DMA D0 เปิดใช้งานการถ่ายโอน memory-to-memory D2 เปิด/ปิดใช้งาน DMA |
| **Status** | หน่วยประมวลผลอ่านรีจิสเตอร์นี้เพื่อกำหนดสถานะ DMA D0–D3 บ่งชี้ว่า channels 0–3 ถึง terminal count หรือไม่ D4–D7 บ่งชี้ว่า channel ใดมี DMA request รอดำเนินการ |
| **Mode** | กำหนดโหมดการทำงานของ DMA รวมถึงทิศทางการถ่ายโอน, auto-initialization, และโหมด single/block/demand/cascade |
| **Single Mask** | เปิด/ปิดใช้งาน DREQ input ของ channel เฉพาะ |
| **All Mask** | เหมือน Single Mask แต่สามารถ mask หรือ unmask ทั้ง 4 channels ด้วยการเขียนครั้งเดียว |

## 7.6 Direct Cache Access (DCA)

### ข้อจำกัดของ DMA สำหรับ I/O เครือข่ายความเร็วสูง

DMA พิสูจน์ว่าเป็นวิธีที่มีประสิทธิภาพในการเพิ่มประสิทธิภาพ I/O กับอุปกรณ์ต่อพ่วงและการรับส่งข้อมูล I/O เครือข่าย อย่างไรก็ตาม สำหรับอัตราที่เพิ่มขึ้นอย่างมากในการรับส่งข้อมูล I/O เครือข่าย DMA ไม่สามารถปรับขนาดได้ตามความต้องการที่เพิ่มขึ้น

แหล่งที่มาหลักของความต้องการนี้มาจากการใช้งาน Ethernet สวิตช์ขนาด 10 Gbps และ 100 Gbps อย่างแพร่หลายสำหรับการถ่ายโอนข้อมูลจำนวนมากไปยังและจากเซิร์ฟเวอร์ฐานข้อมูลและระบบประสิทธิภาพสูงอื่นๆ

ในส่วนนี้เราจะศึกษาว่าการเปิดใช้งานฟังก์ชัน I/O ให้มีการเข้าถึง cache โดยตรงสามารถเพิ่มประสิทธิภาพได้อย่างไร ซึ่งเป็นเทคนิคที่เรียกว่า **Direct Cache Access (DCA)** ตลอดส่วนนี้เราสนใจเฉพาะ cache ที่ใกล้กับหน่วยความจำหลักที่สุดซึ่งเรียกว่า **last-level cache** (L2 หรือ L3 cache)

### DMA โดยใช้ Shared Last-Level Cache

ในระบบ multicore ร่วมสมัย มีทั้ง cache ที่อุทิศให้กับแต่ละ core และ cache ที่แชร์กันระดับเพิ่มเติม ด้วยขนาด last-level cache ที่เพิ่มขึ้น ผู้ออกแบบระบบได้ปรับปรุงฟังก์ชัน DMA เพื่อให้ตัวควบคุม DMA มีการเข้าถึง shared cache ในลักษณะคล้ายกับ cores

#### สถาปัตยกรรม Intel Xeon E5-2600/4600

- กำหนดค่าด้วย cores สูงสุด 8 cores บนชิปเดียว
- แต่ละ core มี L1 และ L2 cache เฉพาะตัว
- มี shared L3 cache สูงสุด 20 MB แบ่งเป็น slices หนึ่งตัวต่อ core
- ใช้ระบบ **bidirectional high-speed ring interconnect** เชื่อมต่อ cores, last-level cache, PCIe และ integrated memory controller (IMC)

**การทำงานของ DMA กับ Cache:**
- สำหรับการส่งออก: เมื่อตัวควบคุม I/O ส่ง read request, MCH ตรวจสอบว่าข้อมูลอยู่ใน L3 cache หรือไม่ ถ้าใช่ MCH ส่งข้อมูลจาก cache ไปยังตัวควบคุม I/O โดยไม่ต้องเข้าถึงหน่วยความจำหลัก
- สำหรับการรับเข้า: ข้อมูลมาจากตัวควบคุม I/O และส่งผ่าน system ring ไปยัง MCH และเขียนออกไปยังหน่วยความจำหลัก

### ประเด็นประสิทธิภาพที่เกี่ยวข้องกับ Cache

การรับส่งข้อมูลเครือข่ายถูกส่งในรูปแบบของบล็อกโปรโตคอลที่เรียกว่า packets หรือ protocol data units โปรโตคอลชั้นต่ำสุดโดยทั่วไปคือ Ethernet ดังนั้นข้อมูลที่มาถึงและออกไปแต่ละบล็อกประกอบด้วย Ethernet packet ที่มี payload เป็น TCP/IP packet

**สองปัจจัยที่ลดประสิทธิภาพ:**
1. Core ใช้รอบนาฬิกาที่มีค่าในการคัดลอกข้อมูลระหว่าง system buffer และ application buffer
2. เนื่องจากความเร็วหน่วยความจำไม่ทัน CPU speed, core สูญเสียเวลาในการรอการอ่านและเขียนหน่วยความจำ

**ลำดับขั้นตอนสำหรับ incoming traffic:**
1. Packet มาถึง: NIC รับ Ethernet packet ที่เข้ามา ประมวลผลและตัด Ethernet control info ออก ส่ง TCP/IP packet ที่เหลือไปยัง DMA module
2. DMA: DMA module ถ่ายโอนข้อมูลรวมถึง packet descriptor ไปยังหน่วยความจำหลัก ต้องทำให้ cache lines ที่สอดคล้องกันไม่ถูกต้องด้วย
3. NIC ขัดจังหวะ host: หลังจาก packet จำนวนหนึ่งถูกถ่ายโอน NIC ส่ง interrupt ไปยัง host processor
4. ดึง descriptors และ headers: Core ประมวลผล interrupt อ่าน descriptor และ header ของ packets ที่ได้รับ
5. Cache miss เกิดขึ้น: Core ต้องหยุดเพื่ออ่านข้อมูลจากหน่วยความจำหลักเข้า cache แล้วไปยัง core registers
6. ประมวลผล Header: ซอฟต์แวร์โปรโตคอลรันบน core เพื่อวิเคราะห์เนื้อหาของ TCP และ IP headers
7. ถ่ายโอน Payload: ส่วนข้อมูลของ packet ถูกถ่ายโอนจาก system buffer ไปยัง application buffer ที่เหมาะสม

### กลยุทธ์ Direct Cache Access

กลยุทธ์ที่ง่ายที่สุดคือการส่ง prefetch hint ไปยัง core ทันทีที่ข้อมูลพร้อมใช้ใน system memory วิธีนี้ช่วยให้ core ดึงข้อมูล packet ล่วงหน้าจาก system buffer หลีกเลี่ยง cache miss และการสูญเสียรอบ core

กลยุทธ์ที่ก้าวหน้ากว่าคือการหลีกเลี่ยง system buffer ใน main memory โดยสมบูรณ์ โดยให้ระบบ I/O มีความสามารถไม่เพียงแต่เข้าถึง main memory โดยตรง แต่ยังเข้าถึง cache ด้วย ทั้งสำหรับการดำเนินงานรับเข้าและส่งออก จากนั้นจะเป็นไปได้ที่จะใช้ last-level cache แทน main memory เพื่อบัฟเฟอร์ packets และ descriptors

### Intel Direct Data I/O (DDIO)

DDIO ถูกนำไปใช้ในโปรเซสเซอร์ Xeon E5 ทั้งหมด

**การรับข้อมูล Packet (DDIO):**
- NIC เริ่มต้นการเขียนหน่วยความจำ
- NIC ทำให้ cache lines ที่สอดคล้องกับ system buffer ไม่ถูกต้องและฝากข้อมูลที่เข้ามาใน cache
- หลังจาก core ที่เหมาะสมรับสัญญาณ DCA interrupt, core สามารถอ่านข้อมูล packet ได้จาก cache

**กลยุทธ์ Cache Write ของ DDIO:**
- **Write update** — ถ้ามี cache hit ไลน์ cache ถูกอัปเดตแต่ไม่ใช่ main memory (write-back strategy)
- **Write allocate** — ถ้ามี cache miss การเขียนเกิดขึ้นในไลน์ใน cache ที่จะถูก write back ไปยัง main memory ในภายหลัง

**การส่งข้อมูล Packet (DDIO):**
- TCP/IP protocol handler สร้าง packet ที่จะส่งและเก็บใน allocated space ใน L3 cache (ไม่ใช่ใน main memory)
- Read operation ที่เริ่มต้นโดย NIC ได้รับข้อมูลจาก cache โดยไม่ทำให้เกิดการ eviction ไปยัง main memory

**เปรียบเทียบ DMA และ DDIO:**

```
DMA (Normal):
I/O controller --> Main memory --> Core (ผ่าน cache)
- หลายขั้นตอน
- Main memory เป็น bottleneck

DDIO (Input):
I/O controller --> Last-level cache --> Core
- น้อยขั้นตอน
- ไม่ต้องเข้า main memory

DDIO (Output):
Core --> Last-level cache --> I/O controller
- ข้อมูลอยู่ใน cache ตลอด
- ไม่มีการเขียนและอ่าน main memory ซ้ำซ้อน
```

## 7.7 I/O Channels และ Processors

### วิวัฒนาการของฟังก์ชัน I/O

ขั้นตอนวิวัฒนาการสามารถสรุปได้ดังนี้:

| ขั้น | คำอธิบาย |
|-----|---------|
| **1** | CPU ควบคุมอุปกรณ์ต่อพ่วงโดยตรง พบในอุปกรณ์ที่ควบคุมด้วยไมโครโปรเซสเซอร์อย่างง่าย |
| **2** | เพิ่ม controller หรือโมดูล I/O CPU ใช้ Programmed I/O โดยไม่มี interrupt CPU หลุดพ้นจากรายละเอียดเฉพาะของอินเทอร์เฟซอุปกรณ์ภายนอก |
| **3** | การกำหนดค่าเหมือนขั้น 2 แต่ใช้ interrupt CPU ไม่ต้องเสียเวลารอการดำเนินงาน I/O เพิ่มประสิทธิภาพ |
| **4** | โมดูล I/O ได้รับการเข้าถึง memory โดยตรงผ่าน DMA สามารถย้ายบล็อกข้อมูลไปยังหรือจาก memory โดยไม่เกี่ยวข้อง CPU ยกเว้นที่จุดเริ่มต้นและจุดสิ้นสุดของการถ่ายโอน |
| **5** | โมดูล I/O ได้รับการปรับปรุงให้เป็น processor ในสิทธิ์ของตัวเอง พร้อม instruction set เฉพาะทางสำหรับ I/O CPU สั่งให้ I/O processor รัน I/O program ใน memory |
| **6** | โมดูล I/O มีหน่วยความจำท้องถิ่นของตัวเองและเป็นคอมพิวเตอร์ในสิทธิ์ของตัวเอง อุปกรณ์ I/O จำนวนมากสามารถควบคุมได้โดยมีการมีส่วนร่วมของ CPU น้อยที่สุด |

### ลักษณะของ I/O Channels

**I/O channel** คือการขยายแนวคิด DMA I/O channel มีความสามารถในการรัน I/O instructions ซึ่งให้การควบคุมอย่างสมบูรณ์เหนือการดำเนินงาน I/O

ในระบบคอมพิวเตอร์ที่มีอุปกรณ์ดังกล่าว CPU ไม่รันคำสั่ง I/O คำสั่งดังกล่าวถูกเก็บในหน่วยความจำหลักเพื่อรันโดย processor เฉพาะทางใน I/O channel เอง ดังนั้น CPU เริ่มต้นการถ่ายโอน I/O โดยสั่งให้ I/O channel รันโปรแกรมใน memory โปรแกรมจะระบุ: อุปกรณ์หรืออุปกรณ์, พื้นที่หน่วยความจำสำหรับจัดเก็บ, ลำดับความสำคัญ และการดำเนินการสำหรับเงื่อนไขข้อผิดพลาดบางอย่าง

**สองประเภทของ I/O Channels:**

| ประเภท | คำอธิบาย |
|--------|---------|
| **Selector channel** | ควบคุมอุปกรณ์ความเร็วสูงหลายตัวและ ณ เวลาใดก็ตาม อุทิศให้กับการถ่ายโอนข้อมูลกับอุปกรณ์หนึ่งตัว |
| **Multiplexor channel** | สามารถจัดการ I/O กับอุปกรณ์หลายตัวพร้อมกัน สำหรับอุปกรณ์ความเร็วต่ำ ใช้ **byte multiplexor** รับหรือส่งตัวอักษรให้เร็วที่สุดเท่าที่เป็นไปได้ไปยังอุปกรณ์หลายตัว สำหรับอุปกรณ์ความเร็วสูง ใช้ **block multiplexor** สลับบล็อกข้อมูลจากอุปกรณ์หลายตัว |

## 7.8 มาตรฐานการเชื่อมต่อภายนอก (External Interconnection Standards)

### Universal Serial Bus (USB)

USB ใช้กันอย่างแพร่หลายสำหรับการเชื่อมต่ออุปกรณ์ต่อพ่วง เป็นอินเทอร์เฟซเริ่มต้นสำหรับอุปกรณ์ความเร็วต่ำ แต่ยังใช้กันทั่วไปสำหรับ I/O ความเร็วสูง

**วิวัฒนาการของ USB:**

| เวอร์ชัน | อัตราข้อมูล |
|---------|-----------|
| USB 1.0 | Low Speed: 1.5 Mbps, Full Speed: 12 Mbps |
| USB 2.0 | 480 Mbps |
| USB 3.0 (SuperSpeed) | 5 Gbps (signaling), ≤4 Gbps (usable) |
| USB 3.1 (SuperSpeed+) | 10 Gbps (signaling), 9.7 Gbps (theoretical usable) |

ระบบ USB ถูกควบคุมโดย root host controller ที่เชื่อมต่อกับอุปกรณ์เพื่อสร้างเครือข่ายท้องถิ่นที่มีโทโพโลยีแบบ hierarchical tree

### FireWire Serial Bus (IEEE 1394)

FireWire พัฒนาขึ้นเพื่อเป็นทางเลือกแทน SCSI สำหรับระบบขนาดเล็กเช่นคอมพิวเตอร์ส่วนตัว workstations และ servers

**คุณสมบัติหลักของ FireWire:**
- การกำหนดค่าแบบ daisy-chain โดยอุปกรณ์สูงสุด 63 ตัวเชื่อมต่อออกจาก port เดียว
- สามารถเชื่อมต่อ FireWire buses สูงสุด 1,022 บัสผ่าน bridges ทำให้ระบบรองรับอุปกรณ์ต่อพ่วงได้มากเท่าที่ต้องการ
- รองรับ **hot plugging** — สามารถเชื่อมต่อและยกเลิกการเชื่อมต่ออุปกรณ์ต่อพ่วงโดยไม่ต้องปิดคอมพิวเตอร์
- กำหนดค่าอัตโนมัติ — ไม่จำเป็นต้องตั้งค่า device ID ด้วยตัวเอง
- กำหนดค่าแบบ tree-structured
- อัตราข้อมูลตั้งแต่ 25 Mbps ถึง 3.2 Gbps

**ชั้นโปรโตคอล 3 ชั้น:**
1. **Physical layer** — กำหนดรูปแบบการส่ง คุณสมบัติไฟฟ้าและสัญญาณ
2. **Link layer** — อธิบายการส่งข้อมูลใน packets
3. **Transaction layer** — กำหนดโปรโตคอล request-response ที่ซ่อนรายละเอียดชั้นล่างของ FireWire จากแอปพลิเคชัน

### Small Computer System Interface (SCSI)

SCSI เป็นมาตรฐานสำหรับการเชื่อมต่ออุปกรณ์ต่อพ่วง (ดิสก์, โมเด็ม, เครื่องพิมพ์ ฯลฯ) กับคอมพิวเตอร์ขนาดเล็กและกลาง แม้ว่า SCSI จะพัฒนาไปสู่อัตราข้อมูลที่สูงขึ้น แต่สูญเสียความนิยมให้กับ USB และ FireWire ในระบบขนาดเล็ก อย่างไรก็ตาม SCSI ความเร็วสูงยังคงเป็นที่นิยมสำหรับการรองรับหน่วยความจำในระบบ enterprise

**คุณสมบัติ SCSI:**
- การจัดองค์กรแบบ shared bus รองรับอุปกรณ์สูงสุด 8 หรือ 32 ตัว
- การส่งข้อมูลแบบ parallel (bus width 16 บิตบนรุ่นก่อนหน้า, 32 บิตบนรุ่นหลัง)
- ความเร็วตั้งแต่ 5 Mbps (SCSI-1) ถึง 160 Mbps (SCSI-3 U3)

### Thunderbolt

Thunderbolt เป็นหนึ่งในเทคโนโลยีการเชื่อมต่ออุปกรณ์ต่อพ่วงที่ล่าสุดและเร็วที่สุด พัฒนาโดย Intel ร่วมกับ Apple

- สาย Thunderbolt เดียวสามารถจัดการงานที่ก่อนหน้านี้ต้องใช้หลายสาย
- รวมข้อมูล วิดีโอ เสียง และพลังงานในการเชื่อมต่อความเร็วสูงเดียว
- รองรับการรับส่งข้อมูลสูงสุด 10 Gbps ในแต่ละทิศทาง
- จ่ายพลังงานสูงสุด 10 watts ไปยังอุปกรณ์ต่อพ่วงที่เชื่อมต่อ

### InfiniBand

InfiniBand เป็นข้อกำหนด I/O สำหรับตลาดเซิร์ฟเวอร์ขั้นสูง กำหนดสถาปัตยกรรมและข้อกำหนดสำหรับการไหลของข้อมูลระหว่าง processors และอุปกรณ์ I/O อัจฉริยะ

- เชื่อมต่อ servers, remote storage และอุปกรณ์เครือข่ายอื่นๆ ในสวิตช์และลิงก์กลาง
- สถาปัตยกรรมที่ใช้สวิตช์สามารถเชื่อมต่อได้สูงสุด 64,000 servers, ระบบจัดเก็บข้อมูล และอุปกรณ์เครือข่าย
- เป็นอินเทอร์เฟซยอดนิยมสำหรับ storage area networking และการกำหนดค่าจัดเก็บข้อมูลขนาดใหญ่

### PCI Express

PCI Express เป็นระบบบัสความเร็วสูงสำหรับการเชื่อมต่ออุปกรณ์ต่อพ่วงหลากหลายประเภทและความเร็ว

### SATA (Serial Advanced Technology Attachment)

- อินเทอร์เฟซสำหรับระบบจัดเก็บข้อมูลดิสก์
- อัตราข้อมูลสูงสุด 6 Gbps (สูงสุด 300 Mbps ต่ออุปกรณ์)
- ใช้กันอย่างแพร่หลายในคอมพิวเตอร์ตั้งโต๊ะ และในแอปพลิเคชัน industrial และ embedded

### Ethernet

Ethernet เป็นเทคโนโลยีเครือข่ายแบบมีสายที่โดดเด่น ใช้ในบ้าน สำนักงาน ศูนย์ข้อมูล และเครือข่ายองค์กร

**ลำดับเวลาของ Ethernet:**

| ปี | ความเร็ว |
|----|---------|
| 1983 | 10 Mbps |
| 1995 | 100 Mbps |
| 1998 | 1 Gbps |
| 2003 | 10 Gbps |
| 2010 | 40 Gbps และ 100 Gbps |

### Wi-Fi (IEEE 802.11)

Wi-Fi เป็นเทคโนโลยีการเข้าถึงอินเทอร์เน็ตแบบไร้สายที่โดดเด่น

**ลำดับเวลาของ Wi-Fi:**

| มาตรฐาน | ปี | อัตราข้อมูลสูงสุด |
|---------|---|-----------------|
| 802.11 | 1997 | 2 Mbps |
| 802.11a | 1999 | 54 Mbps |
| 802.11b | 1999 | 11 Mbps |
| 802.11n | 1999 | 600 Mbps |
| 802.11g | 2003 | 54 Mbps |
| 802.11ad | 2012 | 6.76 Gbps |
| 802.11ac | 2014 | 3.2 Gbps |

## 7.9 โครงสร้าง I/O ของ IBM zEnterprise EC12

zEnterprise EC12 เป็นเมนเฟรมคอมพิวเตอร์ล่าสุดของ IBM ระบบนี้ใช้ชิป zEC12 processor ซึ่งเป็นชิป multicore ที่ทำงานที่ 5.5 GHz พร้อม 6 cores สถาปัตยกรรม zEC12 สามารถมี processor chips สูงสุด 101 ชิป รวม 606 cores

### โครงสร้าง Channel

zEnterprise EC12 มีระบบย่อย I/O เฉพาะทางที่จัดการการดำเนินงาน I/O ทั้งหมด โดยแยกภาระการประมวลผลและหน่วยความจำออกจาก processors หลักอย่างสมบูรณ์

**องค์ประกอบของ Channel Subsystem (CSS) แต่ละตัว:**

| องค์ประกอบ | คำอธิบาย |
|----------|---------|
| **System Assist Processor (SAP)** | Core processor ที่กำหนดค่าสำหรับการดำเนินงาน I/O ทำหน้าที่ offload การดำเนินงาน I/O และจัดการ channels และ I/O operations queues |
| **Hardware System Area (HSA)** | ส่วนสงวนของ system memory ที่มีการกำหนดค่า I/O ใช้โดย SAPs สงวนขนาดคงที่ 32 GB ซึ่งไม่เป็นส่วนหนึ่งของ memory ที่ลูกค้าซื้อ |
| **Logical partitions** | รูปแบบของ virtual machine โดยสาระแก่นสาร คือ logical processor ที่กำหนดในระดับ operating system แต่ละ CSS รองรับ logical partitions สูงสุด 16 ตัว |
| **Subchannels** | subchannel ปรากฏกับโปรแกรมเป็น logical device และมีข้อมูลที่จำเป็นในการดำเนินงาน I/O รองรับสูงสุด 196k subchannels ต่อ CSS |
| **Channel path** | อินเทอร์เฟซเดียวระหว่าง channel subsystem และ control units หนึ่งหรือมากกว่าผ่าน channel แต่ละ CSS มี channel paths สูงสุด 256 ตัว |
| **Channel** | Processors ขนาดเล็กที่สื่อสารกับ I/O control units จัดการการถ่ายโอนข้อมูลระหว่าง memory และอุปกรณ์ภายนอก |

### การจัดองค์กรระบบ I/O

**ข้อกำหนดทางกายภาพของ zEnterprise EC12:**
- น้ำหนัก: 2,430 kg (5,358 lbs)
- ความกว้าง: 1.568 m (5.14 ft)
- ความลึก: 1.69 m (6.13 ft)
- ความสูง: 2.015 m (6.6 ft)

ระบบประกอบด้วยช่องใหญ่สองช่องที่เรียกว่า frames:

**A Frame (ขวา):**
- Upper cage: Processor cage พร้อม 4 slots สำหรับ processor books สูงสุด 4 เล่ม แต่ละ book มี multichip module (MCM), memory cards และ cage connections
- Lower cage: I/O cage มีฮาร์ดแวร์ I/O รวมถึง multiplexors และ channels

**Z Frame (ซ้าย):**
- แบตเตอรี่ภายในและแหล่งจ่ายไฟ
- Slots สำหรับ support elements
- Slots สำหรับ I/O drawers หนึ่งหรือมากกว่า

**โครงสร้างพื้นฐาน I/O:**
- แต่ละ zEC12 processor book รองรับ 2 infrastructure I/O ภายใน:
  - **InfiniBand** สำหรับ I/O cages และ I/O drawers
  - **PCI Express (PCIe)** สำหรับ I/O drawers

- **InfiniBand connections** — ผ่าน Host Channel Adapter (HCA) fanout ซึ่งมี InfiniBand links ไปยัง InfiniBand multiplexors ใน I/O cage หรือ drawer
- **PCIe connections** — ผ่าน PCIe fanout ไปยัง PCIe switches ที่สามารถเชื่อมต่อกับ I/O device controllers จำนวนหนึ่ง

แต่ละ book มี InfiniBand HCA และ PCIe fanouts รวมสูงสุด 8 ตัว แต่ละ fanout รองรับการเชื่อมต่อสูงสุด 32 รายการ รวมสูงสุด 256 การเชื่อมต่อต่อ processor book แต่ละการเชื่อมต่อถูกควบคุมโดย channel processor

## 7.10 คำศัพท์สำคัญ คำถามทบทวน และแบบฝึกหัด

### คำศัพท์สำคัญ (Key Terms)

| คำศัพท์ | ความหมาย |
|--------|---------|
| **Cache injection** | กระบวนการแทรกข้อมูลจาก I/O ไปยัง cache โดยตรง |
| **Cycle stealing** | เทคนิคที่ DMA module ยืม bus cycle จากหน่วยประมวลผลชั่วคราว |
| **Direct Cache Access (DCA)** | เทคนิคที่อนุญาตให้ฟังก์ชัน I/O เข้าถึง cache โดยตรง |
| **Direct Data I/O (DDIO)** | การนำ DCA ไปใช้งานของ Intel ใน Xeon processor |
| **Direct Memory Access (DMA)** | เทคนิคที่โมดูล I/O ถ่ายโอนข้อมูลโดยตรงกับหน่วยความจำโดยไม่ผ่าน CPU |
| **InfiniBand** | ข้อกำหนด I/O สำหรับตลาดเซิร์ฟเวอร์ขั้นสูง |
| **Interrupt** | สัญญาณที่บอกให้ CPU หยุดงานปัจจุบันและประมวลผลเหตุการณ์สำคัญ |
| **Interrupt-driven I/O** | เทคนิค I/O ที่อุปกรณ์ส่งสัญญาณ interrupt เมื่อพร้อมถ่ายโอนข้อมูล |
| **I/O channel** | โมดูล I/O ที่สามารถรัน I/O instructions เองได้ |
| **I/O command** | คำสั่งที่หน่วยประมวลผลส่งไปยังโมดูล I/O |
| **I/O module** | ส่วนประกอบที่เชื่อมต่อ CPU และหน่วยความจำกับอุปกรณ์ต่อพ่วง |
| **I/O processor** | โมดูล I/O ที่มีหน่วยความจำท้องถิ่นและสามารถรันโปรแกรมได้อิสระ |
| **Isolated I/O** | ระบบที่มีพื้นที่ที่อยู่ I/O แยกจากพื้นที่หน่วยความจำ |
| **Last-level cache** | Cache ที่ใกล้กับ main memory มากที่สุด (L2 หรือ L3) |
| **Memory-mapped I/O** | ระบบที่ใช้พื้นที่ที่อยู่ร่วมกันสำหรับหน่วยความจำและอุปกรณ์ I/O |
| **Multiplexor channel** | I/O channel ที่จัดการ I/O กับอุปกรณ์หลายตัวพร้อมกัน |
| **Non-write allocate** | เทคนิค cache ที่แก้ไข block โดยตรงใน main memory เมื่อ write miss |
| **Peripheral device** | อุปกรณ์ภายนอกที่เชื่อมต่อกับโมดูล I/O |
| **Programmed I/O** | เทคนิค I/O ที่ CPU ควบคุมการดำเนินงาน I/O โดยตรงและรอจนเสร็จ |
| **Selector channel** | I/O channel ที่ควบคุมอุปกรณ์ความเร็วสูงหลายตัวแต่ทำงานกับหนึ่งตัวต่อครั้ง |
| **Thunderbolt** | เทคโนโลยีการเชื่อมต่ออุปกรณ์ต่อพ่วงความเร็วสูงพัฒนาโดย Intel และ Apple |
| **Write allocate** | เทคนิค cache ที่โหลด line จาก main memory เข้า cache ก่อนอัปเดต |
| **Write back** | นโยบาย cache ที่อัปเดตเฉพาะใน cache, เขียนกลับ main memory เมื่อ replace |
| **Write through** | นโยบาย cache ที่เขียนทั้งใน cache และ main memory พร้อมกัน |
| **Write update** | การเรียกของ Intel สำหรับกลยุทธ์ write-back ใน DDIO เมื่อ cache hit |

   b. ประมาณอัตราเดียวกันถ้าใช้ DMA

**7.20** แหล่งข้อมูลผลิตตัวอักษร IRA 7 บิต โดยแต่ละตัวมี parity bit ต่อท้าย หา expression สำหรับอัตราข้อมูลที่มีประสิทธิภาพสูงสุด (อัตราข้อมูล IRA) บนสาย R-bps สำหรับ:

   a. Asynchronous transmission พร้อม stop bit 1.5 หน่วย
   b. Bit-synchronous transmission พร้อม frame ประกอบด้วย control bits 48 บิตและ information bits 128 บิต
   c. เหมือน (b) พร้อม information field ขนาด 1024 บิต
   d. Character-synchronous พร้อม 9 control characters ต่อ frame และ 16 information characters
   e. เหมือน (d) พร้อม 128 information characters

## สรุปบทที่ 7

บทนี้ครอบคลุมการออกแบบและการดำเนินงานของระบบ I/O ซึ่งเป็นองค์ประกอบสำคัญลำดับที่สามของสถาปัตยกรรมคอมพิวเตอร์

**ประเด็นสำคัญที่ควรจำ:**

1. **โมดูล I/O** จำเป็นเพราะความหลากหลายของอุปกรณ์, ความไม่ตรงกันของอัตราข้อมูล และความแตกต่างของรูปแบบข้อมูล

2. **Programmed I/O** ง่ายแต่สิ้นเปลืองทรัพยากร CPU เนื่องจากต้องวนซ้ำตรวจสอบสถานะ

3. **Interrupt-driven I/O** มีประสิทธิภาพมากกว่า CPU สามารถทำงานอื่นได้ขณะรอ I/O แต่ยังต้องผ่าน CPU สำหรับการถ่ายโอนข้อมูลแต่ละครั้ง

4. **DMA** เหมาะสำหรับการถ่ายโอนข้อมูลปริมาณมาก ถ่ายโอนบล็อกข้อมูลโดยตรงระหว่าง I/O และ memory โดยไม่ผ่าน CPU

5. **DCA/DDIO** เป็นเทคนิคขั้นสูงที่เปิดใช้งาน I/O ให้เข้าถึง cache โดยตรง ช่วยเพิ่มประสิทธิภาพสำหรับ high-speed network I/O

6. **I/O Channels** เป็นวิวัฒนาการขั้นสูงที่ให้ I/O module มีความสามารถรัน programs อิสระ ลดภาระของ CPU อย่างมาก

7. **มาตรฐานการเชื่อมต่อ** เช่น USB, FireWire, SATA, Thunderbolt, InfiniBand และ Ethernet มีบทบาทสำคัญในการเชื่อมต่ออุปกรณ์ภายนอกกับระบบคอมพิวเตอร์
