

\# บทที่ 8: การสนับสนุนของระบบปฏิบัติการ (Operating System Support) 



\## 8.1 ภาพรวมของระบบปฏิบัติการ (Operating System Overview) 



ระบบปฏิบัติการ (OS) คือโปรแกรมที่ทำหน้าที่ควบคุมการรันแอปพลิเคชันโปรแกรม และเป็นตัวกลาง (Interface) ระหว่างแอปพลิเคชันกับฮาร์ดแวร์คอมพิวเตอร์  โดยมีวัตถุประสงค์หลัก 2 ประการ ได้แก่:

\* \*\*ความสะดวก (Convenience):\*\* ทำให้คอมพิวเตอร์ใช้งานได้สะดวกยิ่งขึ้น 

\* \*\*ความมีประสิทธิภาพ (Efficiency):\*\* ช่วยให้ทรัพยากรของระบบคอมพิวเตอร์ถูกใช้งานอย่างมีประสิทธิภาพสูงสุด 



\### บริการของระบบปฏิบัติการ 

โดยทั่วไป OS จะให้บริการในด้านต่างๆ ดังนี้:

\* \*\*การสร้างโปรแกรม (Program creation):\*\* มีเครื่องมือและบริการต่างๆ เช่น Editor และ Debugger เพื่อช่วยโปรแกรมเมอร์ 

\* \*\*การรันโปรแกรม (Program execution):\*\* จัดการโหลดคำสั่งและข้อมูลลงในหน่วยความจำหลัก กำหนดค่าอุปกรณ์ I/O และเตรียมทรัพยากรต่างๆ 

\* \*\*การเข้าถึงอุปกรณ์ I/O (Access to I/O devices):\*\* จัดการรายละเอียดเฉพาะของอุปกรณ์ I/O เพื่อให้โปรแกรมเมอร์ใช้งานได้ง่ายๆ เพียงแค่สั่ง Read หรือ Write 

\* \*\*การควบคุมการเข้าถึงไฟล์ (Controlled access to files):\*\* จัดการรูปแบบของไฟล์บนสื่อจัดเก็บข้อมูล และมีกลไกป้องกันการเข้าถึงไฟล์ในระบบที่มีผู้ใช้หลายคน 

\* \*\*การเข้าถึงระบบ (System access):\*\* ควบคุมการเข้าถึงระบบส่วนรวมและทรัพยากรเฉพาะ รวมถึงปกป้องข้อมูลจากผู้ใช้ที่ไม่ได้รับอนุญาต 

\* \*\*การตรวจจับและตอบสนองต่อข้อผิดพลาด (Error detection and response):\*\* จัดการกับข้อผิดพลาดของฮาร์ดแวร์ (เช่น หน่วยความจำพัง) และซอฟต์แวร์ (เช่น Arithmetic overflow) โดยให้กระทบต่อแอปพลิเคชันที่รันอยู่น้อยที่สุด 

\* \*\*การทำบัญชี (Accounting):\*\* รวบรวมสถิติการใช้งานทรัพยากรเพื่อใช้ในการปรับปรุงประสิทธิภาพและสำหรับการคิดค่าบริการ 



\### อินเทอร์เฟซหลัก 3 ระดับ 

1\. \*\*Instruction Set Architecture (ISA):\*\* กำหนดชุดคำสั่งภาษาเครื่อง เป็นขอบเขตระหว่างฮาร์ดแวร์และซอฟต์แวร์ 

2\. \*\*Application Binary Interface (ABI):\*\* กำหนดมาตรฐานสำหรับการพกพาไบนารีข้ามโปรแกรม และกำหนดอินเทอร์เฟซการเรียกใช้ระบบ (System call) 

3\. \*\*Application Programming Interface (API):\*\* ให้โปรแกรมเข้าถึงทรัพยากรฮาร์ดแวร์ผ่าน High-level language (HLL) library calls 



\### ระบบปฏิบัติการในฐานะผู้จัดการทรัพยากร (Resource Manager) 

OS ควบคุมการทำงานของคอมพิวเตอร์ในรูปแบบที่แปลกกว่าระบบควบคุมทั่วไป คือ OS เป็นโปรแกรมที่โปรเซสเซอร์ต้องรัน  และ OS จะต้อง "สละสิทธิ์" (Relinquish control) การควบคุมโปรเซสเซอร์ให้กับโปรแกรมอื่นเพื่อทำงานที่มีประโยชน์ จากนั้นจึงกลับมาควบคุมใหม่ 



\### ประเภทของระบบปฏิบัติการ 

\* \*\*Interactive System:\*\* ผู้ใช้หรือโปรแกรมเมอร์โต้ตอบกับคอมพิวเตอร์โดยตรงผ่านคีย์บอร์ดหรือหน้าจอระหว่างที่โปรแกรมกำลังทำงาน 

\* \*\*Batch System:\*\* นำโปรแกรมของผู้ใช้หลายๆ คนมารวมกันเป็นแบตช์ (Batch) แล้วส่งให้เจ้าหน้าที่คอมพิวเตอร์รันโดยไม่มีการโต้ตอบ 

\* \*\*Multiprogramming:\*\* พยายามทำให้โปรเซสเซอร์ยุ่งอยู่เสมอโดยการทำงานหลายโปรแกรมพร้อมกัน โดยโหลดหลายโปรแกรมไว้ในหน่วยความจำแล้วสลับการทำงานไปมาอย่างรวดเร็ว 

\* \*\*Uniprogramming:\*\* ทำงานได้เพียงครั้งละหนึ่งโปรแกรมเท่านั้น 



\*\*วิวัฒนาการ:\*\*

\* \*\*ยุคแรก (Early Systems):\*\* ผู้ใช้ทำงานกับฮาร์ดแวร์โดยตรง (Serial processing) ไม่มี OS ทำให้เสียเวลาไปกับการตั้งค่า (Setup time) และการรอใช้เครื่อง 

\* \*\*Simple Batch Systems:\*\* ใช้โปรแกรม Monitor (Resident monitor) เพื่อช่วยสลับงานอัตโนมัติ ควบคุมผ่าน Job Control Language (JCL) 

\* \*\*Multiprogrammed Batch Systems:\*\* แก้ปัญหา I/O ที่ทำงานช้ากว่าโปรเซสเซอร์ โดยเก็บหลายโปรแกรมไว้ในหน่วยความจำ เมื่อโปรแกรมหนึ่งรอ I/O โปรเซสเซอร์จะสลับไปทำงานโปรแกรมอื่นทันที 

\* \*\*Time-Sharing Systems:\*\* นำ Multiprogramming มาใช้เพื่อรองรับงานแบบ Interactive หลายๆ งานพร้อมกัน โดยผู้ใช้แต่ละคนจะแชร์เวลาการทำงานของโปรเซสเซอร์ผ่านเทอร์มินัล

\## 8.2 การจัดตารางเวลา (Scheduling) 

หัวใจสำคัญของ Multiprogramming คือการจัดตารางเวลา ซึ่งมักเกี่ยวข้องกับ 4 ประเภทหลัก



1\. \*\*Long-Term Scheduling:\*\* ตัดสินใจว่าจะยอมรับโปรแกรมใดเข้าสู่ระบบเพื่อเป็น "โปรเซส (Process)" เป็นการควบคุมระดับของ Multiprogramming 

2\. \*\*Medium-Term Scheduling:\*\* เกี่ยวข้องกับฟังก์ชันการสลับเปลี่ยนหน่วยความจำ (Swapping) เข้าและออก 

3\. \*\*Short-Term Scheduling (Dispatcher):\*\* ทำงานบ่อยที่สุดเพื่อตัดสินใจว่าโปรเซสใดที่มีความพร้อม (อยู่ในสถานะ Ready) จะได้เข้าใช้โปรเซสเซอร์เป็นลำดับถัดไป 

4\. \*\*I/O Scheduling:\*\* ตัดสินใจว่าโปรเซสใดจะได้ใช้อุปกรณ์ I/O 



\### สถานะของโปรเซส (Process States) 

สถานะของโปรเซสมีการเปลี่ยนแปลงตลอดวงจรชีวิต โดยมีอย่างน้อย 5 สถานะหลักดังนี้

\* \*\*New:\*\* โปรแกรมเพิ่งเข้าสู่ระบบ แต่ยังไม่พร้อมทำงาน 

\* \*\*Ready:\*\* โปรเซสพร้อมทำงานและกำลังรอการเข้าถึงโปรเซสเซอร์ 

\* \*\*Running:\*\* โปรเซสกำลังถูกรันโดยโปรเซสเซอร์ 

\* \*\*Waiting / Blocked:\*\* โปรเซสถูกพักการรันเพื่อรอทรัพยากรบางอย่าง เช่น รออุปกรณ์ I/O 

\* \*\*Halted / Exit:\*\* โปรเซสทำงานเสร็จสิ้นและจะถูกทำลายโดย OS 



\## 8.3 การจัดการหน่วยความจำ (Memory Management) 



ในระบบ Uniprogramming หน่วยความจำหลักจะถูกแบ่งออกเป็นสองส่วน: ส่วนหนึ่งสำหรับระบบปฏิบัติการ (OS หรือ Resident monitor) และอีกส่วนหนึ่งสำหรับโปรแกรมที่กำลังทำงานอยู่ในขณะนั้น แต่ในระบบ Multiprogramming ส่วนพื้นที่ของ "ผู้ใช้" ในหน่วยความจำจะถูกแบ่งย่อยลงไปอีกเพื่อรองรับหลายๆ โปรเซสพร้อมกัน  งานแบ่งส่วนนี้จะถูกจัดการแบบไดนามิกโดย OS ซึ่งกระบวนการนี้เรียกว่า \*\*การจัดการหน่วยความจำ (Memory management)\*\* 



การจัดการหน่วยความจำที่มีประสิทธิภาพถือเป็นหัวใจสำคัญในระบบ Multiprogramming  หากมีโปรเซสอยู่ในหน่วยความจำเพียงไม่กี่ตัว เวลาส่วนใหญ่โปรเซสเหล่านั้นก็จะหมดไปกับการรอ I/O ทำให้โปรเซสเซอร์ว่างงาน  ดังนั้น หน่วยความจำจึงจำเป็นต้องถูกจัดสรรอย่างมีประสิทธิภาพ เพื่อที่จะสามารถบรรจุโปรเซสลงในหน่วยความจำให้ได้มากที่สุดเท่าที่จะทำได้ 



\### การสลับเปลี่ยน (Swapping) 



จากที่เราได้พูดถึงคิว 3 ประเภทกันไปแล้ว (คิวระยะยาว, คิวระยะสั้น, และคิวอุปกรณ์ I/O)  เหตุผลที่ต้องมีกลไกคิวที่ซับซ้อนนี้ ก็เพราะกิจกรรม I/O นั้นทำงานช้ากว่าการประมวลผลมาก ทำให้โปรเซสเซอร์มักจะว่างงาน  แต่ระบบคิวนั้นก็ไม่ได้แก้ปัญหาได้ทั้งหมด  แม้ว่าหน่วยความจำจะเก็บได้หลายโปรเซส และโปรเซสเซอร์สามารถสลับไปทำงานโปรเซสอื่นได้เมื่อมีโปรเซสกำลังรอ I/O ก็ตาม  แต่ความเร็วที่ต่างกันมาก ทำให้เป็นเรื่องปกติที่ทุกโปรเซสในหน่วยความจำอาจจะอยู่ในสถานะกำลังรอ I/O พร้อมกันหมดเลยก็ได้ ส่งผลให้โปรเซสเซอร์ก็ยังคงว่างงานอยู่ดี 



วิธีแก้ปัญหาอาจทำได้โดยการขยายหน่วยความจำหลักให้ใหญ่ขึ้นเพื่อรับโปรเซสได้มากขึ้น  แต่วิธีนี้มีข้อบกพร่อง 2 อย่างคือ:

1\. หน่วยความจำหลักมีราคาแพง 

2\. ความต้องการใช้หน่วยความจำของโปรแกรมก็เติบโตขึ้นอย่างรวดเร็วพอๆ กับที่ราคาหน่วยความจำถูกลง  ทำให้หน่วยความจำที่ใหญ่ขึ้น กลับกลายเป็นการรองรับโปรเซสที่มีขนาดใหญ่ขึ้น แทนที่จะได้จำนวนโปรเซสเพิ่มขึ้น 



อีกทางออกหนึ่งคือ \*\*การสลับเปลี่ยน (Swapping)\*\* 

\* ระบบจะมีคิวระยะยาวสำหรับเก็บคำขอโปรเซส ซึ่งมักจะเก็บไว้ในพื้นที่ดิสก์  

\* คำขอเหล่านี้จะถูกนำเข้ามาในหน่วยความจำหลักทีละรายการเมื่อมีพื้นที่ว่าง  และเมื่อทำงานเสร็จก็จะถูกย้ายออกไป 

\* ในกรณีที่ไม่มีโปรเซสใดในหน่วยความจำอยู่ในสถานะ "พร้อม (Ready)"  แทนที่โปรเซสเซอร์จะหยุดรอ OS จะทำการสลับ (Swap) โปรเซสตัวใดตัวหนึ่งออกไปยังดิสก์เพื่อไปเก็บใน "คิวระดับกลาง (Intermediate queue)" ซึ่งเป็นคิวพักโปรเซสที่ถูกเตะออกจากหน่วยความจำชั่วคราว 







อย่างไรก็ตาม การทำ Swapping ถือเป็นฟังก์ชัน I/O ประเภทหนึ่ง ซึ่งอาจมีความเสี่ยงที่จะทำให้ปัญหาประสิทธิภาพแย่ลงกว่าเดิม  แต่เนื่องจาก I/O ของดิสก์มักจะทำงานได้เร็วที่สุดในระบบเมื่อเทียบกับ I/O ตัวอื่น การ Swapping จึงมักช่วยเพิ่มประสิทธิภาพได้ในท้ายที่สุด 



\### การแบ่งพาร์ติชัน (Partitioning) 



วิธีที่ง่ายที่สุดในการแบ่งหน่วยความจำคือการใช้ \*\*พาร์ติชันขนาดคงที่ (Fixed-size partitions)\*\*  ซึ่งพาร์ติชันเหล่านี้ไม่จำเป็นต้องมีขนาดเท่ากันเสมอไป เมื่อโปรเซสถูกดึงเข้าสู่หน่วยความจำ มันจะถูกนำไปวางในพาร์ติชันที่ว่างและมีขนาดเล็กที่สุดที่สามารถรองรับตัวมันได้  แต่วิธีนี้ก็จะยังมีพื้นที่สูญเปล่าอยู่ดี เพราะโดยส่วนใหญ่แล้ว โปรเซสมักไม่ได้ใช้พื้นที่หน่วยความจำพอดีเป๊ะตามที่พาร์ติชันจัดเตรียมไว้ให้ (ทำให้เกิดพื้นที่เหลือทิ้ง) 



วิธีที่มีประสิทธิภาพกว่าคือการใช้ \*\*พาร์ติชันแบบปรับขนาดได้ (Variable-size partitions)\*\*  เมื่อโปรเซสเข้ามาในหน่วยความจำ มันจะถูกจัดสรรพื้นที่ให้พอดีกับที่มันต้องการเป๊ะๆ โดยไม่มีขาดหรือเกิน 

\* \*\*ปัญหา:\*\* วิธีนี้เริ่มต้นได้ดี แต่ท้ายที่สุดจะทำให้เกิดช่องโหว่เล็กๆ (Holes) จำนวนมากกระจายอยู่ทั่วหน่วยความจำ  เมื่อเวลาผ่านไป หน่วยความจำจะยิ่งกระจัดกระจาย (Fragmented) และการใช้งานจะลดลง 

\* \*\*วิธีแก้:\*\* สามารถแก้ได้ด้วยเทคนิค \*\*Compaction (การบีบอัด/จัดระเบียบ)\*\* โดย OS จะทำการขยับโปรเซสต่างๆ ในหน่วยความจำเป็นระยะๆ เพื่อดึงพื้นที่ว่างทั้งหมดมารวมกันเป็นก้อนเดียว  ซึ่งขั้นตอนนี้จะใช้เวลาและสิ้นเปลืองทรัพยากรโปรเซสเซอร์ค่อนข้างมาก 



เพื่อแก้ปัญหาเรื่องตำแหน่งของโปรเซสที่มีการเปลี่ยนไปมาตลอดเวลาจากการ Swapping และ Compaction  OS จะใช้กลไกการแปลงที่อยู่ โดยแยกระหว่าง \*\*Logical Address (ที่อยู่ตรรกะ)\*\* ซึ่งเป็นตำแหน่งที่อยู่สัมพัทธ์เทียบกับจุดเริ่มต้นของโปรแกรม  และ \*\*Physical Address (ที่อยู่กายภาพ)\*\* ซึ่งเป็นตำแหน่งจริงๆ ในหน่วยความจำหลัก  ฮาร์ดแวร์โปรเซสเซอร์จะทำการแปลงที่อยู่นี้ให้โดยอัตโนมัติขณะรันโปรแกรม 



\### การแบ่งหน้า (Paging) 



เพื่อแก้ปัญหาประสิทธิภาพของ Partitioning จึงเกิดแนวคิดการแบ่งหน่วยความจำออกเป็นก้อนเล็กๆ ขนาดเท่าๆ กัน และแบ่งโปรเซสออกเป็นก้อนเล็กๆ ขนาดเท่าๆ กันเช่นกัน 

\* ก้อนย่อยของโปรแกรม เราจะเรียกว่า \*\*หน้า (Pages)\*\* 

\* ก้อนย่อยของหน่วยความจำหลัก เราจะเรียกว่า \*\*เฟรม (Frames หรือ Page frames)\*\* 

\* OS จะทำหน้าที่บรรจุ Pages ลงไปใน Frames ที่ยังว่างอยู่ 







ด้วยวิธี Paging พื้นที่หน่วยความจำที่อาจสูญเปล่าสำหรับโปรเซสนั้นๆ จะมีขนาดมากที่สุดไม่เกินเศษส่วนของหน้าสุดท้ายหน้าเดียวเท่านั้น  เพื่อจัดการกระบวนการนี้ OS จะต้องสร้างและดูแล \*\*ตารางหน้า (Page table)\*\* สำหรับแต่ละโปรเซส  ซึ่งตารางนี้จะระบุตำแหน่ง Frame ในหน่วยความจำกายภาพที่แต่ละ Page ของโปรเซสอาศัยอยู่ ทำให้โปรเซสเซอร์สามารถแปลงที่อยู่ได้อย่างถูกต้อง 



ข้อมูลทั้งหมดเกี่ยวกับสถานะของโปรเซสแต่ละตัวจะถูกเก็บไว้ในโครงสร้างที่เรียกว่า \*\*Process Control Block\*\* ซึ่งจะประกอบด้วย Identifier (ตัวระบุเอกลักษณ์), State (สถานะปัจจุบัน), Priority (ระดับความสำคัญ), Program counter, Memory pointers, และข้อมูลที่เกี่ยวกับ Context และ I/O 



\### หน่วยความจำเสมือน (Virtual Memory)



แนวคิดเรื่องหน่วยความจำเสมือนเกิดขึ้นจากการพัฒนาต่อยอดระบบ Paging  โดยมีการเพิ่มกลไกสำคัญคือ \*\*Demand Paging (การแบ่งหน้าตามความต้องการ)\*\* ซึ่งหมายความว่าแต่ละหน้า (Page) ของโปรเซสจะถูกดึงเข้ามาในหน่วยความจำหลักเฉพาะเมื่อมีการเรียกใช้งานจริงเท่านั้น 



\* \*\*หลักการของพื้นที่ใกล้เคียง (Principle of Locality):\*\* ในช่วงเวลาสั้นๆ การทำงานของโปรแกรมมักจะกระจุกตัวอยู่ในส่วนเล็กๆ เท่านั้น ดังนั้นจึงไม่จำเป็นต้องโหลดหน้าทั้งหมดไว้ในหน่วยความจำ

\* \*\*ข้อผิดพลาดหน้า (Page Fault):\*\* หากโปรแกรมเรียกใช้หน้าที่ไม่ได้อยู่ในหน่วยความจำหลัก ฮาร์ดแวร์จะส่งสัญญาณ Page fault เพื่อบอกให้ OS ไปดึงหน้านั้นมาจากดิสก์ 

\* \*\*การแทนที่หน้า (Page Replacement):\*\* เมื่อต้องนำหน้าใหม่เข้าแต่หน่วยความจำเต็ม OS ต้องเลือกหน้าเดิมออกไป  หากเลือกไม่ดีจะเกิดสภาวะ \*\*Thrashing\*\* คือโปรเซสเซอร์มักจะเสียเวลาไปกับการสลับหน้าไปมาแทนที่จะรันคำสั่งโปรแกรม 





\*\*ผลลัพธ์ที่สำคัญ:\*\*

1\.  โปรเซสสามารถมีขนาดใหญ่กว่าหน่วยความจำหลักที่มีอยู่จริงได้ 

2\.  โปรแกรมเมอร์ไม่ต้องกังวลเรื่องข้อจำกัดของหน่วยความจำอีกต่อไป 

3\.  หน่วยความจำจริงที่ถูกใช้งานจะเรียกว่า \*\*Real memory\*\* ส่วนพื้นที่จัดสรรบนดิสก์ที่ผู้ใช้รับรู้จะเรียกว่า \*\*Virtual memory\*\* 



\### โครงสร้างตารางหน้า (Page Table Structure)



เนื่องจากโปรเซสในปัจจุบันอาจมีขนาดใหญ่มาก (เช่น VAX มีขนาดได้ถึง 2 Gbytes) ทำให้ตารางหน้าที่ต้องใช้มีขนาดใหญ่เกินกว่าจะเก็บไว้ในรีจิสเตอร์ได้ จึงต้องมีวิธีจัดการดังนี้:

\* \*\*ระบบสองระดับ (Two-level scheme):\*\* มี Page directory ที่แต่ละรายการจะชี้ไปยัง Page table อีกทีหนึ่ง 

\* \*\*ตารางหน้าแบบย้อนกลับ (Inverted Page Table):\*\* เป็นโครงสร้างที่เก็บรายการตามจำนวนเฟรม (Frame) ของหน่วยความจำจริง แทนที่จะเก็บตามจำนวนหน้าของหน่วยความจำเสมือน ทำให้ประหยัดพื้นที่คงที่แม้จะมีหลายโปรเซส 



\### Translation Lookaside Buffer (TLB)



การเข้าถึงหน่วยความจำเสมือนแบบปกติจะต้องเข้าถึงหน่วยความจำหลักถึง 2 ครั้ง (ครั้งแรกเพื่อดูตารางหน้า และครั้งที่สองเพื่อดึงข้อมูลจริง) ซึ่งจะทำให้ระบบช้าลงเท่าตัว  เพื่อแก้ปัญหานี้จึงมีการใช้ \*\*TLB\*\*

\* TLB คือแคชพิเศษที่เก็บรายการตารางหน้าที่เพิ่งถูกใช้งานล่าสุด 

\* โดยหลักการ Locality การอ้างอิงส่วนใหญ่มักจะพบใน TLB ทำให้เพิ่มประสิทธิภาพได้อย่างมาก 





\### การแบ่งเซกเมนต์ (Segmentation)



เป็นอีกวิธีในการแบ่งหน่วยความจำ แต่ต่างจาก Paging ตรงที่ \*\*Segmentation จะมองเห็นได้โดยโปรแกรมเมอร์\*\* และทำหน้าที่อำนวยความสะดวกในการจัดระเบียบโปรแกรมและข้อมูล 



\*\*ข้อดีของ Segmentation:\*\* 

\* \*\*จัดการโครงสร้างข้อมูลที่เติบโตได้:\*\* สามารถขยายหรือหดขนาดเซกเมนต์ได้ตามต้องการ 

\* \*\*การคอมไพล์ที่เป็นอิสระ:\*\* แก้ไขและคอมไพล์โปรแกรมแยกกันได้โดยไม่ต้อง Link ใหม่ทั้งหมด 

\* \*\*การแชร์ทรัพยากร:\*\* สามารถกำหนดให้โปรเซสอื่นเข้าถึงบางเซกเมนต์ร่วมกันได้ 

\* \*\*การป้องกัน:\*\* สามารถกำหนดสิทธิการเข้าถึง (เช่น Read-only) ให้กับแต่ละเซกเมนต์ได้อย่างสะดวก 



ในปัจจุบัน บางระบบได้นำข้อดีของทั้งสองวิธีมารวมกันเป็นระบบ \*\*Combined Segmentation and Paging\*\* เพื่อประสิทธิภาพสูงสุด 



\## 8.4 การจัดการหน่วยความจำของ Intel x86 (Intel x86 Memory Management) 



นับตั้งแต่มีการเปิดตัวสถาปัตยกรรม 32 บิต ไมโครโปรเซสเซอร์ได้พัฒนากลไกการจัดการหน่วยความจำที่ซับซ้อนซึ่งต่อยอดมาจากระบบขนาดกลางและขนาดใหญ่ ในหลายๆ กรณี เวอร์ชันของไมโครโปรเซสเซอร์นั้นทำงานได้เหนือกว่าระบบรุ่นก่อนๆ  เนื่องจากกลไกเหล่านี้ถูกพัฒนาขึ้นโดยผู้ผลิตฮาร์ดแวร์และสามารถใช้งานร่วมกับระบบปฏิบัติการได้หลากหลาย จึงมักถูกออกแบบมาให้ใช้งานได้ครอบคลุม (General-purpose) ตัวอย่างที่ชัดเจนคือกลไกที่ใช้ในสถาปัตยกรรม Intel x86 



\### พื้นที่ที่อยู่ (Address Spaces) 

x86 มีฮาร์ดแวร์รองรับทั้งการทำ Segmentation และ Paging  กลไกทั้งสองนี้สามารถปิดการใช้งานได้ ทำให้ผู้ใช้สามารถเลือกมุมมองของหน่วยความจำได้ 4 รูปแบบที่แตกต่างกัน: 



\* \*\*Unsegmented unpaged memory:\*\* ที่อยู่เสมือน (Virtual address) จะเหมือนกับที่อยู่กายภาพ (Physical address) รูปแบบนี้มีประโยชน์สำหรับแอปพลิเคชันตัวควบคุมที่มีความซับซ้อนต่ำแต่ต้องการประสิทธิภาพสูง 

\* \*\*Unsegmented paged memory:\*\* มองหน่วยความจำแบบ Linear address space ที่มีการแบ่งหน้า (Paged)  การป้องกันและจัดการหน่วยความจำจะทำผ่าน Paging ซึ่งเป็นที่นิยมในบางระบบปฏิบัติการ (เช่น Berkeley UNIX) 

\* \*\*Segmented unpaged memory:\*\* มองหน่วยความจำแบบกลุ่มของ Logical address space  ข้อดีคือสามารถป้องกันข้อมูลได้ละเอียดถึงระดับไบต์หากจำเป็น และรับประกันได้ว่าเวลาในการเข้าถึงข้อมูลจะคงที่และคาดเดาได้ 

\* \*\*Segmented paged memory:\*\* ใช้ Segmentation แบ่งพาร์ติชันตรรกะเพื่อควบคุมการเข้าถึง และใช้ Paging จัดการการจัดสรรหน่วยความจำภายในพาร์ติชันนั้น  ระบบปฏิบัติการอย่าง UNIX System V นิยมใช้รูปแบบนี้ 



\### การแบ่งเซกเมนต์ (Segmentation) 

เมื่อใช้งาน Segmentation ที่อยู่เสมือนแต่ละอันจะประกอบด้วย Segment reference ขนาด 16 บิต และ Offset ขนาด 32 บิต บิตควบคุมการป้องกันจะใช้ 2 บิต ทำให้เหลือ 14 บิตสำหรับระบุเซกเมนต์  ทำให้พื้นที่หน่วยความจำเสมือนรวมที่ผู้ใช้มองเห็นมีขนาดสูงสุดถึง $2^{46} = 64$ เทราไบต์ (Tbytes) 



การป้องกันของแต่ละเซกเมนต์มี 2 รูปแบบคือ ระดับสิทธิพิเศษ (Privilege level) และ คุณลักษณะการเข้าถึง (Access attribute)  มีระดับสิทธิพิเศษ 4 ระดับ ตั้งแต่ระดับ 0 (ป้องกันสูงสุด) ถึงระดับ 3 (ป้องกันต่ำสุด)  โปรแกรมที่กำลังทำงานจะเข้าถึงเซกเมนต์ข้อมูลได้ก็ต่อเมื่อมีระดับสิทธิพิเศษต่ำกว่าหรือเท่ากับระดับของเซกเมนต์ข้อมูลนั้น กลไกฮาร์ดแวร์ไม่ได้บังคับว่าต้องใช้ระดับเหล่านี้อย่างไร ขึ้นอยู่กับการออกแบบ OS 



\### การแบ่งหน้า (Paging) 

กลไก Paging ของ x86 เป็นการค้นหาตารางแบบสองระดับ (Two-level table lookup) 

1\. \*\*ระดับแรกคือ Page directory\*\* ซึ่งมีได้สูงสุด 1024 รายการ (Entries)  ทำหน้าที่แบ่งพื้นที่หน่วยความจำเชิงเส้นขนาด 4-Gbyte ออกเป็นกลุ่มหน้า 1024 กลุ่ม โดยแต่ละกลุ่มจะมีขนาด 4 Mbytes 

2\. \*\*ระดับที่สองคือ Page table\*\* โดยแต่ละตารางจะมีได้สูงสุด 1024 รายการ ซึ่งแต่ละรายการจะตรงกับหน้าขนาด 4 Kbyte จำนวน 1 หน้า 



x86 ยังมีการใช้ Translation Lookaside Buffer (TLB) ซึ่งสามารถเก็บรายการตารางหน้าได้ 32 รายการ  ในส่วนขยายใหม่ยังอนุญาตให้ OS กำหนดขนาดของหน้าได้ 2 ขนาด คือ 4 Kbyte หรือ 4 Mbyte  การใช้หน้าขนาด 4 Mbyte ช่วยลดความต้องการพื้นที่จัดเก็บตารางจัดการหน่วยความจำสำหรับหน่วยความจำหลักขนาดใหญ่ได้อย่างมาก

\## 8.5 การจัดการหน่วยความจำของ ARM (ARM Memory Management) 



ARM มีสถาปัตยกรรมระบบหน่วยความจำเสมือนที่ยืดหยุ่นซึ่งสามารถปรับแต่งให้เข้ากับความต้องการของนักออกแบบระบบฝังตัว (Embedded system) ได้ \[



\### โครงสร้างระบบหน่วยความจำ (Memory System Organization) 

ฮาร์ดแวร์แปลงที่อยู่หน่วยความจำเสมือนใช้ตาราง 1 หรือ 2 ระดับในการแปลงที่อยู่เสมือนเป็นที่อยู่กายภาพ  Translation Lookaside Buffer (TLB) ทำหน้าที่เป็นแคชเก็บรายการตารางหน้าที่เพิ่งใช้ล่าสุด  รายการในตารางการแปลงจะมีบิตควบคุมการเข้าถึง (Access control bits) รวมอยู่ด้วย ซึ่งจะกำหนดว่าโปรเซสใดสามารถเข้าถึงส่วนใดของหน่วยความจำได้ หากไม่ได้รับอนุญาต ฮาร์ดแวร์จะส่งสัญญาณ Abort ไปยังโปรเซสเซอร์ ARM 



\### การแปลงที่อยู่หน่วยความจำเสมือน (Virtual Memory Address Translation) 

ARM รองรับการเข้าถึงหน่วยความจำตาม Sections หรือ Pages ดังนี้: 

\* \*\*Supersections (ทางเลือก):\*\* บล็อกหน่วยความจำหลักขนาด 16-MB 

\* \*\*Sections:\*\* บล็อกขนาด 1-MB 

\* \*\*Large pages:\*\* บล็อกขนาด 64-kB 

\* \*\*Small pages:\*\* บล็อกขนาด 4-kB 



ตารางการแปลง (Translation table) ในหน่วยความจำหลักมี 2 ระดับ: 

\* \*\*ตารางระดับ 1 (Level 1 table):\*\* เก็บตัวระบุที่ประกอบด้วยที่อยู่ฐานและคุณสมบัติการแปลงสำหรับ Section และ Supersection และมีตัวชี้ (Pointers) ไปยังตารางระดับ 2 สำหรับหน้าขนาดใหญ่และขนาดเล็ก 

\* \*\*ตารางระดับ 2 (Level 2 table):\*\* เก็บตัวระบุสำหรับหน้าขนาดเล็ก หรือหน้าขนาดใหญ่ โดยตารางระดับ 2 นี้ต้องการหน่วยความจำเพียง 1 kB 







\### การควบคุมการเข้าถึง (Access Control) 

บิตควบคุมการเข้าถึง AP ในตารางแต่ละรายการจะควบคุมการเข้าถึงพื้นที่หน่วยความจำของโปรเซส  สามารถกำหนดพื้นที่หน่วยความจำให้เป็นแบบ No access (ห้ามเข้าถึง), Read only (อ่านอย่างเดียว) หรือ Read-write (อ่านและเขียน) ได้ 



นอกจากนี้ ARM ยังใช้แนวคิดของ \*\*โดเมน (Domain)\*\* ซึ่งคือกลุ่มของ Sections และ/หรือ Pages ที่มีสิทธิการเข้าถึงเฉพาะ สถาปัตยกรรม ARM รองรับได้ 16 โดเมน การเข้าถึงโดเมนแบ่งเป็น 2 ประเภทคือ: 

\* \*\*Clients (ไคลเอนต์):\*\* ผู้ใช้โดเมนที่ต้องปฏิบัติตามสิทธิการเข้าถึงของ Sections และ/หรือ Pages แต่ละอันในโดเมนนั้นอย่างเคร่งครัด 

\* \*\*Managers (ผู้จัดการ):\*\* ควบคุมพฤติกรรมของโดเมนและสามารถข้ามขั้นตอนการตรวจสอบสิทธิการเข้าถึงสำหรับรายการในตารางของโดเมนนั้นได้
