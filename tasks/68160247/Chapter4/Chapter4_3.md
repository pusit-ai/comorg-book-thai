# 4.3 องค์ประกอบของการออกแบบแคช (Elements of Cache Design)

## บทนำ

การออกแบบแคช (Cache Design) เป็นหัวใจสำคัญของการทำให้ระบบคอมพิวเตอร์สามารถลดช่องว่างด้านความเร็วระหว่างตัวประมวลผลกลาง (Central Processing Unit: CPU) กับหน่วยความจำหลัก (Main Memory) ได้อย่างมีประสิทธิภาพ ในทางปฏิบัติ การมีแคชเพียงอย่างเดียวไม่ได้รับประกันว่าระบบจะทำงานเร็วขึ้นเสมอไป หากสถาปัตยกรรมแคชถูกออกแบบไม่เหมาะสม ก็อาจเกิดปัญหาอัตราการพลาดแคช (Miss Rate) สูง ความหน่วงแฝง (Latency) เพิ่มขึ้น และเกิดการแย่งใช้งานทรัพยากรภายในระบบมากขึ้นแทน

**สาระสำคัญของหัวข้อนี้อยู่ที่การทำความเข้าใจว่า “แคชควรถูกออกแบบอย่างไร” จึงจะสามารถเก็บข้อมูลหรือคำสั่งที่โปรเซสเซอร์ต้องใช้ได้ตรงเวลา รวดเร็ว และคุ้มค่าต่อข้อจำกัดด้านฮาร์ดแวร์**

องค์ประกอบหลักของการออกแบบแคช (Elements of Cache Design) ที่ต้องพิจารณา ได้แก่

- ขนาดของแคช (Cache Size)
- ฟังก์ชันการแมป (Mapping Function)
- ขั้นตอนวิธีการแทนที่ข้อมูล (Replacement Algorithm)
- นโยบายการเขียนข้อมูล (Write Policy)
- ขนาดของบล็อกหรือขนาดบรรทัดแคช (Line Size / Block Size)
- จำนวนระดับของแคช (Number of Caches / Multilevel Caches)
- รูปแบบแคชรวมกับแคชแยก (Unified Cache versus Split Cache)

ในเชิงสถาปัตยกรรม ประเด็นทั้งหมดนี้เชื่อมโยงกันโดยตรง กล่าวคือ การเพิ่มขนาดแคชย่อมส่งผลต่อเวลาเข้าถึง (Access Time) การเลือกวิธีแมปย่อมกระทบต่อความซับซ้อนของฮาร์ดแวร์ การเลือกนโยบายการเขียนย่อมกระทบต่อทราฟฟิกบนบัส (Bus Traffic) และการกำหนดขนาดบล็อกก็ย่อมกระทบต่อการใช้ประโยชน์จากหลักการเฉพาะที่ (Locality of Reference) ดังนั้น **การออกแบบแคชจึงไม่ใช่การเลือกค่าตัวแปรแยกส่วน แต่เป็นการหาจุดสมดุลระหว่างความเร็ว ความจุ ความซับซ้อน และต้นทุน**

## 1. ขนาดของแคช (Cache Size)

ขนาดของแคชเป็นปัจจัยแรกที่มักถูกพิจารณา เพราะมีผลโดยตรงต่อความสามารถในการเก็บข้อมูลที่ใช้งานบ่อย หากแคชมีขนาดใหญ่ขึ้น ก็มีแนวโน้มที่จะเก็บบล็อกจากหน่วยความจำหลักได้มากขึ้น ส่งผลให้โอกาสเกิดแคชฮิต (Cache Hit) สูงขึ้น และลดความจำเป็นในการอ้างอิงหน่วยความจำหลักซึ่งช้ากว่าอย่างมาก

อย่างไรก็ตาม การเพิ่มขนาดแคชไม่ได้ให้ประโยชน์แบบเป็นเส้นตรงเสมอไป เนื่องจากเมื่อแคชมีขนาดใหญ่ขึ้น วงจรค้นหาแท็ก (Tag Lookup Circuitry) และโครงสร้างควบคุมภายในก็จะซับซ้อนขึ้น ทำให้เวลาในการเข้าถึงแคชเองอาจเพิ่มขึ้นตามไปด้วย

### 1.1 หลักคิดพื้นฐาน

- แคชขนาดเล็กมีจุดเด่นที่เข้าถึงได้เร็วมาก
- แคชขนาดใหญ่มีจุดเด่นที่เก็บข้อมูลได้มากขึ้น
- หากแคชใหญ่เกินไป เวลาตรวจสอบแท็กและการจัดการภายในอาจช้าลง
- หากแคชเล็กเกินไป จะเกิดการแทนที่ข้อมูลบ่อย และอัตราการพลาดแคชสูง

**ดังนั้น การกำหนดขนาดแคชที่เหมาะสมจึงเป็นการประนีประนอมระหว่าง “ความจุ” กับ “ความเร็วในการเข้าถึง”**

### 1.2 ความสัมพันธ์กับอัตราฮิต (Hit Ratio)

เมื่อขนาดแคชเพิ่มขึ้น จำนวนบล็อกที่เก็บได้ก็เพิ่มขึ้น ส่งผลให้ข้อมูลที่มีแนวโน้มถูกใช้งานซ้ำตามหลักการเฉพาะที่เชิงเวลา (Temporal Locality) และหลักการเฉพาะที่เชิงพื้นที่ (Spatial Locality) ถูกเก็บไว้ได้มากขึ้น โอกาสเกิดแคชฮิตจึงเพิ่มขึ้น

แต่ในทางกลับกัน หากเพิ่มขนาดแคชจนทำให้วงจรค้นหาซับซ้อนมากเกินไป เวลาที่ใช้ในการตรวจสอบว่า “ฮิตหรือมิส” ก็อาจเพิ่มขึ้น จนผลกำไรด้านสมรรถนะลดลง

### 1.3 ภาพรวมข้อดีข้อจำกัด

| ประเด็น | แคชขนาดเล็ก | แคชขนาดใหญ่ |
|---|---|---|
| เวลาเข้าถึงแคช | มักเร็วกว่า | มักช้ากว่าเล็กน้อย |
| อัตราการฮิต | ต่ำกว่า | สูงกว่า |
| ต้นทุนฮาร์ดแวร์ | ต่ำกว่า | สูงกว่า |
| การกินพื้นที่บนชิป | น้อยกว่า | มากกว่า |
| ผลต่อสมรรถนะรวม | ดีในงานง่าย | ดีกว่าในงานซับซ้อน หากออกแบบเหมาะสม |

### 1.4 ขยายความเชิงวิเคราะห์

ในระบบสมัยใหม่ มักไม่แก้ปัญหาด้วยการสร้างแคชก้อนเดียวให้ใหญ่มาก แต่จะแบ่งเป็นหลายระดับ เช่น แคชระดับที่ 1 (Level 1 Cache: L1), แคชระดับที่ 2 (Level 2 Cache: L2) และแคชระดับที่ 3 (Level 3 Cache: L3) เพื่อให้ได้ทั้งความเร็วของแคชเล็กและความจุของแคชใหญ่พร้อมกัน

## 2. ฟังก์ชันการแมป (Mapping Function)

ฟังก์ชันการแมปคือกลไกที่กำหนดว่า บล็อกหนึ่งจากหน่วยความจำหลักจะถูกนำไปวางไว้ที่ตำแหน่งใดในแคช ประเด็นนี้มีความสำคัญมาก เพราะเป็นตัวกำหนดทั้ง

- ความเร็วในการค้นหาข้อมูล
- ระดับความยืดหยุ่นในการเก็บข้อมูล
- ความน่าจะเป็นของการชนกันของบล็อก (Conflict)
- ความซับซ้อนของวงจรฮาร์ดแวร์

**กล่าวอย่างง่าย ฟังก์ชันการแมปคือ “กฎการจับคู่” ระหว่างบล็อกในหน่วยความจำหลักกับบรรทัดในแคช**

โดยทั่วไป วิธีแมปหลักมี 3 แบบ ได้แก่

- การแมปโดยตรง (Direct Mapping)
- การแมปแบบแอสโซซิเอทีฟสมบูรณ์ (Fully Associative Mapping)
- การแมปแบบเซตแอสโซซิเอทีฟ (Set-Associative Mapping)

### 2.1 โครงสร้างที่อยู่หน่วยความจำ (Memory Address Structure)

เมื่อซีพียูต้องการข้อมูลจากตำแหน่งหนึ่งในหน่วยความจำ ที่อยู่หน่วยความจำ (Memory Address) จะถูกแยกออกเป็นส่วนต่าง ๆ เพื่อใช้กับแคช เช่น

- ส่วนแท็ก (Tag)
- ส่วนดัชนีบรรทัดหรือดัชนีเซต (Line Index / Set Index)
- ส่วนออฟเซตภายในบล็อก (Word Offset / Byte Offset)

โดยแนวคิดทั่วไปคือ

- `Offset` ใช้ระบุว่าต้องการคำหรือไบต์ใดภายในบล็อก
- `Index` หรือ `Set` ใช้ระบุตำแหน่งแคชที่จะต้องตรวจสอบ
- `Tag` ใช้ตรวจว่าบล็อกที่อยู่ในแคชตรงกับบล็อกที่ซีพียูร้องขอจริงหรือไม่

### 2.2 ความสัมพันธ์เชิงปริมาณของบล็อกและบรรทัด

หากกำหนดให้

- หน่วยความจำหลักมีหน่วยที่ระบุแอดเดรสได้ทั้งหมด `2^n` หน่วย
- ขนาดบล็อกเท่ากับ `K = 2^w` ไบต์หรือคำ
- แคชมีจำนวนบรรทัดทั้งหมด `m`

จะได้ว่า

- จำนวนบล็อกในหน่วยความจำหลัก = `2^n / K`
- จำนวนบรรทัดในแคช = `m`
- การออกแบบที่อยู่จึงต้องตอบให้ได้ว่า บล็อกจากหน่วยความจำหลักหนึ่งบล็อกจะถูกอ้างถึงอย่างไร และจะถูกส่งไปอยู่ที่ใดในแคช

**จุดนี้เองที่ทำให้ Mapping Function กลายเป็นตัวกำหนดรูปแบบสถาปัตยกรรมของแคชทั้งระบบ**

## 3. การแมปโดยตรง (Direct Mapping)

การแมปโดยตรงเป็นวิธีที่ง่ายที่สุด แต่มีข้อจำกัดเรื่องความยืดหยุ่น โดยหลักการคือ **บล็อกแต่ละบล็อกจากหน่วยความจำหลักจะถูกแมปไปยังบรรทัดแคชได้เพียงบรรทัดเดียวเท่านั้น**

ถ้าให้

- `m` = จำนวนบรรทัดแคช (Cache Lines)
- `j` = หมายเลขบล็อกในหน่วยความจำหลัก

จะได้ความสัมพันธ์ว่า

`i = j mod m`

โดยที่ `i` คือหมายเลขบรรทัดแคชที่บล็อก `j` ต้องไปอยู่

### 3.1 โครงสร้างที่อยู่สำหรับ Direct Mapping

ที่อยู่หน่วยความจำจะถูกแบ่งเป็น 3 ส่วน

- `Tag`
- `Line`
- `Word/Byte Offset`

หาก

- ขนาดบล็อก = `2^w`
- จำนวนบรรทัดแคช = `2^r`

จะได้ว่า

- บิตล่างสุด `w` บิต ใช้เป็น `Offset`
- ถัดขึ้นมา `r` บิต ใช้เป็น `Line Index`
- บิตที่เหลือ ใช้เป็น `Tag`

### 3.2 หลักการทำงาน

เมื่อซีพียูส่งที่อยู่หน่วยความจำเข้ามา

1. ระบบใช้บิต `Line Index` ระบุว่าจะต้องดูที่บรรทัดใดในแคช
2. อ่านค่า `Tag` ที่เก็บอยู่ในบรรทัดนั้น
3. เปรียบเทียบกับ `Tag` จากที่อยู่หน่วยความจำที่ร้องขอ
4. ถ้าตรงกัน ถือว่าเป็นแคชฮิต (Cache Hit)
5. ถ้าไม่ตรงกัน ถือว่าเป็นแคชมิส (Cache Miss) และต้องดึงบล็อกจากหน่วยความจำหลักมาแทนที่บรรทัดนั้น

### 3.2.1 ตัวอย่างเชิงตัวเลขของ Direct Mapping

สมมติระบบมีคุณลักษณะดังนี้

- หน่วยความจำหลักขนาด `4 GB`
- ที่อยู่หน่วยความจำยาว `32 บิต`
- แคชขนาด `32 KB`
- ขนาดบล็อก `64 ไบต์`

สามารถวิเคราะห์ได้ดังนี้

- ขนาดบล็อก `64 ไบต์ = 2^6` ดังนั้น `Offset = 6 บิต`
- จำนวนบรรทัดในแคช = `32 KB / 64 B = 512` บรรทัด
- `512 = 2^9` ดังนั้น `Line Index = 9 บิต`
- จำนวนบิตแท็ก = `32 - 9 - 6 = 17 บิต`

ดังนั้นโครงสร้างที่อยู่จะเป็น

- `Tag = 17 บิต`
- `Index = 9 บิต`
- `Offset = 6 บิต`

ความหมายเชิงสถาปัตยกรรมคือ เมื่อซีพียูส่งที่อยู่ 32 บิตเข้ามา ระบบจะใช้ 9 บิตกลางเพื่อเลือกบรรทัดแคชหนึ่งบรรทัด จากนั้นใช้ 17 บิตบนสุดเพื่อตรวจสอบว่าเป็นบล็อกเดียวกันจริงหรือไม่ และใช้ 6 บิตล่างสุดเพื่อระบุไบต์ภายในบล็อกนั้น

### 3.3 ข้อดี

- โครงสร้างฮาร์ดแวร์ง่าย
- ใช้ต้นทุนต่ำ
- ตรวจสอบได้รวดเร็ว เพราะรู้แน่ชัดว่าต้องตรวจเพียงหนึ่งบรรทัด

### 3.4 ข้อเสีย

- ขาดความยืดหยุ่น
- บล็อกหลายบล็อกจากหน่วยความจำหลักอาจถูกบังคับให้แข่งขันกันใช้บรรทัดเดียวกัน
- หากโปรแกรมอ้างอิงบล็อกที่แมปชนกันซ้ำ ๆ จะเกิดแคชมิสบ่อยมาก

### 3.5 ขยายความเชิงวิชาการ

ปัญหาสำคัญของ Direct Mapping คือ **Conflict Miss** หรือการพลาดแคชที่เกิดจากการชนกันของตำแหน่งแมป แม้ว่าแคชจะยังมีพื้นที่ว่างอยู่ในบรรทัดอื่น แต่บล็อกที่ถูกออกแบบมาให้ลงได้เพียงบรรทัดเดียวก็ไม่สามารถย้ายไปเก็บที่อื่นได้

ตัวอย่างเชิงแนวคิด:

- สมมติบล็อก A และบล็อก B แมปมายังบรรทัดที่ 5 เหมือนกัน
- หากโปรแกรมสลับใช้งาน A กับ B ตลอดเวลา
- ทุกครั้งที่โหลด A จะไล่ B ออก และทุกครั้งที่โหลด B จะไล่ A ออก
- สถานการณ์นี้เรียกว่าอาการแคชกระแทกกัน (Thrashing เชิงแคช)

**ดังนั้น แม้ Direct Mapping จะเร็วและประหยัด แต่ในเวิร์กโหลดที่มีการชนของบล็อกสูง ประสิทธิภาพอาจลดลงอย่างมาก**

## 4. การแมปแบบแอสโซซิเอทีฟสมบูรณ์ (Fully Associative Mapping)

การแมปแบบนี้มีความยืดหยุ่นสูงที่สุด โดยอนุญาตให้ **บล็อกใด ๆ จากหน่วยความจำหลักสามารถถูกวางลงในบรรทัดใดก็ได้ของแคช**

### 4.1 โครงสร้างที่อยู่

ที่อยู่หน่วยความจำในกรณีนี้แบ่งเพียง 2 ส่วน

- `Tag`
- `Offset`

ไม่มี `Line Index` โดยตรง เพราะบล็อกสามารถไปอยู่ตำแหน่งใดก็ได้ในแคช

### 4.2 หลักการทำงาน

เมื่อซีพียูร้องขอข้อมูล

1. ระบบนำ `Tag` จากที่อยู่หน่วยความจำมาเปรียบเทียบกับ `Tag` ของทุกบรรทัดในแคชพร้อมกัน
2. ถ้ามีบรรทัดใดบรรทัดหนึ่งตรงกัน ถือว่าเป็นแคชฮิต
3. ถ้าไม่มีบรรทัดใดตรงกัน ถือว่าเป็นแคชมิส

### 4.2.1 ตัวอย่างเชิงตัวเลขของ Fully Associative Mapping

หากใช้สมมติฐานเดียวกับตัวอย่างก่อนหน้า

- หน่วยความจำหลัก `4 GB`
- ที่อยู่ `32 บิต`
- แคช `32 KB`
- บล็อก `64 ไบต์`

จะได้ว่า

- `Offset = 6 บิต`
- ไม่มี `Index`
- ดังนั้น `Tag = 32 - 6 = 26 บิต`

นั่นหมายความว่า ระบบต้องนำแท็ก 26 บิตไปเปรียบเทียบกับทุกบรรทัดในแคชพร้อมกัน การออกแบบแบบนี้ลดการชนของบล็อกได้ดีมาก แต่ต้องอาศัยฮาร์ดแวร์เปรียบเทียบจำนวนมาก

### 4.3 จุดเด่น

- ยืดหยุ่นมากที่สุด
- ลดปัญหา Conflict Miss ได้ดีที่สุด
- ใช้พื้นที่แคชได้คุ้มค่ากว่าในเชิงตรรกะ

### 4.4 จุดด้อย

- ฮาร์ดแวร์ซับซ้อนมาก
- ต้องมีวงจรเปรียบเทียบแท็กจำนวนมากทำงานแบบขนาน (Parallel Comparison)
- เมื่อแคชมีขนาดใหญ่ ค่าใช้จ่ายและพลังงานที่ใช้จะสูงขึ้น

### 4.5 ขยายความ

แนวคิดของ Fully Associative Mapping สะท้อนหลักการที่ว่า “ถ้าไม่บังคับตำแหน่ง ก็ลดโอกาสชนกันได้” แต่แลกมาด้วยวงจรควบคุมราคาแพงและซับซ้อน ดังนั้นในระบบจริงมักไม่นิยมใช้กับแคชขนาดใหญ่ทั้งก้อน แต่อาจพบในโครงสร้างเฉพาะที่ต้องการความยืดหยุ่นสูง เช่น บัฟเฟอร์บางประเภท หรือแคชขนาดเล็กพิเศษ

## 5. การแมปแบบเซตแอสโซซิเอทีฟ (Set-Associative Mapping)

การแมปแบบเซตแอสโซซิเอทีฟเป็นวิธีที่ใช้กันแพร่หลายมากในสถาปัตยกรรมจริง เพราะเป็นทางสายกลางระหว่าง Direct Mapping กับ Fully Associative Mapping

หลักการคือ

- แคชถูกแบ่งออกเป็นหลายเซต (Sets)
- แต่ละเซตประกอบด้วยหลายบรรทัด (Lines)
- บล็อกจากหน่วยความจำหลักจะถูกแมปมายัง “เซต” ที่กำหนด
- ภายในเซตนั้น บล็อกสามารถลงบรรทัดใดก็ได้

หากแคชเป็นแบบ `k-way set associative`

- `k` หมายถึงจำนวนบรรทัดต่อหนึ่งเซต
- ถ้าเป็น `2-way` หมายถึงหนึ่งเซตมี 2 บรรทัด
- ถ้าเป็น `4-way` หมายถึงหนึ่งเซตมี 4 บรรทัด

### 5.1 หลักการทำงาน

1. ใช้บิตบางส่วนของที่อยู่หน่วยความจำระบุ `Set`
2. เข้าไปยังเซตนั้น
3. เปรียบเทียบ `Tag` กับทุกบรรทัดภายในเซต
4. ถ้ามีบรรทัดใดตรงกัน ถือว่าเป็นแคชฮิต
5. ถ้าไม่ตรงเลย ถือว่าเป็นแคชมิส และต้องเลือกบรรทัดหนึ่งภายในเซตมาแทนที่

### 5.2 โครงสร้างที่อยู่

ที่อยู่หน่วยความจำจะแบ่งเป็น

- `Tag`
- `Set`
- `Offset`

### 5.2.1 ตัวอย่างเชิงตัวเลขของ 4-Way Set-Associative Mapping

หากยังใช้สมมติฐานเดียวกันคือ

- หน่วยความจำหลัก `4 GB`
- แคช `32 KB`
- ขนาดบล็อก `64 ไบต์`
- จำนวนบรรทัดรวมในแคช `512` บรรทัด

และสมมติว่าแคชเป็นแบบ `4-way set associative`

จะได้ว่า

- จำนวนเซต = `512 / 4 = 128` เซต
- `128 = 2^7` ดังนั้น `Set Index = 7 บิต`
- `Offset = 6 บิต`
- `Tag = 32 - 7 - 6 = 19 บิต`

ดังนั้นโครงสร้างที่อยู่จะเป็น

- `Tag = 19 บิต`
- `Set = 7 บิต`
- `Offset = 6 บิต`

ความหมายคือ บล็อกหนึ่งจะถูกบังคับให้ไปอยู่ในเซตหนึ่งที่กำหนด แต่ภายในเซตนั้นยังมีให้เลือกได้ 4 บรรทัด จึงยืดหยุ่นกว่า Direct Mapping อย่างมีนัยสำคัญ

### 5.3 ความหมายเชิงเปรียบเทียบ

- Direct Mapping: เลือกได้ “บ้านหลังเดียว”
- Fully Associative: เลือกได้ “ทุกบ้าน”
- Set-Associative: เลือกได้ “เฉพาะบ้านในหมู่บ้านหนึ่ง”

การออกแบบแบบนี้ช่วยลดการชนกันของบล็อกได้มากกว่า Direct Mapping แต่ยังคงควบคุมความซับซ้อนของฮาร์ดแวร์ไม่ให้สูงเท่า Fully Associative

### 5.4 ข้อดี

- สมดุลดีระหว่างต้นทุนกับประสิทธิภาพ
- ลด Conflict Miss ได้มากกว่า Direct Mapping
- ไม่ซับซ้อนเกินไปเหมือน Fully Associative

### 5.5 ข้อเสีย

- ซับซ้อนกว่า Direct Mapping
- ยิ่งเพิ่มจำนวนทาง (Ways) มาก วงจรเปรียบเทียบแท็กยิ่งซับซ้อนขึ้น
- ต้องมีนโยบายเลือกว่าจะไล่บรรทัดใดออกเมื่อเซตเต็ม

### 5.6 ขยายความสำคัญทางสถาปัตยกรรม

**Set-Associative Mapping ได้รับความนิยมมากที่สุด เพราะให้ผลตอบแทนด้านสมรรถนะต่อความซับซ้อนของฮาร์ดแวร์ค่อนข้างคุ้มค่า** โดยเฉพาะแคช L1 และ L2 ในไมโครโปรเซสเซอร์สมัยใหม่ที่มักใช้แบบ 2-way, 4-way, 8-way หรือมากกว่านั้นตามเป้าหมายการออกแบบ

## 6. ตารางเปรียบเทียบวิธีการแมป (Comparison of Mapping Methods)

| คุณลักษณะ | Direct Mapping | Fully Associative Mapping | Set-Associative Mapping |
|---|---|---|---|
| ตำแหน่งที่บล็อกลงได้ | ตำแหน่งเดียว | ทุกตำแหน่ง | ได้ทุกบรรทัดภายในเซต |
| ความเร็วในการค้นหา | เร็วมาก | ช้ากว่าเพราะต้องเทียบทุกบรรทัด | ปานกลางถึงเร็ว |
| ความซับซ้อนฮาร์ดแวร์ | ต่ำ | สูงมาก | ปานกลาง |
| ความยืดหยุ่น | ต่ำ | สูงที่สุด | สูงปานกลาง |
| โอกาสเกิด Conflict Miss | สูง | ต่ำมาก | ต่ำกว่า Direct Mapping |
| ต้นทุนการออกแบบ | ต่ำ | สูง | ปานกลาง |
| การใช้งานในระบบจริง | พบได้มาก | ใช้เฉพาะบางจุด | พบมากที่สุด |

### 6.1 ข้อสังเกตเชิงวิเคราะห์

- หากเป้าหมายคือความเร็วในการเข้าถึงขั้นต้นและความง่ายในการออกแบบ Direct Mapping มีความน่าสนใจ
- หากเป้าหมายคือการลด Conflict Miss ให้ต่ำที่สุด Fully Associative ให้ผลดีที่สุด
- หากต้องการจุดสมดุลที่ใช้งานจริงได้ในไมโครโปรเซสเซอร์ส่วนใหญ่ Set-Associative มักเป็นคำตอบที่เหมาะสมที่สุด

**นี่เป็นเหตุผลที่ระบบจริงจำนวนมากนิยมใช้ Set-Associative Cache แทนที่จะเลือกสุดโต่งไปทางใดทางหนึ่ง**

## 7. ขั้นตอนวิธีการแทนที่ข้อมูล (Replacement Algorithm)

เมื่อเกิดแคชมิสและจำเป็นต้องนำบล็อกใหม่เข้ามาในแคช ระบบต้องตัดสินใจว่า “จะเอาบรรทัดใดออก” ประเด็นนี้สำคัญมากใน Fully Associative และ Set-Associative Mapping เพราะมีได้หลายตำแหน่งให้เลือกแทนที่

**Replacement Algorithm คือกฎการคัดออกของข้อมูลเก่าเพื่อเปิดที่ให้ข้อมูลใหม่**

### 7.1 กรณีของ Direct Mapping

ใน Direct Mapping ไม่แทบไม่มีปัญหาให้ตัดสินใจ เพราะบล็อกหนึ่งมีตำแหน่งลงเพียงตำแหน่งเดียว หากเกิดมิสก็แทนที่บรรทัดนั้นได้ทันที

### 7.2 กรณีของ Set-Associative และ Fully Associative

กรณีนี้ระบบต้องมีขั้นตอนวิธีช่วยเลือกบรรทัดที่จะถูกแทนที่ โดยวิธีที่สำคัญ ได้แก่

- แบบสุ่ม (Random)
- มาก่อนออกก่อน (First-In First-Out: FIFO)
- ใช้น้อยที่สุดเชิงความถี่ (Least Frequently Used: LFU)
- ใช้งานล่าสุดนานที่สุด (Least Recently Used: LRU)
- แอลอาร์ยูเทียม (Pseudo-LRU) สำหรับกรณีที่ต้องลดความซับซ้อน

### 7.3 Least Recently Used (LRU)

หลักคิดของ LRU คือ บรรทัดใดที่ไม่ได้ถูกใช้งานมานานที่สุด มีแนวโน้มต่ำกว่าที่จะถูกใช้ในอนาคตอันใกล้ จึงควรถูกแทนที่ก่อน

#### จุดเด่น

- สอดคล้องกับหลักการเฉพาะที่เชิงเวลา (Temporal Locality)
- ให้ผลดีในหลายเวิร์กโหลด

#### จุดด้อย

- ยิ่งจำนวนทาง (Ways) มาก การติดตามลำดับ “ใช้ล่าสุด” ยิ่งซับซ้อน
- ต้องใช้บิตควบคุมและตรรกะเพิ่มเติม

#### ขยายความ

สำหรับแคชแบบ `2-way set associative` การทำ LRU ค่อนข้างง่าย เพราะใช้เพียงบิตเดียวต่อเซตเพื่อบอกว่าบรรทัดใดถูกใช้งานล่าสุด แต่เมื่อกลายเป็น 4-way, 8-way หรือมากกว่า การเก็บลำดับการใช้งานครบถ้วนจะยากขึ้นมาก จึงมักใช้ Pseudo-LRU แทน

### 7.4 First-In First-Out (FIFO)

หลักคิดของ FIFO คือ บรรทัดใดอยู่ในแคชมานานที่สุด จะถูกแทนที่ก่อน โดยไม่สนใจว่าถูกใช้งานบ่อยหรือเพิ่งใช้งานไปหรือไม่

#### จุดเด่น

- ตรรกะง่าย
- ต้นทุนฮาร์ดแวร์ไม่สูง

#### จุดด้อย

- ไม่สะท้อนพฤติกรรมการใช้งานจริงเท่า LRU
- อาจแทนที่ข้อมูลที่ยังมีประโยชน์อยู่

### 7.5 Least Frequently Used (LFU)

LFU จะเลือกแทนที่บรรทัดที่ถูกใช้งานน้อยที่สุดในช่วงเวลาที่ติดตาม

#### จุดเด่น

- เหมาะในกรณีที่ข้อมูลบางชุดมีความถี่การใช้งานต่างกันชัดเจน

#### จุดด้อย

- ต้องมีตัวนับ (Counters)
- เพิ่มความซับซ้อนของฮาร์ดแวร์
- ข้อมูลที่เคยถูกใช้บ่อยในอดีตอาจยังค้างคาอยู่ แม้ปัจจุบันจะไม่สำคัญแล้ว

### 7.6 Random

การแทนที่แบบสุ่มคือสุ่มเลือกบรรทัดหนึ่งออกจากกลุ่มที่เลือกได้

#### จุดเด่น

- ฮาร์ดแวร์ง่ายมาก
- ลดภาระในการติดตามประวัติการใช้งาน

#### จุดด้อย

- คุณภาพการตัดสินใจไม่แน่นอน
- อาจแทนที่บรรทัดสำคัญโดยไม่จำเป็น

### 7.7 Pseudo-LRU

Pseudo-LRU เป็นวิธีประมาณพฤติกรรมของ LRU โดยใช้บิตสถานะจำนวนไม่มาก เพื่อหลีกเลี่ยงต้นทุนของ LRU จริง

**สาระสำคัญคือ ระบบพยายาม “เดาอย่างมีหลักการ” ว่าบรรทัดใดน่าจะเก่าที่สุด โดยไม่ต้องเก็บลำดับการใช้งานแบบละเอียดทั้งหมด**

วิธีนี้นิยมในแคชแบบหลายทาง (Higher-Associativity Caches) เพราะให้สมดุลที่ดีระหว่างประสิทธิภาพกับความซับซ้อน

### 7.8 ตารางเปรียบเทียบขั้นตอนวิธีแทนที่

| วิธี | หลักการ | ข้อดี | ข้อจำกัด |
|---|---|---|---|
| LRU | แทนที่บรรทัดที่ไม่ถูกใช้มานานที่สุด | สอดคล้องกับ Temporal Locality | ซับซ้อนเมื่อจำนวน ways สูง |
| FIFO | แทนที่บรรทัดที่เข้ามาก่อน | ทำง่าย | อาจไล่ข้อมูลที่ยังสำคัญออก |
| LFU | แทนที่บรรทัดที่ถูกใช้น้อยที่สุด | เห็นความถี่การใช้งาน | ต้องมีตัวนับและอาจล้าสมัย |
| Random | สุ่มแทนที่ | ง่ายและต้นทุนต่ำ | ประสิทธิภาพไม่แน่นอน |
| Pseudo-LRU | เลียนแบบ LRU แบบลดความซับซ้อน | คุ้มค่าทางฮาร์ดแวร์ | ไม่แม่นเท่า LRU จริง |

## 8. นโยบายการเขียนข้อมูล (Write Policy)

เมื่อซีพียูต้องการ “เขียน” ข้อมูลลงในแคช คำถามสำคัญคือ ข้อมูลใหม่ควรถูกส่งไปยังหน่วยความจำหลักด้วยทันทีหรือไม่ ประเด็นนี้มีผลทั้งต่อความเร็ว ความถูกต้องสอดคล้องของข้อมูล และปริมาณทราฟฟิกบนบัส

นโยบายหลักมี 2 แบบ คือ

- เขียนผ่าน (Write Through)
- เขียนกลับ (Write Back)

### 8.1 เขียนผ่าน (Write Through)

หลักการของ Write Through คือ **ทุกครั้งที่มีการเขียนลงแคช จะต้องเขียนลงหน่วยความจำหลักพร้อมกันด้วย**

#### ข้อดี

- ข้อมูลในหน่วยความจำหลักอัปเดตตลอดเวลา
- ช่วยให้การรักษาความสอดคล้องของแคช (Cache Coherence) ง่ายขึ้นในระบบหลายโปรเซสเซอร์
- หน่วยงานอื่นที่เฝ้าดูทราฟฟิกหน่วยความจำสามารถตรวจพบการเปลี่ยนแปลงได้ชัดเจน

#### ข้อเสีย

- เกิดทราฟฟิกบนบัสมาก
- การเขียนอาจช้าลง เพราะต้องอัปเดตสองตำแหน่ง
- ถ้างานมีการเขียนบ่อย ประสิทธิภาพอาจลดลงชัดเจน

### 8.2 เขียนกลับ (Write Back)

หลักการของ Write Back คือ **เมื่อมีการเขียน ระบบจะอัปเดตเฉพาะข้อมูลในแคชก่อน ส่วนหน่วยความจำหลักจะถูกอัปเดตก็ต่อเมื่อบล็อกนั้นถูกแทนที่ออกจากแคช**

โดยทั่วไปจะมีบิตสกปรก (Dirty Bit / Update Bit) เพื่อระบุว่าบรรทัดนั้นถูกแก้ไขแล้วหรือยัง

#### ขั้นตอนโดยสรุป

- หากซีพียูเขียนข้อมูลลงบรรทัดแคช ให้ตั้งค่า Dirty Bit
- เมื่อถึงเวลาต้องแทนที่บรรทัดนั้น
- ถ้า Dirty Bit เป็น 1 จึงค่อยเขียนข้อมูลกลับไปยังหน่วยความจำหลัก
- ถ้า Dirty Bit เป็น 0 ก็ทิ้งได้ทันที เพราะข้อมูลในหน่วยความจำหลักยังตรงอยู่

#### ข้อดี

- ลดจำนวนการเขียนลงหน่วยความจำหลัก
- ลดทราฟฟิกบนบัส
- ให้ประสิทธิภาพดีในงานที่มีการเขียนซ้ำในบล็อกเดิมหลายครั้ง

#### ข้อเสีย

- การควบคุมซับซ้อนกว่า
- หน่วยความจำหลักอาจยังไม่สะท้อนค่าปัจจุบันในทันที
- ระบบหลายแคชหรืออุปกรณ์อินพุตเอาต์พุต (Input/Output: I/O) ต้องจัดการความสอดคล้องอย่างระมัดระวัง

### 8.3 ตารางเปรียบเทียบนโยบายการเขียน

| ประเด็น | Write Through | Write Back |
|---|---|---|
| การอัปเดตหน่วยความจำหลัก | ทันทีทุกครั้ง | เมื่อบล็อกถูกแทนที่ |
| ทราฟฟิกบนบัส | สูง | ต่ำกว่า |
| ความง่ายในการรักษาความสอดคล้อง | ง่ายกว่า | ยากกว่า |
| ประสิทธิภาพด้านการเขียน | มักช้ากว่า | มักเร็วกว่า |
| ความซับซ้อนของฮาร์ดแวร์ | ต่ำกว่า | สูงกว่า |

### 8.4 ขยายความเพิ่มเติม

ในตำราและการออกแบบจริง มักมีประเด็นต่อเนื่องร่วมด้วย เช่น

- การเขียนเมื่อมิสแล้วจัดสรรบล็อกใหม่ (Write Allocate)
- การเขียนเมื่อมิสโดยไม่ดึงบล็อกเข้าแคช (No-Write Allocate)
- การใช้บัฟเฟอร์เขียน (Write Buffer)

แม้หัวข้อหลักในส่วนนี้เน้น Write Through กับ Write Back เป็นแกนสำคัญ แต่ในทางสถาปัตยกรรม ระบบจริงมักพิจารณากลไกเสริมเหล่านี้ร่วมด้วย เพื่อควบคุมคอขวดด้านการเขียนให้ดียิ่งขึ้น

### 8.5 Write Buffer กับผลเชิงสมรรถนะ

ในการออกแบบจริง บัฟเฟอร์เขียน (Write Buffer) มักถูกใช้ร่วมกับ Write Through หรือบางกรณีกับ Write Back เพื่อป้องกันไม่ให้ซีพียูต้องหยุดรอทุกครั้งที่เกิดการส่งข้อมูลลงหน่วยความจำหลัก

บทบาทของ Write Buffer คือ

- รับข้อมูลที่ต้องเขียนออกจากแคชก่อน
- เปิดทางให้ซีพียูทำงานต่อได้
- ให้การเขียนจริงลงหน่วยความจำหลักเกิดขึ้นแบบเหลื่อมเวลา

อย่างไรก็ตาม หากบัฟเฟอร์เต็ม ซีพียูก็ยังอาจต้องหยุดรออยู่ดี ดังนั้นกลไกนี้ช่วยลดคอขวดได้ แต่ไม่ได้ลบปัญหาออกไปโดยสมบูรณ์

## 9. ขนาดของบรรทัดแคช (Line Size / Block Size)

Line Size หรือ Block Size หมายถึงจำนวนข้อมูลที่ถูกย้ายระหว่างหน่วยความจำหลักกับแคชในหนึ่งครั้ง ไม่ได้ดึงเพียงคำ (Word) ที่ต้องการเท่านั้น แต่ดึงข้อมูลข้างเคียงที่อยู่ติดกันมาด้วย

แนวคิดนี้ตั้งอยู่บนหลักการเฉพาะที่เชิงพื้นที่ (Spatial Locality) ซึ่งระบุว่า **หากโปรแกรมเข้าถึงข้อมูลตำแหน่งหนึ่ง มีโอกาสสูงที่จะเข้าถึงตำแหน่งใกล้เคียงในไม่ช้า**

### 9.1 เหตุผลที่ไม่ดึงมาแค่คำเดียว

- การดึงข้อมูลจากหน่วยความจำหลักมีต้นทุนแฝงสูง
- เมื่อดึงครั้งหนึ่งแล้ว จึงคุ้มที่จะดึงข้อมูลข้างเคียงมาด้วย
- โปรแกรมจำนวนมากอ่านคำสั่งและข้อมูลแบบต่อเนื่อง เช่น ลูป อาร์เรย์ และโครงสร้างข้อมูลเชิงลำดับ

### 9.2 ผลของการเพิ่มขนาดบล็อก

ในระยะแรก การเพิ่ม Line Size มักช่วยให้อัตราฮิตดีขึ้น เพราะข้อมูลที่อยู่ติดกันถูกนำเข้ามาพร้อมกัน แต่เมื่อบล็อกมีขนาดใหญ่เกินไป จะเกิดปัญหาหลายประการ

- จำนวนบล็อกที่ใส่ได้ในแคชลดลง
- ข้อมูลที่ถูกดึงมาบางส่วนอาจไม่ถูกใช้จริง
- อาจไล่บล็อกอื่นที่สำคัญออกเร็วเกินไป
- ค่าปรับจากการพลาดแคช (Miss Penalty) ต่อครั้งอาจสูงขึ้น เพราะต้องขนข้อมูลมากขึ้น

### 9.3 สาระสำคัญ

**Block Size ที่ใหญ่ขึ้นไม่ได้แปลว่าดีกว่าเสมอไป เพราะแม้จะช่วย Spatial Locality แต่ก็ลดจำนวนบล็อกทั้งหมดที่แคชเก็บได้**

### 9.4 แนวโน้มเชิงปฏิบัติ

จากแหล่งอ้างอิงในบทนี้ ขนาดประมาณ `8 ถึง 64 ไบต์` ถูกมองว่าเหมาะสมในบริบทคลาสสิก ขณะที่ระบบสมรรถนะสูง (High-Performance Computing: HPC) และโปรเซสเซอร์สมัยใหม่มักใช้ขนาด `64 ไบต์` หรือ `128 ไบต์` เป็นเรื่องปกติ

### 9.5 ตารางวิเคราะห์ผลของขนาดบล็อก

| ขนาดบล็อก | ผลเชิงบวก | ผลเชิงลบ |
|---|---|---|
| เล็ก | เวลาโหลดต่อครั้งต่ำ เก็บจำนวนบล็อกได้มาก | ใช้ Spatial Locality ได้น้อย |
| ปานกลาง | สมดุลระหว่างจำนวนบล็อกกับข้อมูลข้างเคียง | ต้องออกแบบให้เหมาะกับเวิร์กโหลด |
| ใหญ่ | เก็บข้อมูลติดกันมาได้มาก | ลดจำนวนบล็อกในแคช เพิ่ม Miss Penalty และเสี่ยงดึงข้อมูลเกินจำเป็น |

## 10. จำนวนระดับของแคช (Number of Caches / Multilevel Caches)

เมื่อเทคโนโลยีวงจรรวมมีความหนาแน่นสูงขึ้น นักออกแบบสามารถวางแคชไว้บนชิปได้มากขึ้น จึงเกิดแนวคิดของแคชหลายระดับ (Multilevel Cache)

โครงสร้างที่พบทั่วไป ได้แก่

- แคชระดับที่ 1 (L1 Cache)
- แคชระดับที่ 2 (L2 Cache)
- แคชระดับที่ 3 (L3 Cache)

### 10.1 เหตุผลที่ต้องมีหลายระดับ

แคชระดับเดียวไม่สามารถตอบโจทย์ทุกอย่างพร้อมกันได้ เพราะ

- ถ้าทำให้เล็ก จะเร็ว แต่เก็บได้น้อย
- ถ้าทำให้ใหญ่ จะเก็บได้มาก แต่เวลาเข้าถึงมักสูงขึ้น

ดังนั้นจึงออกแบบเป็นลำดับชั้น

- `L1` มีขนาดเล็กมาก แต่เร็วที่สุด และอยู่ใกล้แกนประมวลผลมากที่สุด
- `L2` ใหญ่กว่า ช้ากว่า L1 แต่ยังเร็วกว่า DRAM มาก
- `L3` ใหญ่ยิ่งขึ้น มักใช้รองรับข้อมูลร่วม หรือเป็นตัวลดโอกาสที่ต้องลงไปถึงหน่วยความจำหลัก

### 10.2 หลักการทำงานโดยภาพรวม

เมื่อซีพียูร้องขอข้อมูล

1. ตรวจที่ L1 ก่อน
2. ถ้าไม่พบจึงตรวจ L2
3. หากยังไม่พบจึงตรวจ L3
4. ถ้ายังไม่พบอีกจึงไปยังหน่วยความจำหลัก

วิธีนี้ช่วยลดเวลาเข้าถึงเฉลี่ย (Average Memory Access Time) อย่างมาก เพราะแม้ L2 หรือ L3 จะช้ากว่า L1 แต่ก็ยังเร็วกว่า Main Memory มาก

### 10.3 ผลต่อสมรรถนะ

การมีหลายระดับช่วยทั้ง

- ลดความหน่วงแฝงเฉลี่ย (Average Latency)
- เพิ่มแบนด์วิดท์ (Bandwidth)
- ลดผลกระทบจากการพลาดแคชของระดับบน

**แก่นของแนวคิดคือ ให้ L1 รับงานที่ต้องเร็วมาก ส่วน L2 และ L3 ทำหน้าที่รองรับความจุ**

### 10.3.1 มุมมองผ่านค่าเวลาเข้าถึงเฉลี่ย (Average Access Time)

หากนิยามอย่างง่ายว่า

- `C` คือเวลาเข้าถึงแคช
- `M` คือค่าปรับจากการพลาดแคช (Miss Penalty)
- `h` คืออัตราฮิต (Hit Ratio)

เวลาเข้าถึงเฉลี่ยสามารถตีความได้ว่า

- กรณีฮิต ระบบเสียเวลาเพียงระดับแคช
- กรณีมิส ระบบต้องจ่ายเวลาค่าปรับเพิ่มเพื่อไปดึงข้อมูลจากระดับล่าง

จึงเข้าใจได้ว่า **การเพิ่มระดับแคชมีเป้าหมายเพื่อลดค่า Miss Penalty ที่ซีพียูต้องเผชิญเมื่อ L1 ไม่พบข้อมูล** เพราะถึงจะพลาด L1 แต่หากยังเจอใน L2 หรือ L3 ก็ยังดีกว่าต้องลงไปถึง Main Memory มาก

### 10.4 ขยายความเชิงสถาปัตยกรรม

ในอดีต L1 มักอยู่บนชิป และ L2 อยู่นอกชิปโดยใช้หน่วยความจำสแตติกแรม (Static Random Access Memory: SRAM) แต่เมื่อเทคโนโลยีก้าวหน้า L2 และแม้แต่ L3 ก็สามารถย้ายเข้ามาอยู่บนชิปได้เช่นกัน การเปลี่ยนแปลงนี้มีผลมหาศาลต่อการลดเวลาที่เกิดจากการเข้าถึงผ่านบัสภายนอก

## 11. แคชรวมกับแคชแยก (Unified Cache versus Split Cache)

อีกประเด็นหนึ่งของการออกแบบแคชคือ จะใช้แคชก้อนเดียวร่วมกันสำหรับทั้งคำสั่งและข้อมูล หรือจะแยกเป็นสองแคช

- แคชรวม (Unified Cache): ใช้พื้นที่เดียวเก็บทั้งคำสั่งและข้อมูล
- แคชแยก (Split Cache): แยกเป็นแคชคำสั่ง (Instruction Cache) และแคชข้อมูล (Data Cache)

### 11.1 ข้อดีของ Unified Cache

- ใช้พื้นที่ได้ยืดหยุ่นกว่า
- ถ้าโปรแกรมช่วงหนึ่งใช้คำสั่งมากแต่ใช้ข้อมูลน้อย พื้นที่แคชทั้งหมดสามารถช่วยรองรับคำสั่งได้
- ถ้าอีกช่วงหนึ่งใช้ข้อมูลมาก ก็สามารถใช้พื้นที่แคชสนับสนุนข้อมูลได้
- ผู้ออกแบบมีแคชเพียงก้อนเดียวให้จัดการ

**จุดแข็งสำคัญของ Unified Cache คือการปรับสมดุลภาระงานระหว่างคำสั่งกับข้อมูลโดยอัตโนมัติ**

### 11.2 ข้อดีของ Split Cache

- ลดการแย่งกันใช้งานระหว่างหน่วยดึงคำสั่ง (Fetch/Decode Unit) กับหน่วยประมวลผลข้อมูล (Execution Unit)
- เหมาะมากกับสถาปัตยกรรมไปป์ไลน์ (Pipelined Architecture)
- สามารถอ่านคำสั่งและข้อมูลพร้อมกันได้ง่ายกว่า
- ทำให้การออกแบบเส้นทางข้อมูลมีประสิทธิภาพขึ้น

### 11.3 ข้อพิจารณาเชิงสถาปัตยกรรม

ในไมโครโปรเซสเซอร์สมัยใหม่ มักพบว่า

- L1 แยกเป็น Instruction Cache และ Data Cache
- L2 หรือ L3 อาจเป็น Unified Cache

รูปแบบนี้สะท้อนแนวคิดประนีประนอมเช่นเดียวกับประเด็นอื่นในแคช กล่าวคือ ระดับที่ต้องเร็วมากและทำงานพร้อมกันหลายทิศทางมักแยก ส่วนระดับที่เน้นความจุรวมมักใช้ร่วม

### 11.4 ตารางเปรียบเทียบ Unified กับ Split Cache

| ประเด็น | Unified Cache | Split Cache |
|---|---|---|
| การใช้พื้นที่ | ยืดหยุ่น | แบ่งตายตัวระหว่างคำสั่งกับข้อมูล |
| การแย่งทรัพยากร | มีโอกาสแย่งกัน | ลดการแย่งกัน |
| ความเหมาะกับไปป์ไลน์ | ปานกลาง | สูง |
| การออกแบบระบบ | ง่ายกว่าในภาพรวม | ซับซ้อนกว่า แต่มีประสิทธิภาพสูง |
| ฮิตเรตในบางงาน | อาจดีเพราะใช้พื้นที่ร่วม | อาจด้อยกว่าหากการแบ่งไม่สมดุล |

## 12. ความสัมพันธ์ระหว่างองค์ประกอบทั้งหมด

การออกแบบแคชไม่ควรถูกมองเป็นหัวข้อย่อยที่แยกจากกัน เพราะทุกองค์ประกอบส่งผลต่อกัน เช่น

- ถ้าเพิ่ม `Cache Size` อาจต้องทบทวน `Hit Time`
- ถ้าเลือก `Fully Associative` อาจต้องใช้ `Replacement Algorithm` ที่ซับซ้อนขึ้น
- ถ้าเลือก `Write Back` ต้องวางกลไก `Dirty Bit` และดูแล `Cache Coherence`
- ถ้าเพิ่ม `Line Size` อาจลด `Miss Rate` บางชนิด แต่เพิ่ม `Miss Penalty`
- ถ้าใช้ `Split Cache` อาจช่วยไปป์ไลน์ แต่ต้องจัดสรรทรัพยากรให้พอดี

**ดังนั้น สถาปนิกคอมพิวเตอร์ต้องพิจารณาแคชในฐานะ “ระบบเชิงสมดุล” ไม่ใช่ชุดตัวเลือกอิสระ**

## 13. สรุปเชิงวิชาการของหัวข้อ 4.3

หัวข้อ 4.3 องค์ประกอบของการออกแบบแคช (Elements of Cache Design) แสดงให้เห็นว่า ประสิทธิภาพของแคชไม่ได้ขึ้นกับการมีหน่วยความจำความเร็วสูงเพียงอย่างเดียว แต่ขึ้นอยู่กับการออกแบบรายละเอียดภายในอย่างเป็นระบบ ตั้งแต่ขนาดของแคช วิธีแมประหว่างหน่วยความจำหลักกับแคช ขั้นตอนวิธีการแทนที่ข้อมูล นโยบายการเขียน ขนาดบล็อก ไปจนถึงการจัดแคชแบบหลายระดับและการเลือกใช้แคชรวมกับแคชแยก

สาระสำคัญที่สุดคือ

- **แคชที่ดีต้องเข้าถึงเร็ว**
- **ต้องมีอัตราฮิตสูง**
- **ต้องลดภาระของหน่วยความจำหลัก**
- **ต้องไม่เพิ่มความซับซ้อนของฮาร์ดแวร์จนเกินประโยชน์ที่ได้รับ**

ในทางสถาปัตยกรรมจริง จึงมักใช้แนวทางแบบผสมผสาน เช่น

- L1 ขนาดเล็กและเร็ว
- ใช้ Set-Associative Mapping
- ใช้ LRU หรือ Pseudo-LRU
- ใช้ Write Back เพื่อลดทราฟฟิก
- ใช้แคชหลายระดับ
- ใช้ Split Cache ในระดับต้นเพื่อรองรับไปป์ไลน์

**กล่าวโดยสรุป แคชเป็นตัวอย่างชัดเจนของการออกแบบฮาร์ดแวร์ที่ต้องอาศัยการประนีประนอมอย่างมีหลักการระหว่างความเร็ว ความยืดหยุ่น ความถูกต้อง และต้นทุน**

## 14. หมายเหตุสำหรับการอ่านเชิงลึก

เพื่อให้เข้าใจหัวข้อนี้ในระดับลึก ควรมององค์ประกอบแต่ละข้อผ่านคำถามต่อไปนี้

- ถ้าเพิ่มความยืดหยุ่น จะทำให้วงจรช้าลงเพียงใด
- ถ้าลดทราฟฟิกการเขียน จะกระทบความสอดคล้องของข้อมูลหรือไม่
- ถ้าดึงข้อมูลครั้งละมากขึ้น จะช่วย Spatial Locality จริงในเวิร์กโหลดนั้นหรือไม่
- ถ้าเพิ่มจำนวนระดับแคช ค่าใช้จ่ายด้านพื้นที่ พลังงาน และความซับซ้อนคุ้มค่าหรือไม่

คำถามเหล่านี้สะท้อนว่า **การออกแบบแคชไม่ใช่เพียงการท่องจำคำจำกัดความ แต่เป็นการวิเคราะห์เชิงวิศวกรรมระบบอย่างแท้จริง**
