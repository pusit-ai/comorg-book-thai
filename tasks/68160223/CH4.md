บทที่ 4: หน่วยความจำแคช (Cache Memory)
4.1 ภาพรวมของระบบหน่วยความจำคอมพิวเตอร์ (Computer Memory System Overview)
คุณลักษณะของระบบหน่วยความจำ (Characteristics of Memory Systems) ระบบหน่วยความจำคอมพิวเตอร์มีความซับซ้อนและสามารถจำแนกตามคุณลักษณะสำคัญได้หลายประการ เช่น ตำแหน่ง (Location) ความจุ (Capacity) และหน่วยของการถ่ายโอนข้อมูล (Unit of Transfer) โดยตำแหน่งของหน่วยความจำแบ่งออกเป็นหน่วยความจำภายใน (Internal Memory) เช่น เรจิสเตอร์ และหน่วยความจำหลัก และหน่วยความจำภายนอก (External Memory) เช่น อุปกรณ์จัดเก็บข้อมูลสำรอง
ลำดับชั้นของหน่วยความจำ (The Memory Hierarchy) ในการออกแบบหน่วยความจำ ผู้ออกแบบต้องเผชิญกับข้อจำกัดสามด้านคือ ความจุ ความเร็ว และราคา เพื่อแก้ปัญหาความขัดแย้งนี้ จึงมีการใช้ ลำดับชั้นของหน่วยความจำ (Memory Hierarchy) เมื่อลำดับชั้นต่ำลงจะส่งผลให้:
• ราคาต่อบิตลดลง
• ความจุเพิ่มขึ้น
• เวลาในการเข้าถึง (Access Time) เพิ่มขึ้น
• ความถี่ในการเข้าถึงหน่วยความจำโดยโปรเซสเซอร์ลดลง
หัวใจสำคัญของความสำเร็จในโครงสร้างนี้คือ หลักการอ้างอิงตำแหน่งที่ใกล้เคียงกัน (Locality of Reference) ซึ่งระบุว่าในระหว่างการประมวลผล การอ้างอิงหน่วยความจำมักจะมีลักษณะเป็นกลุ่มก้อน (Clusters)
4.2 หลักการของหน่วยความจำแคช (Cache Memory Principles)
หน่วยความจำแคชถูกออกแบบมาเพื่อรวมข้อดีของเวลาการเข้าถึงที่รวดเร็วของหน่วยความจำราคาแพง เข้ากับความจุขนาดใหญ่ของหน่วยความจำราคาถูก เมื่อโปรเซสเซอร์ต้องการอ่านข้อมูล จะมีการตรวจสอบในแคชก่อน หากพบข้อมูล (Hit) ข้อมูลจะถูกส่งให้โปรเซสเซอร์ทันที หากไม่พบ (Miss) บล็อกข้อมูลจากหน่วยความจำหลักจะถูกอ่านเข้ามาไว้ในแคช
โครงสร้างของระบบแคชประกอบด้วย:
• Main Memory: ประกอบด้วยบล็อกขนาดคงที่จำนวน บล็อก
• Cache: ประกอบด้วยบรรทัด (Lines) จำนวน บรรทัด ซึ่งแต่ละบรรทัดจะมีส่วนของ Tag เพื่อระบุว่าเก็บข้อมูลจากบล็อกใดของหน่วยความจำหลักไว้
4.3 องค์ประกอบในการออกแบบแคช (Elements of Cache Design)
ที่อยู่ของแคช (Cache Addresses) ผู้ออกแบบสามารถเลือกวางแคชไว้ระหว่างโปรเซสเซอร์กับหน่วยจัดการหน่วยความจำ (MMU) ซึ่งเรียกว่า Logical Cache หรือวางไว้ระหว่าง MMU กับหน่วยความจำหลัก เรียกว่า Physical Cache
ขนาดของแคช (Cache Size) ขนาดของแคชควรเล็กพอเพื่อให้ราคาเฉลี่ยต่อบิตใกล้เคียงกับหน่วยความจำหลัก แต่ต้องใหญ่พอเพื่อให้เวลาการเข้าถึงเฉลี่ยใกล้เคียงกับความเร็วของแคช
ฟังก์ชันการจับคู่ (Mapping Function) เนื่องจากจำนวนบรรทัดของแคชมีน้อยกว่าจำนวนบล็อกของหน่วยความจำหลัก จึงจำเป็นต้องมีฟังก์ชันการจับคู่ข้อมูล 3 รูปแบบหลัก:
1. Direct Mapping: กำหนดให้แต่ละบล็อกของหน่วยความจำหลักสามารถวางในบรรทัดแคชที่ระบุได้เพียงตำแหน่งเดียวเท่านั้น ง่ายต่อการติดตั้งแต่เสี่ยงต่อการเกิดการสลับข้อมูลซ้ำซ้อน (Thrashing)
2. Associative Mapping: อนุญาตให้บล็อกข้อมูลใดๆ สามารถวางในบรรทัดใดก็ได้ของแคช มีความยืดหยุ่นสูงแต่ต้องใช้วงจรที่ซับซ้อนในการตรวจสอบ Tag
3. Set-Associative Mapping: เป็นการผสมผสานข้อดีของสองวิธีแรก โดยแบ่งแคชออกเป็นเซต (Sets) และในแต่ละเซตจะประกอบด้วยบรรทัดข้อมูลหลายบรรทัด
อัลกอริทึมการแทนที่ (Replacement Algorithms) เมื่อแคชเต็มและต้องการนำข้อมูลใหม่เข้ามา ต้องมีกระบวนการเลือกบรรทัดที่จะถูกแทนที่ อัลกอริทึมที่นิยมที่สุดคือ Least Recently Used (LRU) ซึ่งจะแทนที่บรรทัดที่ไม่ได้ถูกอ้างถึงนานที่สุด วิธีอื่นๆ ได้แก่ FIFO, LFU และการสุ่ม (Random)
นโยบายการเขียน (Write Policy) เมื่อมีการแก้ไขข้อมูลในแคช จำเป็นต้องรักษาสม่ำเสมอของข้อมูลระหว่างแคชกับหน่วยความจำหลัก:
• Write Through: เขียนข้อมูลลงทั้งในแคชและหน่วยความจำหลักพร้อมกัน
• Write Back: เขียนข้อมูลลงในแคชเท่านั้น และจะเขียนกลับลงหน่วยความจำหลักเมื่อบรรทัดนั้นถูกแทนที่
ขนาดของบรรทัด (Line Size) การเพิ่มขนาดบรรทัดช่วยเพิ่มอัตราการพบข้อมูล (Hit Ratio) เนื่องด้วยหลักการ Locality แต่หากใหญ่เกินไป อัตราการพบข้อมูลจะเริ่มลดลงเนื่องจากข้อมูลที่นำเข้ามาอาจไม่ได้ถูกใช้งานจริง
จำนวนของแคช (Number of Caches)
• Multilevel Caches: การใช้แคชหลายระดับ (L1, L2, L3) ช่วยเพิ่มประสิทธิภาพโดยลดภาระของระบบบัส
• Unified vs. Split Caches: แคชแบบแยก (Split Cache) แบ่งเป็นแคชสำหรับคำสั่งและแคชสำหรับข้อมูล ช่วยลดปัญหาการแย่งชิงทรัพยากรในระบบ Pipeline
4.4 / Pentium 4 การจัดการแคช

การทำให้แคช L1 ส่งผลต่อประสิทธิภาพนั้นถือว่ามีเหตุผล หากแคช L2 มีขนาดบรรทัด (line size) และความจุเท่ากับแคช L1 เนื้อหาภายในแคช L2 ก็จะใกล้เคียงหรือคล้ายกับแคช L1 โดยรวม

ด้วยความที่พื้นที่บนชิปพร้อมใช้งานสำหรับแคชมีมากขึ้น ไมโครโปรเซสเซอร์สมัยใหม่ส่วนใหญ่จึงย้ายแคช L2 มาไว้บนชิป และเพิ่มแคช L3 เข้ามา เดิมที แคช L3 สามารถเข้าถึงได้ผ่านบัสภายนอก ต่อมาไมโครโปรเซสเซอร์ส่วนใหญ่ได้รวมแคช L3 ไว้บนชิปด้วยเช่นกัน ไม่ว่าจะกรณีใดก็ตาม ดูเหมือนว่าจะมีข้อได้เปรียบด้านประสิทธิภาพเมื่อเพิ่มระดับแคชที่สาม (เช่น ดู [GHAI98]) นอกจากนี้ ระบบขนาดใหญ่ เช่นระบบเมนเฟรม IBM zEnterprise ปัจจุบันมีแคช 3 ระดับบนชิป และมีแคชระดับที่สี่ที่ใช้ร่วมกันระหว่างหลายชิป [CURR11]

unified cache กับ split cache

เมื่อแคชบนชิปเริ่มปรากฏขึ้นใหม่ ๆ การออกแบบหลายแบบใช้แคชเดียวที่เก็บได้ทั้งข้อมูล (data) และคำสั่ง (instructions) ต่อมา ได้มีแนวโน้มแยกแคชออกเป็นสองส่วน: แคชคำสั่ง และแคชข้อมูล ทั้งสองอยู่ระดับเดียวกัน โดยปกติจะเป็นแคช L1 สองชุด เมื่อโปรเซสเซอร์พยายามดึงคำสั่งจากหน่วยความจำหลัก มันจะตรวจสอบแคชคำสั่ง L1 ก่อน และเมื่อโปรเซสเซอร์พยายามดึงข้อมูล ก็จะตรวจสอบแคชข้อมูล L1 ก่อนเช่นกัน

มีข้อดีที่อาจเกิดขึ้นของ unified cache ดังนี้:

สำหรับขนาดแคชที่กำหนด unified cache มีอัตราการ hit สูงกว่า split cache เพราะมันสามารถบาลานซ์โหลดระหว่างการดึงคำสั่งและข้อมูลโดยอัตโนมัติ กล่าวคือ ถ้ารูปแบบการทำงานมีการดึงคำสั่งมากกว่า แคชก็จะเต็มด้วยคำสั่ง และหากมีการดึงข้อมูลมากกว่า ก็จะเป็นตรงกันข้าม

มีแคชเพียงชุดเดียวที่ต้องออกแบบและสร้าง

แนวโน้มปัจจุบันคือใช้ split cache สำหรับ L1 และ unified cache สำหรับระดับสูงกว่า โดยเฉพาะในเครื่อง superscalar ที่เน้นการประมวลผลคำสั่งแบบขนานและการดึงคำสั่งล่วงหน้า (prefetch) ข้อดีหลักของ split cache คือการขจัดการแย่งกันเข้าถึงแคชระหว่างหน่วยดึงคำสั่ง (instruction fetch/decode unit) และหน่วยประมวลผล (execution unit) ซึ่งสำคัญมากในสถาปัตยกรรมที่ใช้ pipelining โดยปกติแล้ว โปรเซสเซอร์จะดึงคำสั่งล่วงหน้าเพื่อเติมในบัฟเฟอร์หรือ pipeline สมมติว่า หากใช้ unified cache เมื่อ execution unit ต้องอ่าน/เขียนข้อมูล คำขอนั้นจะถูกส่งไปที่แคช และถ้าในเวลาเดียวกัน instruction prefetcher ขออ่านคำสั่ง คำขอนั้นจะถูกบล็อกชั่วคราวเพื่อให้ execution unit ทำงานให้เสร็จ การแย่งแคชแบบนี้ทำให้ประสิทธิภาพลดลงเพราะไปขัดขวาง pipeline ของคำสั่ง การใช้ split cache จึงแก้ปัญหานี้ได้

4.4 การจัดการแคชของ Pentium 4

วิวัฒนาการของแคชสามารถเห็นได้ชัดจากวิวัฒนาการของไมโครโปรเซสเซอร์ Intel (ตาราง 4.4) 80386 ไม่มีแคชบนชิป 80486 มีแคชบนชิปขนาด 8 kB ใช้ line size 16 ไบต์ และเป็นแคชแบบ 4-way set associative

ตัวประมวลผลตระกูล Pentium ทั้งหมดมีแคช L1 สองชุด: แยกเป็นข้อมูลและคำสั่ง สำหรับ Pentium 4 แคช L1 (ข้อมูล) มีขนาด 16 kB, line size 64 ไบต์ และเป็นแบบ 4-way set associative ส่วนแคช L1 สำหรับคำสั่งจะกล่าวถึงภายหลัง Pentium II มี L2 cache ที่ป้อนข้อมูลให้ทั้งแคช L1 ขนาด 512 kB แบบ 8-way และ line size 128 ไบต์ มีการเพิ่ม L3 cache ใน Pentium III และนำมาไว้บนชิปใน Pentium 4 รุ่นระดับสูง

Appendix 4A — คุณลักษณะด้านประสิทธิภาพของหน่วยความจำแบบสองระดับ

ในบทนี้ได้กล่าวถึงแคชที่ทำหน้าที่เป็น buffer ระหว่างหน่วยความจำหลักกับโปรเซสเซอร์ ทำให้เกิดโครงสร้างหน่วยความจำแบบสองระดับ โครงสร้างนี้อาศัยคุณสมบัติที่เรียกว่า locality เพื่อให้ได้ประสิทธิภาพสูงขึ้นเมื่อเทียบกับหน่วยความจำระดับเดียว

กลไกแคชของหน่วยความจำหลักเป็นส่วนหนึ่งของสถาปัตยกรรมคอมพิวเตอร์ ซึ่งถูกสร้างขึ้นด้วยฮาร์ดแวร์และโดยทั่วไปจะไม่สามารถมองเห็นได้จากระบบปฏิบัติการ อย่างไรก็ตาม ยังมีกรณีอื่น ๆ ของแนวคิดหน่วยความจำสองระดับที่ใช้ประโยชน์จาก locality เช่นกัน และอย่างน้อยก็มีบางส่วนที่ถูกนำไปใช้ในระดับระบบปฏิบัติการ ได้แก่ virtual memory และ disk cache (ตาราง 4.6) Virtual memory จะกล่าวถึงในบทที่ 8 ส่วน disk cache ไม่ได้อยู่ในขอบเขตของหนังสือเล่มนี้ แต่ศึกษาเพิ่มเติมได้ใน [STAL15] ในภาคผนวกนี้ เราจะพิจารณาลักษณะด้านประสิทธิภาพของหน่วยความจำสองระดับ ซึ่งพบได้ในทั้งสามแนวทางนี้

Locality

พื้นฐานของข้อได้เปรียบด้านประสิทธิภาพของหน่วยความจำสองระดับ คือหลักการที่เรียกว่า locality of reference [DENN68] หลักการนี้กล่าวว่า การอ้างอิงหน่วยความจำมักเกิดขึ้นแบบเป็นกลุ่ม (cluster) เมื่อเวลาผ่านไปนาน ๆ กลุ่มที่ถูกใช้งานอาจเปลี่ยนไป แต่ในช่วงเวลาสั้น ๆ โปรเซสเซอร์จะทำงานอยู่กับชุดข้อมูลที่อยู่ในกลุ่มเดิมเป็นหลัก

ตามสัญชาตญาณ หลักการ locality ฟังดูสมเหตุสมผล ดังจะเห็นได้จากเหตุผลต่อไปนี้:

ยกเว้นคำสั่ง branch และ call ซึ่งมีสัดส่วนเพียงส่วนเล็ก ๆ ของคำสั่งทั้งหมด การทำงานของโปรแกรมจะเป็นแบบลำดับ ดังนั้น ส่วนใหญ่คำสั่งถัดไปที่ถูกดึงมา (fetch) จะอยู่ติดกับคำสั่งก่อนหน้า

แทบไม่เคยมีกรณีที่โปรแกรมมีการเรียกฟังก์ชัน (procedure call) ต่อเนื่องยาว ๆ แล้วตามด้วย sequence ของการคืนค่าทั้งหมด (returns) โดยปกติ โปรแกรมจะทำงานอยู่ในระดับความลึกของการเรียกฟังก์ชันที่ค่อนข้างแคบ ดังนั้นในช่วงเวลาสั้น ๆ การอ้างอิงคำสั่งมักจำกัดอยู่ในไม่กี่ฟังก์ชัน

โครงสร้างการทำงานแบบวนซ้ำ (loop) มักมีคำสั่งเพียงไม่กี่คำสั่งแต่ถูกทำซ้ำหลายครั้ง ดังนั้น ในช่วงเวลาที่วนลูป การประมวลผลจะกระจุกอยู่ในส่วนเล็ก ๆ ของโปรแกรม

เกี่ยวกับ ข้อสรุปที่ 2 (assertion 2) งานวิจัยที่รายงานใน [PATT85a] ได้ยืนยันข้อสรุปนี้ ภาพประกอบใน รูป 4.20 แสดงพฤติกรรมการเรียก/คืนค่าฟังก์ชัน (call–return behavior) โดยในภาพ:

การ call แสดงด้วยเส้นที่ลาดลงไปทางขวา

การ return แสดงด้วยเส้นที่ลาดขึ้นไปทางขวา

มีการกำหนด "หน้าต่าง" (window) ที่มีความลึก 5 ระดับ (depth = 5) ลำดับการ call และ return จะขยับหน้าต่างก็ต่อเมื่อมีการเคลื่อนที่สุทธิ (net movement) มากกว่า 6 ระดับขึ้นไปหรือ ลงไป เท่านั้น จากภาพจะเห็นว่าโปรแกรมสามารถทำงานภายในหน้าต่างที่นิ่ง (stationary window) ได้นานมาก

นักวิเคราะห์กลุ่มเดียวกันยังได้ศึกษาพฤติกรรมของโปรแกรมภาษา C และ Pascal และพบว่า หากกำหนดความลึกหน้าต่างเท่ากับ 8 จะมีความจำเป็นต้องเลื่อนหน้าต่างเพียง ไม่ถึง 1% ของเหตุการณ์ call หรือ return ทั้งหมด [TAMI83]

Spatial Locality และ Temporal Locality

ในวรรณกรรมได้แบ่ง locality ออกเป็น 2 ประเภท:

1. Spatial Locality (โลแคลิตีเชิงพื้นที่)

หมายถึงแนวโน้มที่การประมวลผลจะเข้าถึงตำแหน่งหน่วยความจำที่อยู่ใกล้กัน เช่น:

การเข้าถึงคำสั่งแบบลำดับ

การประมวลผลอาเรย์ ซึ่งข้อมูลที่เข้าถึงมักอยู่ติดกัน

2. Temporal Locality (โลแคลิตีเชิงเวลา)

หมายถึงแนวโน้มที่โปรเซสเซอร์จะเข้าถึงตำแหน่งหน่วยความจำเดิมซ้ำอีกครั้งในช่วงเวลาใกล้เคียง เช่น:

การวนลูปที่ใช้ชุดคำสั่งเดิมซ้ำ ๆ

รูป 4.20 แสดงพฤติกรรม call-return ของโปรแกรม:

แกนตั้ง: ความลึกของการซ้อนฟังก์ชัน (nesting depth)

แกนนอน: เวลา (หน่วยเป็นจำนวนครั้ง call/return)

หน้าต่าง (window) มีความลึก 5
ตัวอย่างนี้แสดงว่าโปรแกรมมักจะไม่กระโดดไปไกลจากความลึกเดิม ทำให้ locality มีอยู่จริง

การใช้ประโยชน์จาก Locality
Temporal Locality

ใช้ประโยชน์โดย:

เก็บค่าที่ใช้งานล่าสุดไว้ใน cache memory

ใช้ cache แบบหลายระดับ (L1, L2, L3)

Spatial Locality

ใช้ประโยชน์โดย:

ใช้ cache block ขนาดใหญ่ขึ้น

ใช้เทคนิค prefetching (ดึงข้อมูลล่วงหน้า)

แม้ว่าจะมีงานวิจัยใหม่ ๆ ปรับปรุงเทคนิคเหล่านี้ แต่หลักพื้นฐานก็ยังเหมือนเดิม

การทำงานของหน่วยความจำสองระดับ (Two-Level Memory)

สามารถใช้ locality เพื่อเพิ่มประสิทธิภาพได้ โดยมี:

M1: หน่วยความจำระดับบน (เช่น Cache) — เล็กกว่า เร็วกว่า แพงกว่า

M2: หน่วยความจำระดับล่าง (เช่น Main memory) — ใหญ่กว่า ช้ากว่า ถูกกว่า

ขั้นตอนทำงาน:

เมื่อมีการอ้างอิงข้อมูล จะตรวจสอบใน M1 ก่อน

ถ้า Hit → ดึงจาก M1 ได้ทันที (เร็ว)

ถ้า Miss → คัดลอกบล็อกข้อมูลจาก M2 มายัง M1 แล้วค่อยอ่านจาก M1

ด้วยหลัก locality เมื่อบล็อกถูกย้ายเข้ามาใน M1 จะมีการเข้าถึงข้อมูลภายในบล็อกนั้นหลายครั้ง ทำให้ระบบทำงานเร็วขึ้นโดยรวม
