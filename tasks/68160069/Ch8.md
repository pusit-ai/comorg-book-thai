## วัตถุประสงค์การเรียนรู้

หลังจากศึกษาบทนี้แล้ว ผู้เรียนควรสามารถ:

- สรุปหน้าที่หลักของระบบปฏิบัติการ (OS) ในระดับภาพรวมได้
- อภิปรายวิวัฒนาการของระบบปฏิบัติการ ตั้งแต่ระบบแบตช์แบบง่ายในยุคแรกจนถึงระบบสมัยใหม่ที่ซับซ้อน
- อธิบายความแตกต่างระหว่างการจัดตารางระยะยาว ระยะกลาง และระยะสั้น
- เข้าใจเหตุผลของการแบ่งพาร์ติชันหน่วยความจำ และอธิบายเทคนิคต่างๆ ที่ใช้
- ประเมินข้อได้เปรียบเปรียบเทียบของการแบ่งหน้าและการแบ่งส่วน
- นิยามความหมายของหน่วยความจำเสมือน

## บทนำ

แม้ว่าจุดเน้นหลักของตำราเล่มนี้จะเป็นฮาร์ดแวร์คอมพิวเตอร์ แต่มีด้านหนึ่งของซอฟต์แวร์ที่จำเป็นต้องกล่าวถึง นั่นคือระบบปฏิบัติการ (OS) ของคอมพิวเตอร์ ระบบปฏิบัติการคือโปรแกรมที่ทำหน้าที่จัดการทรัพยากรของคอมพิวเตอร์ ให้บริการแก่นักพัฒนาโปรแกรม และจัดตารางการทำงานของโปรแกรมอื่นๆ ความเข้าใจเกี่ยวกับระบบปฏิบัติการมีความจำเป็นอย่างยิ่งต่อการเข้าใจกลไกที่ CPU ใช้ควบคุมระบบคอมพิวเตอร์ โดยเฉพาะอย่างยิ่ง การอธิบายผลกระทบของการขัดจังหวะ (interrupt) และการจัดการลำดับชั้นของหน่วยความจำ (memory hierarchy) จะเข้าใจได้ดีที่สุดในบริบทนี้

บทนี้เริ่มต้นด้วยภาพรวมและประวัติโดยย่อของระบบปฏิบัติการ และส่วนหลักของบทจะพิจารณาหน้าที่สองประการของ OS ที่เกี่ยวข้องมากที่สุดกับการศึกษาองค์กรและสถาปัตยกรรมคอมพิวเตอร์ ได้แก่ การจัดตารางเวลา (scheduling) และการจัดการหน่วยความจำ (memory management)

## 8.1 ภาพรวมของระบบปฏิบัติการ

### วัตถุประสงค์และหน้าที่ของระบบปฏิบัติการ

ระบบปฏิบัติการ (OS) คือโปรแกรมที่ทำหน้าที่ควบคุมการทำงานของโปรแกรมประยุกต์ และทำหน้าที่เป็นส่วนต่อประสาน (interface) ระหว่างโปรแกรมประยุกต์กับฮาร์ดแวร์คอมพิวเตอร์ โดยอาจมองได้ว่ามีวัตถุประสงค์สองประการ คือ:

- **ความสะดวก (Convenience):** ระบบปฏิบัติการทำให้การใช้งานคอมพิวเตอร์สะดวกยิ่งขึ้น
- **ประสิทธิภาพ (Efficiency):** ระบบปฏิบัติการช่วยให้ทรัพยากรของระบบคอมพิวเตอร์ถูกใช้งานอย่างมีประสิทธิภาพ

#### ระบบปฏิบัติการในฐานะส่วนต่อประสานระหว่างผู้ใช้กับคอมพิวเตอร์

ฮาร์ดแวร์และซอฟต์แวร์ที่ใช้ในการให้บริการโปรแกรมประยุกต์แก่ผู้ใช้ สามารถมองได้ในลักษณะแบบลำดับชั้น (layered หรือ hierarchical) ดังแสดงในรูปที่ 8.1 ผู้ใช้ปลายทาง (end user) โดยทั่วไปจะไม่สนใจสถาปัตยกรรมภายในของคอมพิวเตอร์ ผู้ใช้ปลายทางจะมองระบบคอมพิวเตอร์ในรูปของโปรแกรมประยุกต์ โปรแกรมประยุกต์นั้นสามารถเขียนด้วยภาษาโปรแกรมและพัฒนาโดยนักพัฒนาโปรแกรมประยุกต์ (application programmer)

การพัฒนาโปรแกรมประยุกต์ในรูปของชุดคำสั่งของโปรเซสเซอร์ที่รับผิดชอบการควบคุมฮาร์ดแวร์คอมพิวเตอร์ทั้งหมดนั้นเป็นงานที่ซับซ้อนอย่างมาก เพื่อลดความยุ่งยากดังกล่าว จึงมีการจัดเตรียมชุดโปรแกรมระบบ (system programs) ไว้ให้ บางส่วนของโปรแกรมเหล่านี้เรียกว่า **ยูทิลิตี (utilities)** ซึ่งทำหน้าที่ประมวลผลฟังก์ชันที่ใช้บ่อยๆ เพื่อช่วยในการสร้างโปรแกรม การจัดการไฟล์ และการควบคุมอุปกรณ์ I/O นักพัฒนาโปรแกรมใช้งานสิ่งอำนวยความสะดวกเหล่านี้ในการพัฒนาโปรแกรมประยุกต์ และโปรแกรมประยุกต์ขณะทำงานก็จะเรียกใช้ยูทิลิตีเพื่อทำหน้าที่ต่างๆ โปรแกรมระบบที่สำคัญที่สุดคือ ระบบปฏิบัติการ ซึ่งทำหน้าที่ซ่อนรายละเอียดของฮาร์ดแวร์จากนักพัฒนาโปรแกรม และจัดเตรียมส่วนต่อประสานที่สะดวกสำหรับการใช้งานระบบ โดยทำหน้าที่เป็นตัวกลาง (mediator) ที่ทำให้การเข้าถึงและใช้งานสิ่งอำนวยความสะดวกและบริการต่างๆ ของนักพัฒนาและโปรแกรมประยุกต์สะดวกยิ่งขึ้น

โดยสรุป ระบบปฏิบัติการมักให้บริการในด้านต่างๆ ดังนี้:

- **การสร้างโปรแกรม (Program creation):** ระบบปฏิบัติการจัดเตรียมสิ่งอำนวยความสะดวกและบริการหลากหลาย เช่น โปรแกรมแก้ไขข้อความ (editors) และโปรแกรมดีบัก (debuggers) เพื่อช่วยนักพัฒนาในการสร้างโปรแกรม บริการเหล่านี้มักอยู่ในรูปของโปรแกรมยูทิลิตีที่ไม่ใช่ส่วนหนึ่งของ OS โดยตรง แต่สามารถเข้าถึงได้ผ่าน OS

- **การรันโปรแกรม (Program execution):** การรันโปรแกรมต้องมีการดำเนินการหลายขั้นตอน ได้แก่ การโหลดคำสั่งและข้อมูลเข้าหน่วยความจำหลัก การกำหนดค่าเริ่มต้นให้อุปกรณ์ I/O และไฟล์ รวมถึงการเตรียมทรัพยากรอื่นๆ OS จัดการทั้งหมดนี้แทนผู้ใช้

- **การเข้าถึงอุปกรณ์ I/O (Access to I/O devices):** อุปกรณ์ I/O แต่ละชนิดต้องการชุดคำสั่งหรือสัญญาณควบคุมเฉพาะของตนเอง OS จัดการรายละเอียดเหล่านี้ เพื่อให้นักพัฒนาสามารถมองการทำงานในรูปของการอ่านและเขียนข้อมูลที่เรียบง่าย

- **การเข้าถึงไฟล์อย่างมีการควบคุม (Controlled access to files):** การควบคุมไฟล์ต้องเข้าใจทั้งลักษณะของอุปกรณ์ I/O (เช่น ดิสก์ไดรฟ์ เทปไดรฟ์) และรูปแบบของไฟล์บนสื่อจัดเก็บข้อมูล OS ดูแลรายละเอียดเหล่านี้ นอกจากนี้ในระบบที่มีผู้ใช้หลายคนพร้อมกัน OS ยังสามารถจัดเตรียมกลไกการป้องกัน (protection mechanisms) เพื่อควบคุมการเข้าถึงไฟล์ได้

- **การเข้าถึงระบบ (System access):** ในระบบที่ใช้ร่วมกันหรือระบบสาธารณะ OS ทำหน้าที่ควบคุมการเข้าถึงระบบโดยรวมและทรัพยากรระบบเฉพาะส่วน ฟังก์ชันการเข้าถึงต้องจัดเตรียมการป้องกันทรัพยากรและข้อมูลจากผู้ใช้ที่ไม่ได้รับอนุญาต และต้องแก้ไขข้อขัดแย้งในการแย่งชิงทรัพยากร

- **การตรวจจับและตอบสนองต่อข้อผิดพลาด (Error detection and response):** ข้อผิดพลาดหลากหลายชนิดอาจเกิดขึ้นขณะระบบคอมพิวเตอร์ทำงาน ทั้งข้อผิดพลาดของฮาร์ดแวร์ภายในและภายนอก เช่น ข้อผิดพลาดของหน่วยความจำหรืออุปกรณ์ขัดข้อง รวมถึงข้อผิดพลาดของซอฟต์แวร์ต่างๆ เช่น การล้นของการคำนวณเลขคณิต การพยายามเข้าถึงตำแหน่งหน่วยความจำที่ไม่ได้รับอนุญาต และการที่ OS ไม่สามารถตอบสนองคำขอของโปรแกรมประยุกต์ได้ ในแต่ละกรณี OS ต้องดำเนินการตอบสนองที่แก้ไขสภาวะข้อผิดพลาดโดยกระทบต่อโปรแกรมที่กำลังทำงานให้น้อยที่สุด การตอบสนองอาจอยู่ในรูปของการยุติโปรแกรมที่ทำให้เกิดข้อผิดพลาด การลองทำงานใหม่ หรือเพียงแค่รายงานข้อผิดพลาดให้โปรแกรมประยุกต์ทราบ

- **การบัญชี (Accounting):** ระบบปฏิบัติการที่ดีจะเก็บสถิติการใช้งานทรัพยากรต่างๆ และตรวจสอบพารามิเตอร์ประสิทธิภาพ เช่น เวลาตอบสนอง (response time) ในระบบใดๆ ข้อมูลนี้มีประโยชน์ในการคาดการณ์ความต้องการปรับปรุงในอนาคต และในการปรับแต่งระบบเพื่อพัฒนาประสิทธิภาพ ในระบบที่มีผู้ใช้หลายคน ข้อมูลนี้ยังสามารถนำไปใช้เพื่อวัตถุประสงค์ทางการเงิน (billing) ได้อีกด้วย

รูปที่ 8.1 ยังแสดงส่วนต่อประสานหลักสามส่วนในระบบคอมพิวเตอร์ทั่วไป:

- **สถาปัตยกรรมชุดคำสั่ง (Instruction Set Architecture — ISA):** ISA กำหนดรายการคำสั่งภาษาเครื่อง (machine language instructions) ที่คอมพิวเตอร์สามารถดำเนินการได้ ส่วนต่อประสานนี้คือขอบเขตระหว่างฮาร์ดแวร์และซอฟต์แวร์ โดยทั้งโปรแกรมประยุกต์และยูทิลิตีสามารถเข้าถึง ISA โดยตรงได้ สำหรับโปรแกรมเหล่านี้ จะมีเพียงส่วนย่อยของรายการคำสั่งที่ใช้ได้ (user ISA) ส่วน OS จะมีสิทธิ์เข้าถึงคำสั่งภาษาเครื่องเพิ่มเติมที่เกี่ยวข้องกับการจัดการทรัพยากรระบบ (system ISA)

- **ส่วนต่อประสานไบนารีโปรแกรมประยุกต์ (Application Binary Interface — ABI):** ABI กำหนดมาตรฐานสำหรับความสามารถในการพกพาไฟล์ไบนารีข้ามโปรแกรม ABI กำหนดส่วนต่อประสานการเรียกระบบ (system call interface) ไปยังระบบปฏิบัติการ และทรัพยากรและบริการของฮาร์ดแวร์ที่มีในระบบผ่าน user ISA

- **ส่วนต่อประสานการเขียนโปรแกรมประยุกต์ (Application Programming Interface — API):** API ให้โปรแกรมสามารถเข้าถึงทรัพยากรและบริการของฮาร์ดแวร์ที่มีในระบบผ่าน user ISA ที่เสริมด้วยการเรียกใช้ไลบรารีภาษาระดับสูง (HLL) การเรียกระบบใดๆ มักดำเนินการผ่านไลบรารี การใช้ API ช่วยให้ซอฟต์แวร์ประยุกต์สามารถนำไปใช้งานบนระบบอื่นที่รองรับ API เดียวกันได้ง่าย ผ่านการคอมไพล์ใหม่

#### ระบบปฏิบัติการในฐานะผู้จัดการทรัพยากร

คอมพิวเตอร์คือชุดทรัพยากรสำหรับการเคลื่อนย้าย จัดเก็บ และประมวลผลข้อมูล รวมถึงการควบคุมฟังก์ชันเหล่านี้ OS มีหน้าที่รับผิดชอบในการจัดการทรัพยากรเหล่านี้

กลไกการควบคุมของ OS มีลักษณะพิเศษสองประการที่แตกต่างจากกลไกการควบคุมทั่วไป:

1. OS ทำงานในลักษณะเดียวกับซอฟต์แวร์คอมพิวเตอร์ทั่วไป กล่าวคือ เป็นโปรแกรมที่โปรเซสเซอร์ทำการรัน
2. OS มักสละการควบคุม (relinquish control) และต้องพึ่งพาโปรเซสเซอร์เพื่อให้สามารถกลับมาควบคุมได้อีกครั้ง

เช่นเดียวกับโปรแกรมคอมพิวเตอร์อื่นๆ OS ให้คำสั่งแก่โปรเซสเซอร์ ความแตกต่างหลักอยู่ที่เจตนาของโปรแกรม OS ชี้นำโปรเซสเซอร์ในการใช้ทรัพยากรระบบอื่นๆ และในการกำหนดเวลาการทำงานของโปรแกรมอื่นๆ แต่เพื่อให้โปรเซสเซอร์ดำเนินการเหล่านี้ได้ โปรเซสเซอร์ต้องหยุดทำงานโปรแกรม OS และไปรันโปรแกรมอื่นแทน ดังนั้น OS จึงสละการควบคุมเพื่อให้โปรเซสเซอร์ทำงาน "ที่มีประโยชน์" และจะกลับมาควบคุมอีกครั้งเมื่อต้องเตรียมโปรเซสเซอร์สำหรับงานชิ้นต่อไป

ส่วนหนึ่งของ OS จะอยู่ในหน่วยความจำหลักเสมอ ซึ่งรวมถึง **เคอร์เนล (kernel)** หรือ **นิวเคลียส (nucleus)** ที่บรรจุฟังก์ชันที่ใช้บ่อยที่สุดใน OS และส่วนอื่นๆ ของ OS ที่ใช้งานอยู่ในขณะนั้น ส่วนที่เหลือของหน่วยความจำหลักบรรจุโปรแกรมและข้อมูลของผู้ใช้ การจัดสรรทรัพยากรนี้ (หน่วยความจำหลัก) ถูกควบคุมร่วมกันโดย OS และฮาร์ดแวร์จัดการหน่วยความจำในโปรเซสเซอร์ OS ตัดสินใจว่าโปรแกรมที่กำลังทำงานสามารถใช้อุปกรณ์ I/O เมื่อใด และควบคุมการเข้าถึงและการใช้งานไฟล์ โปรเซสเซอร์เองก็เป็นทรัพยากร และ OS ต้องกำหนดว่าจะใช้เวลาของโปรเซสเซอร์เท่าใดในการทำงานโปรแกรมผู้ใช้ใดๆ ในระบบที่มีหลายโปรเซสเซอร์ การตัดสินใจนี้ต้องครอบคลุมโปรเซสเซอร์ทั้งหมด

### ประเภทของระบบปฏิบัติการ

คุณลักษณะสำคัญบางประการช่วยแยกแยะระบบปฏิบัติการประเภทต่างๆ คุณลักษณะเหล่านี้แบ่งออกเป็นสองมิติที่เป็นอิสระต่อกัน

**มิติที่หนึ่ง:** ระบุว่าระบบเป็นแบบแบตช์ (batch) หรือแบบโต้ตอบ (interactive)
- **ระบบโต้ตอบ (Interactive System):** ผู้ใช้/นักพัฒนาโต้ตอบกับคอมพิวเตอร์โดยตรง โดยปกติผ่านแป้นพิมพ์/เทอร์มินัลแสดงผล เพื่อร้องขอการทำงานของงาน (job) หรือดำเนินการธุรกรรม
- **ระบบแบตช์ (Batch System):** เป็นสิ่งตรงกันข้ามกับระบบโต้ตอบ โปรแกรมของผู้ใช้จะถูกรวมกับโปรแกรมจากผู้ใช้รายอื่น และส่งโดยผู้ดำเนินการคอมพิวเตอร์ (computer operator) หลังจากโปรแกรมทำงานเสร็จ ผลลัพธ์จะถูกพิมพ์ออกมาให้ผู้ใช้

**มิติที่สอง:** ระบุว่าระบบใช้การมัลติโปรแกรมมิ่ง (multiprogramming) หรือไม่
- **มัลติโปรแกรมมิ่ง (Multiprogramming):** พยายามให้โปรเซสเซอร์ทำงานอยู่ตลอดเวลาให้มากที่สุด โดยให้ทำงานกับโปรแกรมมากกว่าหนึ่งโปรแกรมในเวลาเดียวกัน โปรแกรมหลายๆ โปรแกรมจะถูกโหลดเข้าหน่วยความจำ และโปรเซสเซอร์จะสลับระหว่างโปรแกรมเหล่านั้นอย่างรวดเร็ว
- **ยูนิโปรแกรมมิ่ง (Uniprogramming):** ทำงานกับโปรแกรมเพียงหนึ่งโปรแกรมในแต่ละเวลา

#### ระบบยุคแรก (Early Systems)

ด้วยคอมพิวเตอร์รุ่นแรกๆ ตั้งแต่ช่วงปลายทศวรรษ 1940 จนถึงกลางทศวรรษ 1950 นักพัฒนาโปรแกรมโต้ตอบกับฮาร์ดแวร์คอมพิวเตอร์โดยตรง ยังไม่มี OS โปรเซสเซอร์เหล่านี้ถูกใช้งานจากคอนโซล ซึ่งประกอบด้วยไฟแสดงสถานะ สวิตช์แบบ toggle อุปกรณ์อินพุตบางรูปแบบ และเครื่องพิมพ์ โปรแกรมที่เขียนด้วยรหัสโปรเซสเซอร์จะถูกโหลดผ่านอุปกรณ์อินพุต (เช่น card reader) หากเกิดข้อผิดพลาดทำให้โปรแกรมหยุดทำงาน สภาวะข้อผิดพลาดจะแสดงผ่านไฟแสดงสถานะ นักพัฒนาสามารถตรวจสอบรีจิสเตอร์และหน่วยความจำหลักเพื่อหาสาเหตุของข้อผิดพลาด หากโปรแกรมทำงานเสร็จสมบูรณ์ตามปกติ ผลลัพธ์จะปรากฏบนเครื่องพิมพ์

ระบบยุคแรกเหล่านี้มีปัญหาหลักสองประการ:

1. **การจัดตาราง (Scheduling):** ส่วนใหญ่ใช้แผ่นลงชื่อจองเวลาโปรเซสเซอร์ ผู้ใช้อาจลงชื่อจองหนึ่งชั่วโมงแต่ทำงานเสร็จใน 45 นาที ส่งผลให้คอมพิวเตอร์ว่างโดยเปล่าประโยชน์ หรืออาจพบปัญหาและไม่สามารถทำงานเสร็จในเวลาที่กำหนด ทำให้ต้องหยุดก่อนแก้ปัญหาได้

2. **เวลาการตั้งค่า (Setup time):** โปรแกรมเดียวอาจต้องโหลดคอมไพเลอร์และโปรแกรมภาษาระดับสูง (source program) เข้าหน่วยความจำ บันทึกโปรแกรมที่คอมไพล์แล้ว (object program) จากนั้นโหลดและลิงก์โปรแกรม object พร้อมฟังก์ชันที่ใช้ร่วมกัน แต่ละขั้นตอนอาจต้องใส่หรือถอดเทป หรือจัดวางชุดบัตร (card decks) หากเกิดข้อผิดพลาด ผู้ใช้มักต้องกลับไปเริ่มต้นลำดับการตั้งค่าใหม่ตั้งแต่ต้น ทำให้สูญเสียเวลาจำนวนมากในการตั้งค่าโปรแกรมเพื่อให้พร้อมทำงาน

รูปแบบการทำงานนี้อาจเรียกว่า **การประมวลผลแบบอนุกรม (serial processing)** สะท้อนถึงข้อเท็จจริงที่ว่าผู้ใช้มีสิทธิ์เข้าถึงคอมพิวเตอร์ตามลำดับ เมื่อเวลาผ่านไป เครื่องมือซอฟต์แวร์ระบบต่างๆ ได้รับการพัฒนาเพื่อพยายามทำให้การประมวลผลแบบอนุกรมมีประสิทธิภาพมากขึ้น ซึ่งรวมถึงไลบรารีของฟังก์ชันที่ใช้ร่วมกัน ลิงก์เกอร์ โลดเดอร์ ดีบักเกอร์ และโปรแกรมขับเคลื่อนอุปกรณ์ I/O (I/O driver routines) ที่มีให้ใช้เป็นซอฟต์แวร์ส่วนกลางสำหรับผู้ใช้ทุกคน

#### ระบบแบตช์แบบง่าย (Simple Batch Systems)

โปรเซสเซอร์ยุคแรกมีราคาแพงมาก จึงมีความสำคัญอย่างยิ่งที่จะต้องใช้โปรเซสเซอร์ให้คุ้มค่าสูงสุด เวลาที่สูญเสียไปเนื่องจากการจัดตารางและการตั้งค่าเป็นสิ่งที่ยอมรับไม่ได้

เพื่อปรับปรุงการใช้งาน จึงได้มีการพัฒนาระบบปฏิบัติการแบบแบตช์แบบง่าย (simple batch operating systems) ด้วยระบบดังกล่าว ซึ่งเรียกอีกชื่อว่า **มอนิเตอร์ (monitor)** ผู้ใช้จะไม่มีสิทธิ์เข้าถึงโปรเซสเซอร์โดยตรงอีกต่อไป แต่ผู้ใช้จะส่งงานบนบัตรหรือเทปให้ผู้ดำเนินการคอมพิวเตอร์ ซึ่งจะรวมงานเข้าด้วยกันตามลำดับและวางทั้งชุดไว้บนอุปกรณ์อินพุตเพื่อให้มอนิเตอร์ใช้งาน

**มุมมองของมอนิเตอร์:** มอนิเตอร์ควบคุมลำดับของเหตุการณ์ เพื่อให้เป็นเช่นนี้ มอนิเตอร์ส่วนใหญ่ต้องอยู่ในหน่วยความจำหลักและพร้อมสำหรับการทำงานเสมอ ส่วนนั้นเรียกว่า **มอนิเตอร์ประจำ (resident monitor)** ส่วนที่เหลือของมอนิเตอร์ประกอบด้วยยูทิลิตีและฟังก์ชันที่ใช้ร่วมกัน ซึ่งจะถูกโหลดเป็นซับโปรแกรมเข้าในโปรแกรมผู้ใช้เมื่อเริ่มต้นงานใดๆ ที่ต้องการใช้งาน มอนิเตอร์อ่านงานทีละงานจากอุปกรณ์อินพุต เมื่ออ่านงานเข้ามาแล้ว งานปัจจุบันจะถูกวางในพื้นที่โปรแกรมผู้ใช้ และควบคุมการทำงานจะถูกส่งต่อให้งานนั้น เมื่องานเสร็จสิ้น ก็จะคืนการควบคุมให้มอนิเตอร์ ซึ่งจะอ่านงานถัดไปทันที ผลลัพธ์ของแต่ละงานจะถูกพิมพ์ออกมาเพื่อส่งมอบให้ผู้ใช้

มอนิเตอร์แก้ปัญหาการจัดตารางได้ โดยงานจะถูกจัดคิวและทำงานอย่างรวดเร็วที่สุดโดยไม่มีเวลาว่างระหว่างงาน และยังจัดการปัญหาเวลาการตั้งค่างานผ่าน **ภาษาควบคุมงาน (Job Control Language — JCL)** ซึ่งเป็นภาษาโปรแกรมพิเศษที่ใช้ให้คำสั่งแก่มอนิเตอร์

ฮาร์ดแวร์บางอย่างยังเป็นที่ต้องการสำหรับการทำงานของระบบแบตช์:

- **การป้องกันหน่วยความจำ (Memory protection):** ขณะที่โปรแกรมผู้ใช้กำลังทำงาน ต้องไม่แก้ไขพื้นที่หน่วยความจำที่บรรจุมอนิเตอร์ หากมีการพยายามดังกล่าว ฮาร์ดแวร์โปรเซสเซอร์ควรตรวจจับข้อผิดพลาดและโอนการควบคุมไปยังมอนิเตอร์
- **ตัวจับเวลา (Timer):** ใช้เพื่อป้องกันไม่ให้งานเดียวผูกขาดระบบ ตัวจับเวลาจะถูกตั้งค่าเมื่อเริ่มต้นแต่ละงาน หากตัวจับเวลาหมดเวลา จะเกิดการขัดจังหวะและการควบคุมจะคืนสู่มอนิเตอร์
- **คำสั่งสิทธิพิเศษ (Privileged instructions):** คำสั่งบางคำสั่งถูกกำหนดให้เป็นสิทธิพิเศษและสามารถรันได้โดยมอนิเตอร์เท่านั้น ในบรรดาคำสั่งสิทธิพิเศษรวมถึงคำสั่ง I/O เพื่อให้มอนิเตอร์รักษาการควบคุมอุปกรณ์ I/O ทั้งหมด
- **การขัดจังหวะ (Interrupts):** คอมพิวเตอร์รุ่นแรกๆ ไม่มีความสามารถนี้ ฟีเจอร์นี้ให้ความยืดหยุ่นมากขึ้นแก่ OS ในการสละและกลับมาควบคุมจากโปรแกรมผู้ใช้

#### ระบบแบตช์แบบมัลติโปรแกรม (Multiprogrammed Batch Systems)

แม้จะมีการจัดลำดับงานอัตโนมัติโดยระบบแบตช์แบบง่าย โปรเซสเซอร์มักจะว่างอยู่บ่อยครั้ง ปัญหาคืออุปกรณ์ I/O ทำงานช้ากว่าโปรเซสเซอร์มาก ตัวอย่างเช่น โปรแกรมที่ประมวลผลไฟล์ระเบียนโดยเฉลี่ย 100 คำสั่งโปรเซสเซอร์ต่อระเบียน คอมพิวเตอร์จะใช้เวลากว่า 96% รอให้อุปกรณ์ I/O ถ่ายโอนข้อมูลเสร็จสิ้น

วิธีแก้ไขคือการใช้ **มัลติโปรแกรมมิ่ง (multiprogramming)** หรือ **มัลติแทสกิ้ง (multitasking)** ซึ่งเป็นธีมหลักของระบบปฏิบัติการสมัยใหม่ หากมีพื้นที่สำหรับ OS และโปรแกรมผู้ใช้สองโปรแกรม เมื่องานหนึ่งต้องรอ I/O โปรเซสเซอร์สามารถสลับไปทำอีกงานหนึ่งที่ไม่รอ I/O ได้ ยิ่งไปกว่านั้น เราอาจขยายหน่วยความจำให้รองรับโปรแกรมสาม สี่ หรือมากกว่านั้น และสลับระหว่างโปรแกรมเหล่านั้นทั้งหมด

ระบบปฏิบัติการแบบมัลติโปรแกรมมิ่งค่อนข้างซับซ้อนเมื่อเทียบกับระบบโปรแกรมเดียว เพื่อให้มีงานหลายงานพร้อมทำงาน งานเหล่านั้นต้องถูกเก็บในหน่วยความจำหลัก ซึ่งต้องการการจัดการหน่วยความจำบางรูปแบบ นอกจากนี้ หากมีหลายงานพร้อมทำงาน โปรเซสเซอร์ต้องตัดสินใจว่าจะรันงานใด ซึ่งต้องใช้อัลกอริทึมสำหรับการจัดตาราง

#### ระบบแบ่งเวลา (Time-Sharing Systems)

ด้วยการใช้มัลติโปรแกรมมิ่ง การประมวลผลแบบแบตช์สามารถมีประสิทธิภาพสูง อย่างไรก็ตาม สำหรับงานหลายประเภท เป็นที่ต้องการให้มีโหมดที่ผู้ใช้โต้ตอบกับคอมพิวเตอร์โดยตรง

**ระบบแบ่งเวลา (time-sharing)** ใช้มัลติโปรแกรมมิ่งเพื่อจัดการงานโต้ตอบหลายงานพร้อมกัน โดยเวลาของโปรเซสเซอร์จะถูกแบ่งปันระหว่างผู้ใช้หลายคน ในระบบแบ่งเวลา ผู้ใช้หลายคนเข้าถึงระบบพร้อมกันผ่านเทอร์มินัล โดย OS แทรกสลับการทำงานของโปรแกรมผู้ใช้แต่ละรายในช่วงสั้นๆ หรือ "quantum" ของการคำนวณ ดังนั้นหากมีผู้ใช้ n รายร้องขอบริการในเวลาเดียวกัน ผู้ใช้แต่ละรายจะเห็นความเร็วคอมพิวเตอร์เฉลี่ยเพียง 1/n เท่านั้น (ไม่รวม overhead ของ OS) อย่างไรก็ตาม เมื่อพิจารณาจากเวลาตอบสนองที่ค่อนข้างช้าของมนุษย์ เวลาตอบสนองบนระบบที่ออกแบบมาอย่างเหมาะสมควรเทียบเคียงได้กับคอมพิวเตอร์เฉพาะบุคคล

## 8.2 การจัดตารางเวลา (Scheduling)

กุญแจสำคัญของมัลติโปรแกรมมิ่งคือการจัดตาราง โดยมีการจัดตารางสี่ประเภทที่เกี่ยวข้อง ก่อนอื่น จำเป็นต้องแนะนำแนวคิดของ **กระบวนการ (process)** ซึ่งคำนี้ถูกใช้ครั้งแรกโดยผู้ออกแบบ Multics OS ในช่วงทศวรรษ 1960 เป็นคำที่ทั่วไปกว่า "งาน (job)" คำนิยามต่างๆ ที่ให้ไว้สำหรับคำว่ากระบวนการ ได้แก่:

- โปรแกรมที่กำลังทำงาน (A program in execution)
- "วิญญาณที่เคลื่อนไหว" ของโปรแกรม (The "animated spirit" of a program)
- หน่วยงานที่โปรเซสเซอร์ได้รับมอบหมาย (That entity to which a processor is assigned)

**ตาราง 8.4 ประเภทของการจัดตาราง**

| ประเภท | คำอธิบาย |
|--------|----------|
| การจัดตารางระยะยาว (Long-term scheduling) | การตัดสินใจเพิ่มกระบวนการเข้าในกลุ่มที่รอดำเนินการ |
| การจัดตารางระยะกลาง (Medium-term scheduling) | การตัดสินใจเพิ่มจำนวนกระบวนการที่อยู่ในหน่วยความจำหลักบางส่วนหรือทั้งหมด |
| การจัดตารางระยะสั้น (Short-term scheduling) | การตัดสินใจว่ากระบวนการที่มีอยู่ใดจะถูกดำเนินการโดยโปรเซสเซอร์ |
| การจัดตาราง I/O (I/O scheduling) | การตัดสินใจว่าคำขอ I/O ที่รอดำเนินการของกระบวนการใดจะได้รับการจัดการโดยอุปกรณ์ I/O ที่ว่างอยู่ |

### การจัดตารางระยะยาว (Long-Term Scheduling)

ตัวจัดตารางระยะยาว (long-term scheduler) กำหนดว่าโปรแกรมใดได้รับอนุญาตให้เข้าสู่ระบบเพื่อประมวลผล จึงควบคุมระดับของมัลติโปรแกรมมิ่ง (จำนวนกระบวนการในหน่วยความจำ) เมื่อได้รับอนุญาต งานหรือโปรแกรมผู้ใช้จะกลายเป็นกระบวนการและถูกเพิ่มในคิวสำหรับตัวจัดตารางระยะสั้น ในบางระบบ กระบวนการที่สร้างขึ้นใหม่จะเริ่มต้นในสภาวะถูกสลับออก (swapped-out) ซึ่งในกรณีนี้จะถูกเพิ่มในคิวสำหรับตัวจัดตารางระยะกลาง

ในระบบแบตช์ งานที่ส่งใหม่จะถูกส่งไปยังดิสก์และเก็บในคิวแบตช์ ตัวจัดตารางระยะยาวสร้างกระบวนการจากคิวเมื่อเงื่อนไขอนุญาต การตัดสินใจที่เกี่ยวข้องมีสองประการ:

1. ตัวจัดตารางต้องตัดสินใจว่า OS สามารถรับกระบวนการเพิ่มเติมได้หรือไม่
2. ตัวจัดตารางต้องตัดสินใจว่าจะรับงานหรืองานใดบ้างและแปลงเป็นกระบวนการ

เกณฑ์ที่ใช้อาจรวมถึงลำดับความสำคัญ (priority) เวลาทำงานที่คาดหวัง และข้อกำหนด I/O

สำหรับโปรแกรมโต้ตอบในระบบแบ่งเวลา คำขอกระบวนการจะถูกสร้างขึ้นเมื่อผู้ใช้พยายามเชื่อมต่อกับระบบ OS จะยอมรับผู้ใช้ที่ได้รับอนุญาตทั้งหมดจนกว่าระบบจะอิ่มตัว โดยใช้มาตรวัดความอิ่มตัวที่กำหนดไว้ล่วงหน้า เมื่อถึงจุดนั้น คำขอการเชื่อมต่อจะได้รับข้อความแจ้งว่าระบบเต็มและให้ลองใหม่ในภายหลัง

### การจัดตารางระยะกลาง (Medium-Term Scheduling)

การจัดตารางระยะกลางเป็นส่วนหนึ่งของฟังก์ชันการสลับ (swapping) โดยทั่วไป การตัดสินใจสลับเข้า (swapping-in) ขึ้นอยู่กับความจำเป็นในการจัดการระดับของมัลติโปรแกรมมิ่ง ในระบบที่ไม่ใช้หน่วยความจำเสมือน การจัดการหน่วยความจำก็เป็นประเด็นสำคัญด้วย ดังนั้น การตัดสินใจสลับเข้าจะพิจารณาความต้องการหน่วยความจำของกระบวนการที่ถูกสลับออก

### การจัดตารางระยะสั้น (Short-Term Scheduling)

ตัวจัดตารางระยะยาวทำงานค่อนข้างไม่บ่อยและตัดสินใจในระดับหยาบว่าจะรับกระบวนการใหม่หรือไม่และควรรับตัวไหน ตัวจัดตารางระยะสั้น หรือที่รู้จักกันในชื่อ **ผู้จัดส่ง (dispatcher)** ทำงานบ่อยครั้งและตัดสินใจในระดับละเอียดว่าจะรันงานใดถัดไป

#### สถานะของกระบวนการ (Process States)

เพื่อทำความเข้าใจการทำงานของตัวจัดตารางระยะสั้น เราต้องพิจารณาแนวคิดของสถานะกระบวนการ (process state) ตลอดอายุของกระบวนการ สถานะของมันจะเปลี่ยนแปลงหลายครั้ง อย่างน้อยต้องมีห้าสถานะที่กำหนดสำหรับกระบวนการ (รูปที่ 8.7):

1. **ใหม่ (New):** โปรแกรมได้รับการยอมรับจากตัวจัดตารางระดับสูง แต่ยังไม่พร้อมทำงาน OS จะเริ่มต้นกระบวนการ ย้ายมันไปสู่สถานะพร้อม
2. **พร้อม (Ready):** กระบวนการพร้อมทำงานและรอการเข้าถึงโปรเซสเซอร์
3. **กำลังทำงาน (Running):** กระบวนการกำลังถูกดำเนินการโดยโปรเซสเซอร์
4. **รอ (Waiting):** กระบวนการถูกระงับการทำงานรอทรัพยากรระบบบางอย่าง เช่น I/O
5. **สิ้นสุด (Halted):** กระบวนการสิ้นสุดแล้วและจะถูกทำลายโดย OS

สำหรับแต่ละกระบวนการในระบบ OS ต้องรักษาข้อมูลที่ระบุสถานะของกระบวนการและข้อมูลอื่นๆ ที่จำเป็นสำหรับการทำงานของกระบวนการ เพื่อวัตถุประสงค์นี้ แต่ละกระบวนการจะถูกแทนด้วย **บล็อกควบคุมกระบวนการ (Process Control Block — PCB)** ซึ่งโดยทั่วไปประกอบด้วย:

- **ตัวระบุ (Identifier):** แต่ละกระบวนการปัจจุบันมีตัวระบุที่ไม่ซ้ำกัน
- **สถานะ (State):** สถานะปัจจุบันของกระบวนการ
- **ลำดับความสำคัญ (Priority):** ระดับความสำคัญสัมพัทธ์
- **ตัวนับโปรแกรม (Program counter):** ที่อยู่ของคำสั่งถัดไปในโปรแกรมที่จะทำงาน
- **ตัวชี้หน่วยความจำ (Memory pointers):** ตำแหน่งเริ่มต้นและสิ้นสุดของกระบวนการในหน่วยความจำ
- **ข้อมูลบริบท (Context data):** ข้อมูลที่อยู่ในรีจิสเตอร์ของโปรเซสเซอร์ขณะที่กระบวนการกำลังทำงาน
- **ข้อมูลสถานะ I/O (I/O status information):** รวมถึงคำขอ I/O ที่ค้างอยู่ อุปกรณ์ I/O ที่ถูกกำหนดให้กระบวนการนี้ รายการไฟล์ที่ถูกกำหนดให้กระบวนการ ฯลฯ
- **ข้อมูลการบัญชี (Accounting information):** อาจรวมถึงจำนวนเวลาโปรเซสเซอร์และเวลานาฬิกาที่ใช้ขีดจำกัดเวลา หมายเลขบัญชี ฯลฯ

#### เทคนิคการจัดตาราง (Scheduling Techniques)

OS รักษาคิวหลายคิว โดยแต่ละคิวคือรายชื่อกระบวนการที่รอทรัพยากรบางอย่าง:

- **คิวระยะยาว (Long-term queue):** รายชื่องานที่รอใช้ระบบ ตัวจัดตารางระดับสูงจะจัดสรรหน่วยความจำและสร้างกระบวนการสำหรับรายการที่รอ
- **คิวระยะสั้น (Short-term queue):** ประกอบด้วยกระบวนการทั้งหมดในสถานะพร้อม กระบวนการใดๆ เหล่านี้สามารถใช้โปรเซสเซอร์ต่อไปได้ โดยทั่วไปจะดำเนินการโดยใช้อัลกอริทึม round-robin ให้เวลาแก่แต่ละกระบวนการ และอาจใช้ระดับความสำคัญด้วย
- **คิว I/O (I/O queues):** มีคิว I/O สำหรับอุปกรณ์ I/O แต่ละชนิด กระบวนการที่รอใช้อุปกรณ์แต่ละชนิดจะเข้าคิวของอุปกรณ์นั้น

กระบวนการที่กำลังทำงานอาจถูกระงับด้วยเหตุผลต่างๆ:

1. **กระบวนการออกคำเรียกบริการ (service call) ไปยัง OS:** เช่น คำขอ I/O การทำงานของกระบวนการจะถูกระงับจนกว่าการเรียกนั้นจะได้รับการตอบสนองโดย OS
2. **กระบวนการทำให้เกิดการขัดจังหวะ (interrupt):** การขัดจังหวะคือสัญญาณที่สร้างโดยฮาร์ดแวร์ส่งไปยังโปรเซสเซอร์ สาเหตุต่างๆ เกี่ยวกับกระบวนการจะทำให้เกิดการขัดจังหวะ เช่น ข้อผิดพลาด หรือการหมดเวลา (timeout)
3. **เหตุการณ์ที่ไม่เกี่ยวข้องกับกระบวนการที่ต้องการความสนใจ:** เช่น การสิ้นสุดของการดำเนินการ I/O

## 8.3 การจัดการหน่วยความจำ (Memory Management)

ในระบบยูนิโปรแกรม หน่วยความจำหลักแบ่งออกเป็นสองส่วน: ส่วนหนึ่งสำหรับ OS (มอนิเตอร์ประจำ) และส่วนหนึ่งสำหรับโปรแกรมที่กำลังทำงาน ในระบบมัลติโปรแกรม ส่วน "ผู้ใช้" ของหน่วยความจำถูกแบ่งย่อยเพื่อรองรับกระบวนการหลายกระบวนการ งานการแบ่งย่อยนี้ดำเนินการแบบไดนามิกโดย OS และเรียกว่า **การจัดการหน่วยความจำ (memory management)**

การจัดการหน่วยความจำที่มีประสิทธิภาพมีความสำคัญอย่างยิ่งในระบบมัลติโปรแกรม หากมีเพียงไม่กี่กระบวนการในหน่วยความจำ ส่วนใหญ่ของเวลา กระบวนการทั้งหมดจะรอ I/O และโปรเซสเซอร์จะว่าง ดังนั้นต้องจัดสรรหน่วยความจำอย่างมีประสิทธิภาพเพื่อบรรจุกระบวนการให้มากที่สุดในหน่วยความจำ

### การสลับเข้าออก (Swapping)

แม้มัลติโปรแกรมมิ่งจะช่วยให้หน่วยความจำบรรจุกระบวนการหลายกระบวนการและโปรเซสเซอร์สามารถสลับไปกระบวนการอื่นเมื่อกระบวนการหนึ่งรอ แต่โปรเซสเซอร์เร็วกว่า I/O มากจนเป็นเรื่องปกติที่กระบวนการทั้งหมดในหน่วยความจำจะรอ I/O พร้อมกัน ดังนั้นแม้กับมัลติโปรแกรมมิ่ง โปรเซสเซอร์อาจว่างอยู่ส่วนใหญ่ของเวลา

**การสลับ (Swapping)** แก้ปัญหานี้โดยมีคิวระยะยาวของคำขอกระบวนการที่จัดเก็บบนดิสก์ และนำเข้ามาทีละรายการเมื่อมีพื้นที่ว่าง เมื่อกระบวนการทำงานเสร็จ ก็จะถูกย้ายออกจากหน่วยความจำหลัก หากไม่มีกระบวนการใดในหน่วยความจำอยู่ในสถานะพร้อม แทนที่จะว่างอยู่ โปรเซสเซอร์จะสลับกระบวนการหนึ่งกลับออกไปยังดิสก์สู่ **คิวกลาง (intermediate queue)** ซึ่งเป็นคิวของกระบวนการที่มีอยู่ที่ถูกย้ายออกจากหน่วยความจำชั่วคราว OS จากนั้นนำกระบวนการอื่นจากคิวกลางหรือรับคำขอกระบวนการใหม่จากคิวระยะยาว

การสลับเป็นการดำเนินการ I/O แต่เนื่องจาก I/O ดิสก์โดยทั่วไปเป็น I/O ที่เร็วที่สุดในระบบ การสลับมักจะเพิ่มประสิทธิภาพ โครงร่างที่ซับซ้อนกว่า ซึ่งเกี่ยวข้องกับหน่วยความจำเสมือน (virtual memory) จะปรับปรุงประสิทธิภาพเหนือการสลับแบบง่าย

### การแบ่งพาร์ติชัน (Partitioning)

โครงร่างที่ง่ายที่สุดสำหรับการแบ่งหน่วยความจำที่มีอยู่คือการใช้ **พาร์ติชันขนาดคงที่ (fixed-size partitions)** แม้ว่าพาร์ติชันจะมีขนาดคงที่ แต่ไม่จำเป็นต้องมีขนาดเท่ากัน เมื่อกระบวนการถูกนำเข้าหน่วยความจำ จะถูกวางในพาร์ติชันที่เล็กที่สุดที่ว่างอยู่และสามารถบรรจุมันได้

แม้ใช้พาร์ติชันขนาดคงที่ที่ไม่เท่ากัน ก็ยังมีหน่วยความจำที่สูญเสียไป ในกรณีส่วนใหญ่ กระบวนการจะไม่ต้องการหน่วยความจำเท่ากับที่พาร์ติชันจัดเตรียมไว้พอดี ตัวอย่างเช่น กระบวนการที่ต้องการ 3M bytes ของหน่วยความจำจะถูกวางในพาร์ติชัน 4M ทำให้สูญเสีย 1M ที่อาจใช้โดยกระบวนการอื่นได้

วิธีที่มีประสิทธิภาพมากกว่าคือการใช้ **พาร์ติชันขนาดแปรผัน (variable-size partitions)** เมื่อกระบวนการถูกนำเข้าหน่วยความจำ จะได้รับการจัดสรรหน่วยความจำเท่าที่ต้องการและไม่มากกว่านั้น

วิธีนี้เริ่มต้นได้ดี แต่ในที่สุดนำไปสู่สถานการณ์ที่มีช่องว่างเล็กๆ (holes) จำนวนมากในหน่วยความจำ เมื่อเวลาผ่านไป หน่วยความจำจะกระจัดกระจายมากขึ้นเรื่อยๆ และการใช้หน่วยความจำจะลดลง เทคนิคที่ใช้แก้ปัญหานี้คือ **การบดอัด (compaction):** OS จะย้ายกระบวนการในหน่วยความจำเป็นครั้งคราวเพื่อรวมหน่วยความจำว่างทั้งหมดไว้เป็นบล็อกเดียว

ก่อนพิจารณาวิธีจัดการข้อบกพร่องของการแบ่งพาร์ติชัน จำเป็นต้องชี้แจงประเด็นหนึ่ง กระบวนการไม่น่าจะถูกโหลดไปยังตำแหน่งเดิมในหน่วยความจำหลักทุกครั้งที่สลับเข้ามา ดังนั้นจึงมีการแยกแยะระหว่าง **ที่อยู่ตรรกะ (logical address)** และ **ที่อยู่กายภาพ (physical address)**:

- **ที่อยู่ตรรกะ (Logical address):** แสดงเป็นตำแหน่งสัมพัทธ์กับจุดเริ่มต้นของโปรแกรม คำสั่งในโปรแกรมบรรจุเฉพาะที่อยู่ตรรกะ
- **ที่อยู่กายภาพ (Physical address):** ตำแหน่งจริงในหน่วยความจำหลัก เมื่อโปรเซสเซอร์ทำงานกระบวนการ จะแปลงจากที่อยู่ตรรกะเป็นที่อยู่กายภาพโดยอัตโนมัติ โดยการบวก **ที่อยู่ฐาน (base address)** ปัจจุบันของกระบวนการเข้ากับแต่ละที่อยู่ตรรกะ

### การแบ่งหน้า (Paging)

ทั้งพาร์ติชันขนาดคงที่ไม่เท่ากันและพาร์ติชันขนาดแปรผันต่างก็ไม่มีประสิทธิภาพในการใช้หน่วยความจำ สมมติว่าหน่วยความจำถูกแบ่งออกเป็นชิ้นขนาดเท่ากันขนาดเล็กที่คงที่ และแต่ละกระบวนการก็ถูกแบ่งออกเป็นชิ้นขนาดเล็กขนาดคงที่เช่นกัน ชิ้นส่วนของโปรแกรม เรียกว่า **หน้า (pages)** สามารถกำหนดให้ชิ้นส่วนของหน่วยความจำที่ว่าง เรียกว่า **เฟรม (frames)** หรือ page frames ได้ อย่างมากที่สุด หน่วยความจำที่สูญเสียสำหรับกระบวนการนั้นจะเป็นเพียงเศษของหน้าสุดท้าย

เมื่อถึงเวลาที่จะโหลดกระบวนการ OS จะหาเฟรมว่างสี่เฟรมและโหลดสี่หน้าของกระบวนการ A เข้าในสี่เฟรม

หากไม่มีเฟรมที่ไม่ใช้งานที่ต่อเนื่องกันเพียงพอสำหรับบรรจุกระบวนการ OS ยังคงสามารถโหลดได้โดยใช้แนวคิดของที่อยู่ตรรกะ OS รักษา **ตารางหน้า (page table)** สำหรับแต่ละกระบวนการ ตารางหน้าแสดงตำแหน่งเฟรมสำหรับแต่ละหน้าของกระบวนการ ภายในโปรแกรม แต่ละที่อยู่ตรรกะประกอบด้วยหมายเลขหน้า (page number) และที่อยู่สัมพัทธ์ภายในหน้า (relative address within page)

กระบวนการแปลงที่อยู่ตรรกะเป็นที่อยู่กายภาพยังคงทำโดยฮาร์ดแวร์โปรเซสเซอร์ โดยโปรเซสเซอร์ต้องรู้วิธีเข้าถึงตารางหน้าของกระบวนการปัจจุบัน เมื่อได้รับที่อยู่ตรรกะ (หมายเลขหน้า, ที่อยู่สัมพัทธ์) โปรเซสเซอร์จะใช้ตารางหน้าเพื่อสร้างที่อยู่กายภาพ (หมายเลขเฟรม, ที่อยู่สัมพัทธ์)

วิธีนี้แก้ปัญหาที่กล่าวถึงก่อนหน้า:
- หน่วยความจำหลักถูกแบ่งออกเป็นเฟรมขนาดเท่ากันจำนวนมาก
- แต่ละกระบวนการถูกแบ่งออกเป็นหน้าขนาดเฟรม
- กระบวนการขนาดเล็กต้องการหน้าน้อยกว่า กระบวนการขนาดใหญ่ต้องการหน้ามากกว่า
- เมื่อกระบวนการถูกนำเข้า หน้าของมันจะถูกโหลดเข้าในเฟรมที่ว่างอยู่ และตารางหน้าจะถูกตั้งค่า

### หน่วยความจำเสมือน (Virtual Memory)

#### การแบ่งหน้าตามความต้องการ (Demand Paging)

ด้วยการใช้การแบ่งหน้า ระบบมัลติโปรแกรมที่มีประสิทธิภาพอย่างแท้จริงได้รับการพัฒนา ยิ่งไปกว่านั้น กลวิธีง่ายๆ ในการแบ่งกระบวนการออกเป็นหน้าได้นำไปสู่การพัฒนาแนวคิดสำคัญอีกประการหนึ่ง: **หน่วยความจำเสมือน (virtual memory)**

**การแบ่งหน้าตามความต้องการ (Demand paging)** หมายความว่าแต่ละหน้าของกระบวนการจะถูกนำเข้ามาเฉพาะเมื่อต้องการ นั่นคือ ตามความต้องการ

พิจารณากระบวนการขนาดใหญ่ที่ประกอบด้วยโปรแกรมยาวและอาร์เรย์ข้อมูลจำนวนมาก ในช่วงเวลาสั้นๆ ใดๆ การทำงานอาจจำกัดอยู่ในส่วนเล็กๆ ของโปรแกรม (เช่น ซับโปรแกรม) และอาจมีเพียงอาร์เรย์ข้อมูลหนึ่งหรือสองชุดที่กำลังใช้งาน นี่คือ **หลักการของความท้องถิ่น (principle of locality)** จะเป็นการสูญเปล่าหากโหลดหน้าจำนวนมากสำหรับกระบวนการนั้นเมื่อจะมีเพียงไม่กี่หน้าที่จะถูกใช้งานก่อนที่โปรแกรมจะถูกระงับ

หากโปรแกรมแตกสาขาไปยังคำสั่งในหน้าที่ไม่ได้อยู่ในหน่วยความจำหลัก หรืออ้างอิงข้อมูลในหน้าที่ไม่อยู่ในหน่วยความจำ **ข้อผิดพลาดหน้า (page fault)** จะถูกทริกเกอร์ นี่บอก OS ให้นำหน้าที่ต้องการเข้ามา

ดังนั้น ณ เวลาใดๆ มีเพียงไม่กี่หน้าของกระบวนการใดๆ ที่อยู่ในหน่วยความจำ จึงสามารถรักษากระบวนการได้มากขึ้นในหน่วยความจำ

อย่างไรก็ตาม OS ต้องจัดการโครงร่างนี้อย่างชาญฉลาด เมื่อนำหน้าหนึ่งเข้ามา ต้องย้ายหน้าอื่นออก ซึ่งเรียกว่า **การแทนที่หน้า (page replacement)** หากย้ายหน้าออกไปก่อนที่จะถูกใช้งาน ก็ต้องไปเอาหน้านั้นมาใหม่เกือบทันที ซึ่งนำไปสู่ **ภาวะแกว่ง (thrashing):** โปรเซสเซอร์ใช้เวลาส่วนใหญ่ไปกับการสลับหน้าแทนที่จะทำงานคำสั่ง เทคนิคที่มีประสิทธิภาพที่อาจใช้ได้คือ **ใช้น้อยที่สุดล่าสุด (Least Recently Used — LRU)** ซึ่งเป็นอัลกอริทึมเดียวกับที่ใช้ในบทที่ 4 สำหรับการแทนที่แคช

ด้วยการแบ่งหน้าตามความต้องการ ไม่จำเป็นต้องโหลดกระบวนการทั้งหมดเข้าหน่วยความจำหลัก ผลที่น่าทึ่งคือ **กระบวนการสามารถมีขนาดใหญ่กว่าหน่วยความจำหลักทั้งหมด** ข้อจำกัดพื้นฐานที่สุดประการหนึ่งในการเขียนโปรแกรมได้ถูกยกเลิกไปแล้ว

เนื่องจากกระบวนการทำงานเฉพาะในหน่วยความจำหลัก หน่วยความจำนั้นจึงเรียกว่า **หน่วยความจำจริง (real memory)** แต่นักพัฒนาหรือผู้ใช้รับรู้ถึงหน่วยความจำขนาดใหญ่กว่ามาก ซึ่งจัดสรรบนดิสก์ นั่นจึงเรียกว่า **หน่วยความจำเสมือน (virtual memory)** หน่วยความจำเสมือนช่วยให้มัลติโปรแกรมมิ่งมีประสิทธิภาพสูงมาก และปลดปล่อยผู้ใช้จากข้อจำกัดที่ไม่จำเป็นของหน่วยความจำหลัก

#### โครงสร้างตารางหน้า (Page Table Structure)

กลไกพื้นฐานสำหรับการอ่านคำจากหน่วยความจำเกี่ยวข้องกับการแปลที่อยู่เสมือน (virtual/logical address) ซึ่งประกอบด้วยหมายเลขหน้าและออฟเซต เป็นที่อยู่กายภาพ ซึ่งประกอบด้วยหมายเลขเฟรมและออฟเซต โดยใช้ตารางหน้า เนื่องจากตารางหน้ามีความยาวแปรผันขึ้นอยู่กับขนาดของกระบวนการ จึงไม่สามารถเก็บในรีจิสเตอร์ได้ ต้องอยู่ในหน่วยความจำหลักเพื่อการเข้าถึง

ในระบบส่วนใหญ่มีตารางหน้าหนึ่งต่อกระบวนการ แต่แต่ละกระบวนการสามารถครอบครองพื้นที่เสมือนจำนวนมหาศาลได้ เพื่อแก้ปัญหานี้ ระบบหน่วยความจำเสมือนส่วนใหญ่จัดเก็บตารางหน้าในหน่วยความจำเสมือนแทนที่จะเป็นหน่วยความจำจริง

บางโปรเซสเซอร์ใช้ **โครงร่างสองระดับ (two-level scheme)** เพื่อจัดระเบียบตารางหน้าขนาดใหญ่ โดยมี **ไดเรกทอรีหน้า (page directory)** ซึ่งแต่ละรายการชี้ไปยังตารางหน้า

แนวทางอื่นคือการใช้ **โครงสร้างตารางหน้าแบบกลับด้าน (inverted page table structure)** ซึ่งใช้ใน PowerPC, UltraSPARC และสถาปัตยกรรม IA-64 ในแนวทางนี้ หมายเลขหน้าส่วนหนึ่งของที่อยู่เสมือนถูกแมปเป็นค่าแฮชโดยใช้ฟังก์ชันแฮชง่ายๆ ค่าแฮชคือตัวชี้ไปยังตารางหน้าแบบกลับด้าน มีรายการหนึ่งในตารางหน้าแบบกลับด้านสำหรับแต่ละเฟรมหน้าหน่วยความจำจริง โครงสร้างตารางหน้าเรียกว่า "กลับด้าน" เพราะจัดทำดัชนีรายการตารางหน้าตามหมายเลขเฟรมแทนที่จะเป็นหมายเลขหน้าเสมือน

### บัฟเฟอร์ค้นหาการแปลที่อยู่ (Translation Lookaside Buffer)

โดยหลักการแล้ว การอ้างอิงหน่วยความจำเสมือนทุกครั้งอาจทำให้เกิดการเข้าถึงหน่วยความจำกายภาพสองครั้ง: ครั้งหนึ่งเพื่อดึงรายการตารางหน้าที่เหมาะสม และอีกครั้งเพื่อดึงข้อมูลที่ต้องการ ดังนั้นโครงร่างหน่วยความจำเสมือนแบบตรงไปตรงมาจะมีผลทำให้เวลาเข้าถึงหน่วยความจำเพิ่มเป็นสองเท่า

เพื่อแก้ปัญหานี้ ระบบหน่วยความจำเสมือนส่วนใหญ่ใช้ **แคชพิเศษสำหรับรายการตารางหน้า (special cache for page table entries)** ที่เรียกว่า **บัฟเฟอร์ค้นหาการแปลที่อยู่ (Translation Lookaside Buffer — TLB)** แคชนี้ทำงานในลักษณะเดียวกับแคชหน่วยความจำ และบรรจุรายการตารางหน้าที่ใช้งานล่าสุด โดยหลักการของความท้องถิ่น การอ้างอิงหน่วยความจำเสมือนส่วนใหญ่จะอยู่ในหน้าที่ใช้งานล่าสุด ดังนั้นการอ้างอิงส่วนใหญ่จะเกี่ยวข้องกับรายการตารางหน้าในแคช

กลไกหน่วยความจำเสมือนต้องโต้ตอบกับระบบแคช (ไม่ใช่แคช TLB แต่เป็นแคชหน่วยความจำหลัก) ที่อยู่เสมือนโดยทั่วไปจะอยู่ในรูปของหมายเลขหน้าและออฟเซต ก่อนอื่น ระบบหน่วยความจำจะตรวจสอบ TLB ว่ามีรายการตารางหน้าที่ตรงกันหรือไม่ หากมี จะสร้างที่อยู่จริง (กายภาพ) โดยการรวมหมายเลขเฟรมกับออฟเซต หากไม่มี จะเข้าถึงรายการจากตารางหน้า เมื่อสร้างที่อยู่จริงแล้ว จะตรวจสอบแคชว่ามีบล็อกที่บรรจุคำนั้นหรือไม่ หากมี จะถูกส่งคืนให้โปรเซสเซอร์ หากไม่มี คำนั้นจะถูกดึงมาจากหน่วยความจำหลัก

### การแบ่งส่วน (Segmentation)

มีอีกวิธีหนึ่งในการแบ่งย่อยหน่วยความจำที่สามารถระบุที่อยู่ได้ ซึ่งเรียกว่า **การแบ่งส่วน (segmentation)** ในขณะที่การแบ่งหน้าเป็นสิ่งที่นักพัฒนาไม่มองเห็นและมีจุดประสงค์เพื่อให้พื้นที่ที่อยู่ขนาดใหญ่แก่นักพัฒนา การแบ่งส่วนมักมองเห็นได้โดยนักพัฒนาและจัดเตรียมไว้เพื่อความสะดวกในการจัดระเบียบโปรแกรมและข้อมูล รวมถึงเป็นวิธีการเชื่อมโยงสิทธิ์สิทธิพิเศษ (privilege) และแอตทริบิวต์การป้องกัน (protection attributes) กับคำสั่งและข้อมูล

การแบ่งส่วนช่วยให้นักพัฒนามองหน่วยความจำว่าประกอบด้วยพื้นที่ที่อยู่หลายพื้นที่หรือส่วน (segments) ส่วนมีขนาดแปรผัน แท้จริงแล้วเป็นแบบไดนามิก โดยทั่วไป นักพัฒนาหรือ OS จะกำหนดโปรแกรมและข้อมูลให้กับส่วนต่างๆ อาจมีส่วนโปรแกรมหลายส่วนสำหรับโปรแกรมประเภทต่างๆ รวมถึงส่วนข้อมูลหลายส่วน แต่ละส่วนอาจได้รับสิทธิ์การเข้าถึงและสิทธิ์การใช้งาน การอ้างอิงหน่วยความจำประกอบด้วยที่อยู่ในรูปแบบ (หมายเลขส่วน, ออฟเซต)

การจัดระเบียบนี้มีข้อดีหลายประการสำหรับนักพัฒนาเมื่อเทียบกับพื้นที่ที่อยู่ที่ไม่มีส่วน:

1. **ง่ายต่อการจัดการโครงสร้างข้อมูลที่เติบโต:** หากนักพัฒนาไม่ทราบล่วงหน้าว่าโครงสร้างข้อมูลจะใหญ่เพียงใด ไม่จำเป็นต้องเดา สามารถกำหนดส่วนของตัวเองให้โครงสร้างข้อมูล และ OS จะขยายหรือย่นส่วนตามความจำเป็น

2. **ช่วยให้โปรแกรมถูกแก้ไขและคอมไพล์ใหม่ได้อย่างอิสระ:** โดยไม่จำเป็นต้องลิงก์ใหม่และโหลดชุดโปรแกรมทั้งหมด ซึ่งทำได้โดยใช้ส่วนหลายส่วน

3. **เหมาะสำหรับการแบ่งปันระหว่างกระบวนการ:** นักพัฒนาสามารถวางโปรแกรมยูทิลิตีหรือตารางข้อมูลที่มีประโยชน์ในส่วนที่กระบวนการอื่นสามารถอ้างอิงได้

4. **เหมาะสำหรับการป้องกัน:** เนื่องจากส่วนสามารถสร้างให้บรรจุชุดโปรแกรมหรือข้อมูลที่กำหนดไว้ชัดเจน นักพัฒนาหรือผู้ดูแลระบบสามารถกำหนดสิทธิ์การเข้าถึงได้อย่างสะดวก

ข้อดีเหล่านี้ไม่มีในการแบ่งหน้า ซึ่งมองไม่เห็นจากมุมมองของนักพัฒนา ในทางกลับกัน การแบ่งหน้าให้รูปแบบการจัดการหน่วยความจำที่มีประสิทธิภาพ เพื่อรวมข้อดีของทั้งสอง บางระบบจึงมีทั้งฮาร์ดแวร์และซอฟต์แวร์ OS เพื่อให้บริการทั้งสองอย่าง

## 8.4 การจัดการหน่วยความจำของ Intel x86

ตั้งแต่การแนะนำสถาปัตยกรรม 32 บิต ไมโครโปรเซสเซอร์ได้พัฒนาโครงร่างการจัดการหน่วยความจำที่ซับซ้อน ซึ่งอิงจากบทเรียนที่ได้รับจากระบบขนาดกลางและขนาดใหญ่ ตัวอย่างที่เป็นตัวแทนคือโครงร่างที่ใช้ในสถาปัตยกรรม Intel x86

### พื้นที่ที่อยู่ (Address Spaces)

x86 รวมฮาร์ดแวร์สำหรับทั้งการแบ่งส่วนและการแบ่งหน้า กลไกทั้งสองสามารถปิดใช้งานได้ ช่วยให้ผู้ใช้เลือกจากมุมมองหน่วยความจำสี่แบบที่แตกต่างกัน:

- **หน่วยความจำที่ไม่แบ่งส่วนและไม่แบ่งหน้า (Unsegmented unpaged memory):** ในกรณีนี้ ที่อยู่เสมือนเหมือนกับที่อยู่กายภาพ มีประโยชน์ในแอปพลิเคชันควบคุมที่ซับซ้อนน้อยและมีประสิทธิภาพสูง

- **หน่วยความจำที่ไม่แบ่งส่วนแต่แบ่งหน้า (Unsegmented paged memory):** หน่วยความจำมองเห็นเป็นพื้นที่ที่อยู่เชิงเส้นแบบแบ่งหน้า การป้องกันและการจัดการหน่วยความจำทำผ่านการแบ่งหน้า ระบบปฏิบัติการบางระบบชอบแนวทางนี้ (เช่น Berkeley UNIX)

- **หน่วยความจำที่แบ่งส่วนแต่ไม่แบ่งหน้า (Segmented unpaged memory):** หน่วยความจำมองเห็นเป็นกลุ่มของพื้นที่ที่อยู่ตรรกะ ข้อดีของมุมมองนี้เหนือแนวทางการแบ่งหน้าคือให้การป้องกันลงถึงระดับไบต์เดียว หากจำเป็น นอกจากนี้ ต่างจากการแบ่งหน้า รับประกันว่าตารางการแปล (segment table) ที่จำเป็นจะอยู่บนชิปเมื่อส่วนอยู่ในหน่วยความจำ ดังนั้นหน่วยความจำที่แบ่งส่วนแต่ไม่แบ่งหน้าจึงให้เวลาเข้าถึงที่คาดเดาได้

- **หน่วยความจำที่แบ่งส่วนและแบ่งหน้า (Segmented paged memory):** ใช้การแบ่งส่วนเพื่อกำหนดพาร์ติชันหน่วยความจำตรรกะที่อยู่ภายใต้การควบคุมการเข้าถึง และใช้การแบ่งหน้าเพื่อจัดการการจัดสรรหน่วยความจำภายในพาร์ติชัน ระบบปฏิบัติการ เช่น UNIX System V ชอบมุมมองนี้

### การแบ่งส่วน (Segmentation) — Intel x86

เมื่อใช้การแบ่งส่วน แต่ละที่อยู่เสมือน (เรียกว่า logical address ในเอกสาร Intel) ประกอบด้วยการอ้างอิงส่วน 16 บิตและออฟเซต 32 บิต สองบิตของการอ้างอิงส่วนจัดการกลไกการป้องกัน เหลือ 14 บิตสำหรับระบุส่วนเฉพาะ ดังนั้น ด้วยหน่วยความจำที่ไม่แบ่งส่วน หน่วยความจำเสมือนของผู้ใช้คือ 2^32 = 4 Gbytes ด้วยหน่วยความจำที่แบ่งส่วน พื้นที่หน่วยความจำเสมือนทั้งหมดที่ผู้ใช้มองเห็นคือ 2^46 = 64 terabytes (Tbytes) ส่วนพื้นที่ที่อยู่กายภาพใช้ที่อยู่ 32 บิตสำหรับสูงสุด 4 Gbytes

ที่เกี่ยวข้องกับแต่ละส่วนมีรูปแบบการป้องกันสองรูปแบบ: **ระดับสิทธิพิเศษ (privilege level)** และ **แอตทริบิวต์การเข้าถึง (access attribute)** มีระดับสิทธิพิเศษสี่ระดับ จากการป้องกันมากที่สุด (ระดับ 0) ถึงการป้องกันน้อยที่สุด (ระดับ 3)

ฮาร์ดแวร์ไม่ได้กำหนดวิธีใช้ระดับสิทธิพิเศษเหล่านี้ ขึ้นอยู่กับการออกแบบและการใช้งาน OS มีการตั้งใจให้ใช้ระดับสิทธิพิเศษดังนี้:
- **ระดับ 0:** ส่วนย่อยเล็กๆ ของ OS ที่จัดการหน่วยความจำ การป้องกัน และการควบคุมการเข้าถึง
- **ระดับ 1:** OS ส่วนใหญ่
- **ระดับ 2:** ระบบย่อยแอปพลิเคชันพิเศษที่ต้องการการป้องกัน เช่น ระบบการจัดการฐานข้อมูล
- **ระดับ 3:** โปรแกรมประยุกต์

กลไกการแปลที่อยู่สำหรับการแบ่งส่วนเกี่ยวข้องกับการแมปที่อยู่เสมือนไปยังสิ่งที่เรียกว่า **ที่อยู่เชิงเส้น (linear address)** ที่อยู่เสมือนประกอบด้วยออฟเซต 32 บิตและตัวเลือกส่วน (segment selector) 16 บิต ตัวเลือกส่วนประกอบด้วยฟิลด์:
- **ตัวบ่งชี้ตาราง (Table Indicator — TI):** ระบุว่าควรใช้ตารางส่วนทั่วโลก (global segment table) หรือตารางส่วนท้องถิ่น (local segment table) สำหรับการแปล
- **หมายเลขส่วน (Segment Number):** หมายเลขของส่วน ทำหน้าที่เป็นดัชนีในตารางส่วน
- **ระดับสิทธิพิเศษที่ร้องขอ (Requested Privilege Level — RPL):** ระดับสิทธิพิเศษที่ร้องขอสำหรับการเข้าถึงนี้

### การแบ่งหน้า (Paging) — Intel x86

การแบ่งส่วนเป็นฟีเจอร์ทางเลือกและอาจปิดใช้งานได้ เมื่อใช้การแบ่งส่วน ที่อยู่ที่ใช้ในโปรแกรมคือที่อยู่เสมือนและถูกแปลงเป็นที่อยู่เชิงเส้น เมื่อไม่ใช้การแบ่งส่วน จะใช้ที่อยู่เชิงเส้นในโปรแกรม ในทั้งสองกรณี ขั้นตอนต่อไปคือการแปลงที่อยู่เชิงเส้นนั้นเป็นที่อยู่จริง 32 บิต

กลไกการแบ่งหน้าของ x86 เป็นการค้นหาตารางสองระดับ ระดับแรกคือ **ไดเรกทอรีหน้า (page directory)** ซึ่งบรรจุรายการได้สูงสุด 1,024 รายการ แบ่งพื้นที่หน่วยความจำเชิงเส้น 4 Gbyte ออกเป็น 1,024 กลุ่มหน้า แต่ละกลุ่มมีตารางหน้าของตัวเองและมีความยาว 4 Mbytes แต่ละตารางหน้าบรรจุรายการได้สูงสุด 1,024 รายการ แต่ละรายการสอดคล้องกับหน้าขนาด 4 Kbyte เดียว

x86 ยังใช้บัฟเฟอร์ค้นหาการแปลที่อยู่ (TLB) ด้วย บัฟเฟอร์สามารถเก็บรายการตารางหน้าได้ 32 รายการ ทุกครั้งที่ไดเรกทอรีหน้าถูกเปลี่ยน บัฟเฟอร์จะถูกล้าง

นอกจากนี้ x86 รวมส่วนขยายใหม่ที่ไม่พบใน 80386 หรือ 80486 รุ่นก่อนหน้า คือการจัดเตรียมสำหรับขนาดหน้าสองขนาด หากบิต PSE (page size extension) ในรีจิสเตอร์ควบคุม 4 ถูกตั้งค่าเป็น 1 หน่วยการแบ่งหน้าจะอนุญาตให้โปรแกรมเมอร์ OS กำหนดหน้าเป็นขนาด 4 Kbyte หรือ 4 Mbyte

## 8.5 การจัดการหน่วยความจำของ ARM

ARM จัดเตรียมสถาปัตยกรรมระบบหน่วยความจำเสมือนที่หลากหลายซึ่งสามารถปรับให้เหมาะกับความต้องการของนักออกแบบระบบฝังตัวได้

### การจัดองค์กรระบบหน่วยความจำ (Memory System Organization)

ฮาร์ดแวร์การแปลหน่วยความจำเสมือนใช้ตารางหนึ่งหรือสองระดับสำหรับการแปลจากที่อยู่เสมือนเป็นที่อยู่กายภาพ TLB คือแคชของรายการตารางหน้าล่าสุด หากมีรายการใน TLB TLB จะส่งที่อยู่กายภาพตรงไปยังหน่วยความจำหลักสำหรับการดำเนินการอ่านหรือเขียน

รายการในตารางการแปลยังรวมถึงบิตควบคุมการเข้าถึง (access control bits) ซึ่งกำหนดว่ากระบวนการที่กำหนดสามารถเข้าถึงพื้นที่หน่วยความจำที่กำหนดได้หรือไม่ หากถูกปฏิเสธ ฮาร์ดแวร์ควบคุมการเข้าถึงจะส่งสัญญาณยกเลิก (abort signal) ไปยังโปรเซสเซอร์ ARM

### การแปลที่อยู่หน่วยความจำเสมือน (Virtual Memory Address Translation)

ARM รองรับการเข้าถึงหน่วยความจำตามส่วน (sections) หรือหน้า (pages):

- **ซูเปอร์เซกชัน (Supersections) — ทางเลือก:** ประกอบด้วยบล็อก 16 MB ของหน่วยความจำหลัก
- **เซกชัน (Sections):** ประกอบด้วยบล็อก 1 MB ของหน่วยความจำหลัก
- **หน้าขนาดใหญ่ (Large pages):** ประกอบด้วยบล็อก 64 kB ของหน่วยความจำหลัก
- **หน้าขนาดเล็ก (Small pages):** ประกอบด้วยบล็อก 4 kB ของหน่วยความจำหลัก

เซกชันและซูเปอร์เซกชันได้รับการรองรับเพื่อให้แมปพื้นที่หน่วยความจำขนาดใหญ่โดยใช้รายการเพียงรายการเดียวใน TLB ตารางการแปลที่เก็บในหน่วยความจำหลักมีสองระดับ:

- **ตารางระดับ 1 (Level 1 table):** เก็บตัวอธิบายระดับ 1 ที่บรรจุที่อยู่ฐานและคุณสมบัติการแปลสำหรับเซกชันและซูเปอร์เซกชัน และคุณสมบัติการแปลและตัวชี้ไปยังตารางระดับ 2 สำหรับหน้าขนาดใหญ่หรือขนาดเล็ก
- **ตารางระดับ 2 (Level 2 table):** เก็บตัวอธิบายระดับ 2 ที่บรรจุที่อยู่ฐานและคุณสมบัติการแปลสำหรับหน้าขนาดเล็กหรือขนาดใหญ่ ตารางระดับ 2 ต้องใช้หน่วยความจำ 1 kB

หน่วยจัดการหน่วยความจำ (Memory Management Unit — MMU) แปลที่อยู่เสมือนที่สร้างโดยโปรเซสเซอร์เป็นที่อยู่กายภาพเพื่อเข้าถึงหน่วยความจำหลัก และยังระบุและตรวจสอบสิทธิ์การเข้าถึงด้วย

กระบวนการแปลที่อยู่สองระดับสำหรับหน้าขนาดเล็กเป็นดังนี้ มีตาราง L1 เพียงหนึ่งตารางที่มี 4K รายการ 32 บิต แต่ละรายการ L1 ชี้ไปยังตาราง L2 ที่มี 256 รายการ 32 บิต แต่ละรายการ L2 ชี้ไปยังหน้า 4 kB ในหน่วยความจำหลัก ที่อยู่เสมือน 32 บิตถูกตีความดังนี้: 12 บิตที่มีนัยสำคัญสูงสุดเป็นดัชนีในตาราง L1 อีก 8 บิตถัดมาเป็นดัชนีในตาราง L2 ที่เกี่ยวข้อง 12 บิตที่มีนัยสำคัญน้อยที่สุดจะสร้างดัชนีไบต์ในหน้าที่เกี่ยวข้องในหน่วยความจำหลัก

### รูปแบบการจัดการหน่วยความจำ (Memory-Management Formats)

สำหรับตาราง L1 แต่ละรายการเป็นตัวอธิบายว่าช่วงที่อยู่เสมือน 1 MB ที่เกี่ยวข้องถูกแมปอย่างไร แต่ละรายการมีหนึ่งในสี่รูปแบบทางเลือก:

- **บิต [1:0] = 00:** ที่อยู่เสมือนที่เกี่ยวข้องไม่ถูกแมป และการพยายามเข้าถึงจะสร้างข้อผิดพลาดการแปล (translation fault)
- **บิต [1:0] = 01:** รายการให้ที่อยู่กายภาพของตาราง L2 ซึ่งระบุวิธีแมปช่วงที่อยู่เสมือนที่เกี่ยวข้อง
- **บิต [1:0] = 01 และบิต 19 = 0:** รายการเป็นตัวอธิบายเซกชันสำหรับที่อยู่เสมือนที่เกี่ยวข้อง
- **บิต [1:0] = 01 และบิต 19 = 1:** รายการเป็นตัวอธิบายซูเปอร์เซกชันสำหรับที่อยู่เสมือนที่เกี่ยวข้อง

สำหรับหน่วยความจำที่มีโครงสร้างเป็นหน้า จำเป็นต้องเข้าถึงตารางหน้าสองระดับ สำหรับหน้าขนาดเล็ก รายการ L2 บรรจุตัวชี้ 20 บิตไปยังที่อยู่ฐานของหน้า 4 kB ในหน่วยความจำหลัก

ช่วงพื้นที่ที่อยู่กายภาพสามารถขยายด้วยบิตที่อยู่เพิ่มเติมได้สูงสุด 8 บิต บิตเพิ่มเติมเหล่านี้สามารถตีความว่าขยายขนาดของหน่วยความจำกายภาพได้มากถึง 2^8 = 256 เท่า ดังนั้นหน่วยความจำกายภาพอาจใหญ่ถึง 256 เท่าของพื้นที่หน่วยความจำที่มีสำหรับแต่ละกระบวนการแต่ละตัว

### การควบคุมการเข้าถึง (Access Control)

บิตควบคุมการเข้าถึง AP ในแต่ละรายการตารางควบคุมการเข้าถึงพื้นที่หน่วยความจำโดยกระบวนการที่กำหนด พื้นที่หน่วยความจำสามารถกำหนดเป็น ไม่มีการเข้าถึง (no access) อ่านอย่างเดียว (read only) หรืออ่านเขียน (read-write) นอกจากนี้ พื้นที่ยังสามารถกำหนดเป็นการเข้าถึงสิทธิพิเศษเท่านั้น สำรองไว้สำหรับ OS และไม่ใช่โปรแกรมประยุกต์

ARM ยังใช้แนวคิดของ **โดเมน (domain)** ซึ่งเป็นกลุ่มของส่วนและ/หรือหน้าที่มีสิทธิ์การเข้าถึงเฉพาะ สถาปัตยกรรม ARM รองรับ 16 โดเมน ฟีเจอร์โดเมนช่วยให้กระบวนการหลายกระบวนการใช้ตารางการแปลเดียวกันในขณะที่รักษาการป้องกันจากกันและกัน

แต่ละรายการตารางหน้าและรายการ TLB บรรจุฟิลด์ที่ระบุว่ารายการนั้นอยู่ในโดเมนใด ฟิลด์ 2 บิตในรีจิสเตอร์ควบคุมการเข้าถึงโดเมน (Domain Access Control Register) ควบคุมการเข้าถึงโดเมนแต่ละโดเมน แต่ละฟิลด์ช่วยให้การเข้าถึงโดเมนทั้งหมดสามารถเปิดและปิดได้อย่างรวดเร็วมาก การเข้าถึงโดเมนสองประเภทได้รับการรองรับ:

- **ไคลเอนต์ (Clients):** ผู้ใช้โดเมน (รันโปรแกรมและเข้าถึงข้อมูล) ที่ต้องปฏิบัติตามสิทธิ์การเข้าถึงของส่วนและ/หรือหน้าแต่ละส่วนที่ประกอบเป็นโดเมนนั้น
- **ผู้จัดการ (Managers):** ควบคุมพฤติกรรมของโดเมน (ส่วนและหน้าปัจจุบันในโดเมน และการเข้าถึงโดเมน) และข้ามผ่านสิทธิ์การเข้าถึงของรายการตารางในโดเมนนั้น

โปรแกรมหนึ่งสามารถเป็นไคลเอนต์ของโดเมนบางโดเมน และเป็นผู้จัดการของโดเมนบางโดเมนอื่น และไม่มีสิทธิ์เข้าถึงโดเมนที่เหลือ สิ่งนี้ช่วยให้มีการป้องกันหน่วยความจำที่ยืดหยุ่นมากสำหรับโปรแกรมที่เข้าถึงทรัพยากรหน่วยความจำต่างๆ

### โจทย์ปัญหา (Problems)

**8.1** สมมติว่าเรามีคอมพิวเตอร์มัลติโปรแกรมที่แต่ละงานมีคุณสมบัติเหมือนกัน ในหนึ่งช่วงการคำนวณ T สำหรับงาน ครึ่งหนึ่งของเวลาใช้กับ I/O และอีกครึ่งหนึ่งใช้กับกิจกรรมโปรเซสเซอร์ แต่ละงานทำงานเป็นเวลา N ช่วงรวม สมมติว่าใช้ลำดับความสำคัญแบบ round-robin อย่างง่าย และการดำเนินการ I/O สามารถซ้อนทับกับการดำเนินการโปรเซสเซอร์ได้ กำหนดปริมาณต่อไปนี้:

- เวลาหมุนเวียน (Turnaround time) = เวลาจริงในการทำงานให้เสร็จ
- อัตราการผ่าน (Throughput) = จำนวนงานเฉลี่ยที่เสร็จต่อช่วงเวลา T
- การใช้โปรเซสเซอร์ (Processor utilization) = เปอร์เซ็นต์ของเวลาที่โปรเซสเซอร์ทำงานอยู่ (ไม่รอ)

คำนวณปริมาณเหล่านี้สำหรับงานพร้อมกันหนึ่ง สอง และสี่งาน โดยสมมติว่าช่วง T ถูกกระจายในแต่ละวิธีต่อไปนี้:

- **a.** I/O ครึ่งแรก โปรเซสเซอร์ครึ่งหลัง
- **b.** I/O ไตรมาสแรกและสี่ โปรเซสเซอร์ไตรมาสสองและสาม

**8.2** โปรแกรมที่ผูกกับ I/O คือโปรแกรมที่หากรันคนเดียว จะใช้เวลารอ I/O มากกว่าใช้โปรเซสเซอร์ โปรแกรมที่ผูกกับโปรเซสเซอร์คือสิ่งตรงกันข้าม สมมติว่าอัลกอริทึมการจัดตารางระยะสั้นชอบโปรแกรมที่ใช้เวลาโปรเซสเซอร์น้อยในช่วงเวลาล่าสุด จงอธิบายว่าทำไมอัลกอริทึมนี้จึงชอบโปรแกรมที่ผูกกับ I/O แต่ไม่ปฏิเสธเวลาโปรเซสเซอร์แก่โปรแกรมที่ผูกกับโปรเซสเซอร์อย่างถาวร

**8.3** โปรแกรมคำนวณผลรวมของแถว
