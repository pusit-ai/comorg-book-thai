# 🖥️ Operating System Support: Scheduling Summary

เอกสารฉบับนี้สรุปเนื้อหาบทที่ 8.2 เรื่องการจัดการลำดับงาน (Scheduling) ในระบบปฏิบัติการ เพื่อรองรับการทำงานแบบ **Multiprogramming**

---

## 1. ประเภทของการจัดการลำดับงาน (Types of Scheduling)

| ประเภทการจัดการ | หน้าที่หลัก (Key Responsibility) |
| :--- | :--- |
| **Long-term scheduling** | ตัดสินใจรับโปรแกรมเข้าสู่ระบบ เพื่อควบคุมจำนวน Process ใน Memory (**Degree of Multiprogramming**) |
| **Medium-term scheduling** | จัดการการสลับงานเข้า-ออก (**Swapping**) ระหว่างหน่วยความจำหลักและดิสก์ |
| **Short-term scheduling** | ทำหน้าที่เป็น **Dispatcher** เลือก Process ที่พร้อม (Ready) เข้าสู่ CPU (ทำงานบ่อยที่สุด) |
| **I/O scheduling** | ตัดสินใจลำดับการจัดการคำขอ I/O ที่ค้างอยู่เพื่อส่งให้อุปกรณ์ทำงาน |



---

## 2. แนวคิดเรื่อง Process และสถานะ (Process Concept & States)

**Process** คือ "โปรแกรมที่กำลังประมวลผล" (*A program in execution*) โดยมีวงจรชีวิตผ่าน 5 สถานะหลัก ดังนี้:

* **New (ใหม่)**: สร้าง Process แล้วแต่ยังไม่ถูกยอมรับเข้าคิว Ready
* **Ready (พร้อม)**: พร้อมทำงาน แต่กำลังรอคิวเข้าใช้ CPU
* **Running (กำลังทำงาน)**: กำลังถูกประมวลผลโดย CPU
* **Waiting/Blocked (รอ)**: หยุดชั่วคราวเพื่อรอทรัพยากร (เช่น รอ I/O หรือ Event)
* **Halted/Exit (จบการทำงาน)**: ทำงานเสร็จสิ้นและถูกทำลายออกจากระบบ



---

## 3. Process Control Block (PCB)

เพื่อให้ระบบปฏิบัติการ (OS) จำได้ว่าแต่ละ Process ทำงานถึงไหนแล้ว OS จะต้องเก็บข้อมูลไว้ใน **PCB** ซึ่งประกอบด้วยข้อมูลสำคัญดังนี้:

* 🆔 **Identifier**: เลข ID ประจำตัว Process
* 📊 **State**: สถานะปัจจุบัน (Ready, Running, ฯลฯ)
* 📍 **Program Counter**: ที่อยู่ของคำสั่งถัดไปที่จะประมวลผล
* 💾 **Context Data**: ข้อมูลใน Register ของ CPU ที่บันทึกไว้ก่อนสลับงาน
* 📑 **I/O Status**: รายการไฟล์หรืออุปกรณ์ที่ Process กำลังใช้งานอยู่



---

## 4. กลไกการทำงานและคิว (Queuing Mechanism)

OS บริหารจัดการทรัพยากรผ่านระบบคิว (Queues):

1.  **Long-term queue**: งานที่รอเข้าสู่ระบบ
2.  **Short-term queue**: Process ที่พร้อมรัน (มักใช้ Algorithm แบบ **Round-robin**)
3.  **I/O queues**: คิวรอสำหรับอุปกรณ์ I/O แต่ละประเภท

### 🔄 ขั้นตอนการสลับงาน (Context Switch)
เมื่อเกิด **Interrupt** หรือ **Service Call**:
1.  OS เข้ามาควบคุม CPU แทน Process เดิม
2.  บันทึกสถานะปัจจุบันลงใน **PCB** ของ Process นั้น
3.  **Short-term scheduler** เลือก Process ใหม่จากคิว
4.  คืนค่าสถานะจาก PCB ของ Process ใหม่เข้า CPU แล้วเริ่มทำงานต่อ

---

## 5. เปรียบเทียบ: Batch vs Time Sharing

| หัวข้อเปรียบเทียบ | Batch Multiprogramming | Time Sharing |
| :--- | :--- | :--- |
| **วัตถุประสงค์หลัก** | ใช้งาน CPU ให้คุ้มค่าที่สุด (**Maximize CPU Use**) | ลดระยะเวลาตอบสนอง (**Minimize Response Time**) |
| **แหล่งคำสั่ง** | JCL (Job Control Language) | คำสั่งจาก Terminal (Interactive) |
